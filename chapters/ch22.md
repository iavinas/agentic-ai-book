# Chapter 22: Verification, Self-Correction, and Refinement

> An LLM can generate a thousand lines of Python in seconds. It can also generate a thousand lines of subtly broken Python just as fast. The gap between generation and correctness is the central reliability challenge for agents. This chapter closes that gap: we explore how agents verify their own outputs, correct their own mistakes, and know when to stop trying.

---

## 1. Self-Checking Mechanisms

### 1.1 Why Generation Is Not Enough

A language model produces text by sampling tokens from a learned distribution. At no point in this process is there a guarantee that the output is correct, consistent, or even executable. The model predicts what *looks* right, not what *is* right. For an agent that writes code, plans tasks, or answers factual questions, this distinction is everything.

Consider a coding agent asked to implement a function that computes the median of a list. The model might generate code that handles even-length lists incorrectly, forgets to sort the input, or uses integer division where float division is required. These errors are invisible to the generator — the code is syntactically valid and superficially plausible. Only execution reveals the bug.

This is why **self-checking** is not an optional enhancement. It is a necessary layer of the agent architecture. The core idea is simple: after generating an output, the agent subjects it to one or more independent verification procedures before returning it to the user or using it as the basis for further action.

Verification procedures fall into three broad categories. *Execution-based* verifiers run the output and check the result: execute code, evaluate mathematical expressions, or simulate plans. *Reference-based* verifiers compare the output against an external source of truth: unit tests, type checkers, theorem provers, or search engines. *Consistency-based* verifiers check internal coherence: does the answer match the reasoning, and does the reasoning match itself?

### 1.2 Tool-Interactive Verification

The most reliable form of self-checking uses external tools. An agent that writes Python can run the code. An agent that computes a mathematical result can pass it to a calculator. An agent that makes a factual claim can query a search engine. These tools ground the verification process in reality rather than in the model's own parametric knowledge.

**Unit tests** are the canonical execution-based verifier for code. Given a function and a set of input-output pairs, the agent executes the function on each input and checks that the output matches the expected value. A passing test suite does not prove correctness — it proves consistency with the test cases — but it catches a wide class of implementation errors.

**Type checkers** provide a lighter-weight form of verification. By running `mypy` or a similar tool over generated code, the agent can detect type mismatches, undefined variables, and incompatible signatures without executing the program. Type checking is fast and deterministic, making it an ideal first-line verifier before expensive test execution.

**Theorem provers** offer the strongest guarantees for mathematical or logical outputs. If an agent generates a proof, a theorem prover like Lean or Coq can verify that every inference step is valid. This is computationally expensive and requires the output to be in a formal language, but for domains where correctness is critical, it is unmatched.

The key insight is that the verifier and the generator should be decoupled. The generator is a large language model optimized for fluency and coherence. The verifier is a specialized tool optimized for correctness. Neither can do the other's job well, but together they form a system that is more reliable than either alone.

### 1.3 Consistency Checking

Not all outputs can be verified by external tools. An agent summarizing a document, drafting an email, or answering an open-ended question has no oracle to query. In these cases, the agent can still perform **consistency checking** — verifying that the output is internally coherent and aligned with the reasoning that produced it.

The simplest form of consistency checking is answer-reasoning alignment. If the agent's reasoning trace says "the capital of France is Paris because it has been the seat of government since 987," but the final answer says "Lyon," the inconsistency is obvious. More subtly, the agent can check that numerical values mentioned in the reasoning match those in the answer, that logical deductions follow from stated premises, and that no contradictions appear across the reasoning trace.

A stronger form of consistency checking uses **self-consistency sampling**. The agent generates multiple independent answers to the same question and checks whether they agree. If three samples all produce the same result, confidence is high. If they diverge, the agent knows it is uncertain. This technique, introduced by Wang et al. (2022) for chain-of-thought reasoning, requires no external tools — only the ability to sample multiple times from the same model.

**SelfCheckGPT** (Manakul et al., 2023) extends this idea to hallucination detection. The agent generates multiple responses to the same prompt and measures semantic consistency between them. Low consistency indicates that the model is hallucinating — generating plausible-sounding but ungrounded information. The agent can then flag the output for human review or trigger a retrieval step to ground the answer in external documents.

### 1.4 Multi-Modal Verification

Agents increasingly operate across modalities: they generate code from text descriptions, describe images, or produce structured data from visual inputs. In these settings, verification can be **multi-modal** — cross-checking outputs against inputs from a different modality.

If an agent generates a caption for an image, the caption can be fed back to a vision-language model along with the original image. The model is asked: "Does this caption accurately describe the image?" If the answer is no, the caption is revised. Similarly, if an agent writes code to render a plot, the generated image can be compared against the requested specification using visual similarity metrics or a second VLM critique.

Multi-modal verification is expensive — it requires inference on a vision model in addition to the text model — but it is essential for tasks where text-only verification is insufficient. An agent that generates HTML from a screenshot, for example, cannot verify its output without rendering the HTML and comparing the visual result.

## 2. Self-Correction Strategies

Verification tells an agent that something is wrong. Self-correction tells it what to do about it. The strategies differ in what feedback they use — internal monologue, external tools, or stored memory from past failures.

### 2.1 Reflexion: Verbal Reinforcement Learning

**Reflexion** (Shinn et al., 2023) treats self-correction as a form of verbal reinforcement learning. After each episode — whether successful or not — the agent generates a textual **self-reflection** that diagnoses what went wrong and how to do better next time. This reflection is stored in memory and prepended to the agent's context on subsequent episodes.

The insight is powerful: instead of updating neural network weights, the agent updates its prompt. The reflection acts as a learned hint, guiding the model away from previously observed failure modes without any gradient computation. A reflection might read: "When the user asks for a sorting function, I should remember to handle empty lists and verify the output is actually sorted. I failed on the previous task because I assumed the input was non-empty."

Reflexion works because language models are few-shot learners. A well-written reflection provides the model with a relevant example of failure and recovery, shifting the conditional distribution of its outputs toward better solutions. Over multiple episodes, the accumulated reflections form a personalized instruction manual for the agent.

The key design choice is what to include in the reflection. A good reflection is specific — it names the exact error, the condition that triggered it, and a concrete strategy for avoidance. Vague reflections like "I should be more careful" are useless. The reflection must also be concise; stuffing the context window with verbose post-mortems leaves less room for the actual task.

Here is a minimal implementation of the Reflexion loop. The agent attempts a simple block-stacking task across three episodes, storing reflections and improving its success rate.

```python
import random
from typing import List, Tuple

class ReflexionAgent:
    """Minimal Reflexion agent for a deterministic block-stacking task."""

    def __init__(self):
        self.memory: List[str] = []

    def solve(self, task: str, episode: int) -> Tuple[str, bool]:
        """Return (plan, success) for the given task."""
        # Simulate LLM reasoning with reflections prepended
        reflections = "\n".join(self.memory)
        prompt = f"{reflections}\nTask: {task}\nPlan:"

        # Deterministic simulation: agent fails on episode 1,
        # improves based on reflections for episodes 2 and 3.
        if episode == 1:
            plan = "move A to B; move C to A"
            success = False  # forgot to clear B first
        elif episode == 2:
            if "clear the destination" in reflections:
                plan = "clear B; move A to B; move C to A"
                success = True
            else:
                plan = "move A to B; move C to A"
                success = False
        else:
            if len(self.memory) >= 2:
                plan = "clear B; move A to B; move C to A; verify stack"
                success = True
            else:
                plan = "move A to B; move C to A"
                success = False
        return plan, success

    def reflect(self, task: str, plan: str, success: bool) -> str:
        """Generate a self-reflection string."""
        if success:
            return f"Task '{task}': succeeded with plan '{plan}'."
        return (
            f"Task '{task}': FAILED with plan '{plan}'. "
            "Lesson: always clear the destination block before moving."
        )

    def run_episodes(self, task: str, n: int = 3):
        for ep in range(1, n + 1):
            plan, success = self.solve(task, ep)
            reflection = self.reflect(task, plan, success)
            self.memory.append(reflection)
            print(f"Episode {ep}: success={success}, plan='{plan}'")
        print(f"\nFinal memory has {len(self.memory)} reflections.")


agent = ReflexionAgent()
agent.run_episodes("Stack C on A on B")
```

In this toy example, the first episode fails because the destination block is not cleared. The reflection captures this lesson explicitly. By episode 3, the agent has accumulated enough guidance to generate a correct plan with a verification step. Real Reflexion agents use an actual LLM to generate the reflection text, but the principle is identical: memory of past failures prevents future failures.

### 2.2 Self-Refine: Iterative Self-Correction

**Self-Refine** (Madaan et al., 2023) takes a different approach. Instead of storing reflections across episodes, it iteratively refines a single output within one episode. The loop is: generate an initial output, critique it, revise it based on the critique, and repeat.

The critique is generated by the same model that produced the output, but with a different prompt. The model is asked to act as a critic: "Review the following text for factual errors, logical inconsistencies, and stylistic problems. List specific issues and suggest corrections." The critique is then fed back into the model along with the original output, and the model is asked to produce a revised version that addresses the identified problems.

This works surprisingly well for tasks where the model has enough knowledge to recognize its own errors. A model that generates a flawed mathematical proof may also be able to spot the logical gap when prompted to critique it. A model that writes stilted prose may recognize the awkward phrasing when asked to review it.

But Self-Refine has a critical limitation: the model can only correct errors that it can recognize. If the model lacks the knowledge to identify a bug, the critique will miss it, and the revision will preserve it. This is why Self-Refine works best for tasks where the model's knowledge of the domain is strong enough to support both generation and evaluation.

### 2.3 CRITIC: Tool-Interactive Critiquing

**CRITIC** (Gou et al., 2024) addresses the core weakness of Self-Refine by grounding the critique in external tools. Instead of asking the model to critique its own output, CRITIC asks the model to verify its output using tools: execute generated code, search the web for factual claims, or calculate mathematical results.

The pipeline is structured. First, the agent generates an initial answer. Second, the agent extracts verifiable claims from the answer: code snippets, numerical facts, named entities. Third, each claim is sent to an appropriate verifier. Fourth, any discrepancies between the claim and the verifier's output are collected into a critique. Fifth, the agent revises the answer to resolve the discrepancies. The process repeats up to a maximum number of iterations.

The power of CRITIC comes from the separation of critique and generation. The generator is optimized for fluency and plausibility. The verifier is optimized for correctness. The critique bridges them by translating verifier failures into actionable revision instructions.

Here is a lightweight CRITIC pipeline for Python code generation. A mock LLM produces buggy code, a verifier runs it in a sandboxed subprocess, and the critique-revision loop continues until the code passes or the iteration limit is reached.

```python
import subprocess
import tempfile
import os
from typing import Optional, Tuple

class MockLLM:
    """Deterministic mock that cycles through buggy and fixed versions."""

    VERSIONS = [
        "def median(nums):\n    return nums[len(nums)//2]",
        "def median(nums):\n    nums = sorted(nums)\n    return nums[len(nums)//2]",
        "def median(nums):\n    nums = sorted(nums)\n    n = len(nums)\n    if n % 2 == 1:\n        return nums[n//2]\n    return (nums[n//2 - 1] + nums[n//2]) / 2",
    ]

    def __init__(self):
        self.call_count = 0

    def generate(self, prompt: str) -> str:
        code = self.VERSIONS[min(self.call_count, len(self.VERSIONS) - 1)]
        self.call_count += 1
        return code

    def critique(self, code: str, error: Optional[str]) -> str:
        if error is None:
            return "Code executes but may be incorrect. Check edge cases."
        return f"Execution failed with: {error}. Review logic and fix."


class CriticPipeline:
    """CRITIC: generate, verify, critique, revise — max 3 iterations."""

    def __init__(self, llm: MockLLM, max_iter: int = 3):
        self.llm = llm
        self.max_iter = max_iter

    def verify(self, code: str) -> Tuple[bool, Optional[str]]:
        """Run code in a temp file with restricted globals."""
        with tempfile.NamedTemporaryFile(
            mode="w", suffix=".py", delete=False
        ) as f:
            f.write(code + "\n")
            f.write("print(median([1, 3, 2]))\n")
            f.write("print(median([1, 2, 3, 4]))\n")
            path = f.name

        try:
            result = subprocess.run(
                ["python", path],
                capture_output=True,
                text=True,
                timeout=5,
            )
            os.unlink(path)
            if result.returncode != 0:
                return False, result.stderr.strip()[:200]
            lines = result.stdout.strip().splitlines()
            if lines != ["2", "2.5"]:
                return False, f"Wrong output: {lines}"
            return True, None
        except Exception as e:
            return False, str(e)

    def run(self, task: str) -> Tuple[str, bool, int]:
        """Return (final_code, success, iterations_used)."""
        code = self.llm.generate(task)
        for iteration in range(1, self.max_iter + 1):
            passed, error = self.verify(code)
            if passed:
                return code, True, iteration
            critique = self.llm.critique(code, error)
            revision_prompt = (
                f"Task: {task}\nCode:\n{code}\n"
                f"Critique: {critique}\nProvide corrected code."
            )
            code = self.llm.generate(revision_prompt)
        return code, False, self.max_iter


llm = MockLLM()
pipeline = CriticPipeline(llm)
final_code, success, iters = pipeline.run("Write a median function.")
print(f"Success: {success} after {iters} iteration(s)")
print(final_code)
```

In this example, the first version fails because it does not sort the input. The second version sorts but handles only odd-length lists correctly. The third version finally handles both odd and even lengths. Each failure produces an execution trace that the critique uses to guide revision. Without the verifier, the model would have no signal that the first two versions were wrong.

### 2.4 When Self-Correction Helps and When It Loops

Self-correction is not universally beneficial. There are three conditions under which it helps, and three failure modes where it does not.

It helps when the agent has access to a reliable verifier. CRITIC outperforms Self-Refine precisely because tool-based verification is more accurate than self-critique. It helps when the error is localizable — the agent can identify which part of the output is wrong and revise only that part. Global failures, where the entire approach is flawed, are harder to correct incrementally. It helps when the revision space is constrained — there are only a few plausible ways to fix a bug, and the agent can search among them effectively.

It fails when the agent enters a **correction loop**. The agent generates output, the verifier rejects it, the agent revises, the verifier rejects again, and the cycle continues. Without a maximum iteration count, this loop is infinite. Even with a bound, the agent may oscillate between two versions that both fail for different reasons.

It fails when the critique is wrong. A hallucinated critique — the model inventing errors that do not exist — can degrade a correct output into an incorrect one. This is rare but documented: models sometimes "over-correct," introducing errors in the process of fixing imagined ones.

It fails when the verifier itself is flawed. If the unit test has a bug, the agent will chase a phantom error forever. If the search engine returns misinformation, the agent will revise correct answers into incorrect ones. The reliability of the entire self-correction pipeline is bounded by the reliability of its weakest component.

## 3. Confidence-Guided Refinement

Verification and self-correction are reactive: they respond to failures after they occur. **Confidence-guided refinement** is proactive: it estimates the model's uncertainty before committing to an output, and uses that estimate to decide whether to halt, rethink, or explore alternatives.

### 3.1 Estimating Confidence

Confidence estimation asks a simple question: how sure is the model that this output is correct? The answer can come from multiple sources.

**Token-level logprobs** are the most direct signal. When a model generates a token, it also outputs the log-probability of that token under its distribution. A sequence of high-probability tokens indicates confidence; a sequence of low-probability tokens indicates uncertainty. The average logprob across a reasoning step, or the minimum logprob within it, can serve as a scalar confidence score.

**Consistency across samples** provides a model-agnostic alternative. Generate $N$ independent answers and measure agreement. If all $N$ agree, confidence is high. If they split evenly between two answers, confidence is low. This requires no access to internal model probabilities and works for any model that supports sampling.

**Trained confidence models** learn to predict correctness directly. Given a dataset of (output, label) pairs, a small neural network — often just a few layers — is trained to map the output or the model's hidden states to a correctness probability. These models can be more accurate than raw logprobs because they learn to aggregate multiple uncertainty signals.

Each method has trade-offs. Logprobs are cheap but noisy — a model can be confidently wrong. Consistency sampling is robust but expensive — it requires $N$ forward passes. Trained models are accurate but require labeled data and add inference overhead.

### 3.2 CoRefine: Adaptive Test-Time Refinement

**CoRefine** (Feb 2026) unifies confidence estimation and self-correction into a single adaptive controller. Instead of always revising or always halting, CoRefine uses token-level confidence to choose among three actions: **HALT** (the output is good enough), **RETHINK** (revise the current approach), or **ALTERNATIVE** (try a completely different reasoning path).

The controller is lightweight — a Conv1D network with approximately 211K parameters — that processes the sequence of per-token confidence scores and outputs a discrete action. Despite its small size, it achieves high precision: 92.6% of HALT decisions are correct, meaning the agent rarely stops when it should have continued.

CoRefine reduces token consumption by 190x compared to self-consistency with 512 samples, and cuts latency by 63%. The average problem requires only 2.7 refinement steps before the controller issues a HALT signal. This efficiency comes from the controller's ability to distinguish between momentary uncertainty — a single low-confidence token in an otherwise solid chain — and systematic uncertainty that signals a flawed approach.

The key insight is that not all uncertainty is equal. A model may be uncertain about a specific arithmetic operation but confident in the overall strategy. CoRefine's controller learns to make this distinction, applying expensive revision only when the uncertainty pattern indicates a real problem.

### 3.3 The Halting Problem for Agents

Knowing when to stop is as important as knowing how to improve. An agent that revises forever consumes infinite compute and never delivers an answer. An agent that halts too early returns incorrect outputs. The challenge is to find the right stopping criterion.

The theoretical problem is a variant of the **halting problem**: no general procedure can determine, for all possible agents and all possible tasks, whether additional revision would improve the output. But in practice, agents operate on specific tasks with specific verifiers, and effective heuristics exist.

The simplest heuristic is a **maximum iteration count**. After $k$ revisions, the agent returns its best output regardless of quality. This guarantees termination but may return garbage if the agent has not converged. A more sophisticated heuristic uses **confidence saturation**: if the confidence score stops improving across iterations, the agent halts. If revision no longer helps, there is no point in continuing.

**Verifier convergence** is another signal. If the agent's output passes all available verifiers and the confidence score is above a threshold, it halts. If the output fails verifiers but the failure mode is the same across iterations — the agent keeps making the same mistake — it halts and reports failure rather than looping.

The most robust approach combines multiple signals: halt if (confidence $> \tau_1$ AND verifiers pass) OR (iterations $> k$) OR (confidence improvement $< \tau_2$ over last 2 iterations). This composite rule balances quality, efficiency, and termination guarantees.

Here is a lightweight confidence-guided decision function that uses sample consistency to estimate confidence and maps it to HALT, RETHINK, or ALTERNATIVE.

```python
from typing import List, Literal

def consistency_confidence(samples: List[str]) -> float:
    """Estimate confidence from agreement among N samples.

    Args:
        samples: N independent model outputs for the same prompt.

    Returns:
        A confidence score in [0.0, 1.0] based on majority agreement.
    """
    if not samples:
        return 0.0
    from collections import Counter
    counts = Counter(samples)
    majority = counts.most_common(1)[0][1]
    return majority / len(samples)


def decide_action(
    confidence: float,
    verifier_passed: bool,
    iteration: int,
    max_iter: int = 3,
    halt_threshold: float = 0.9,
    rethink_threshold: float = 0.5,
) -> Literal["HALT", "RETHINK", "ALTERNATIVE"]:
    """Return the next action for a confidence-guided refinement loop.

    The policy is a simple state machine:
      - HALT if confidence is high and verifiers pass.
      - RETHINK if confidence is moderate or verifiers fail with budget left.
      - ALTERNATIVE if confidence is low and we have exhausted revision budget.
    """
    if confidence >= halt_threshold and verifier_passed:
        return "HALT"
    if iteration >= max_iter:
        return "ALTERNATIVE"
    if confidence >= rethink_threshold:
        return "RETHINK"
    return "ALTERNATIVE"


# Example: three samples agree on the answer "42"
samples = ["42", "42", "42"]
conf = consistency_confidence(samples)
action = decide_action(conf, verifier_passed=True, iteration=1)
print(f"Confidence={conf:.2f} → Action={action}")  # HALT

# Example: samples disagree
samples = ["42", "43", "40"]
conf = consistency_confidence(samples)
action = decide_action(conf, verifier_passed=False, iteration=2)
print(f"Confidence={conf:.2f} → Action={action}")  # RETHINK
```

This function is deliberately simple — it uses string equality as the agreement metric, which works for short answers but not for free-form text. In production, semantic similarity or embedding distance would replace exact matching. The core logic, however, is universal: estimate uncertainty, compare against thresholds, and choose an action that balances quality against compute cost.

## Summary

- **Verification is the foundation of reliability.** Every agent output should pass through at least one verifier — execution, reference, consistency, or multi-modal — before it is trusted.
- **Self-correction strategies differ in their feedback source.** Reflexion learns from past episodes through stored reflections. Self-Refine iterates within a single episode using self-critique. CRITIC grounds critique in external tools, making it the most reliable of the three.
- **Self-correction can loop.** Without maximum iteration bounds, accurate verifiers, and correct critiques, agents oscillate between flawed revisions or chase phantom errors. Bounding iterations and verifying the verifier are essential safeguards.
- **Confidence-guided refinement is proactive.** CoRefine uses token-level confidence to decide whether to halt, rethink, or try an alternative. This adaptive approach saves tokens and latency compared to uniform self-consistency sampling.
- **Halting is a practical necessity.** No general solution exists, but composite rules combining confidence thresholds, verifier status, and iteration limits provide effective termination guarantees.
