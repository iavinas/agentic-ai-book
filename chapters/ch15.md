# Chapter 15: Training Single Agents — SFT, RL, and Process Rewards

> In 2024, the best open-source web agent scored 8.5% on WebArena-Lite. Eighteen months later, a 7-billion-parameter model trained with reinforcement learning reached 44.8%, surpassing OpenAI o3. The difference was not a larger base model or a better prompt. It was training: not just showing the model examples of tool use, but letting it explore, fail, and learn from the outcome. Single-agent training has crossed a threshold. Supervised fine-tuning on curated trajectories gives agents the basics; RL teaches them to recover from errors; Process Reward Models tell them exactly which step went wrong. This chapter maps the modern training stack for single agents — from scaling trajectory datasets to long-horizon RL and step-level reward models — and ends with a pure-PyTorch project that trains a Process Reward Model from scratch.

---

## 1. SFT on Agent Trajectories

### 1.1 Scaling Trajectory Data: From AgentTuning to AgentBank

An agent is only as good as the trajectories it learns from. A trajectory is the record of a task: the user query, the agent's thoughts, the tools it calls, and the observations it receives. Early work like **AgentTuning** (Zhu et al., 2023) proved that fine-tuning on as few as 1,866 high-quality trajectories could turn a 7B Llama model into a capable agent, matching GPT-3.5-turbo on unseen tasks. But 1,866 trajectories cover only six task types — web shopping, embodied AI, knowledge graphs, databases, operating systems, and ALFWorld. A generalist agent needs more.

**AgentBank** (2024) scaled this paradigm by two orders of magnitude. It contains 51,287 trajectories across 16 tasks spanning five skill dimensions: reasoning, mathematics, programming, web browsing, and embodied AI. Where AgentTuning filtered out failed trajectories — creating an artificially easy training set — AgentBank introduced **Answer Forcing** and **Heuristic Action Search** to salvage hard examples. If an agent failed to reach the answer, Answer Forcing re-annotates the trajectory using the gold solution, preserving the difficulty distribution of the real world. The resulting **Samoyed** models (7B and 13B) outperform GPT-3.5-turbo on held-in tasks and generalize strongly to held-out domains.

The lesson is not merely that more data is better. It is that **diversity and difficulty calibration matter more than cleanliness**. An agent trained only on perfect trajectories learns to imitate success but not to recover from failure. Real-world agents fail constantly — APIs return errors, web pages change, and assumptions turn out wrong. Training data must reflect that noise.

### 1.2 Observation Masking and Partial Masking

When a trajectory is converted into training tokens, the model should learn to generate its own thoughts and actions, not to regurgitate observations that were produced by the environment. This is the principle of **observation masking** introduced in Chapter 7: compute the cross-entropy loss only on assistant-generated tokens (thoughts, actions, final answers) and mask everything else to zero gradient.

But masking observations is only the first filter. A second, sharper filter is **partial masking**: ignoring not just environment outputs but also the agent's own mistakes. **STeP** (Self-training via Planning, 2025) demonstrated that if a trajectory contains an incorrect reasoning step followed by a correction, the model should learn from the correction while forgetting the error. During SFT, STeP masks loss on any assistant turn that is labeled incorrect, even if it is an action the agent itself generated.

Here is a pure Python helper that implements partial masking for a ReAct trace. It extends the basic observation mask by also checking a correctness flag on every assistant turn.

```python
def apply_partial_mask(trajectory):
    """Return a list of (text, mask) pairs for a ReAct trace.

    trajectory: list of dicts with keys 'role', 'content', 'is_correct'.
        'is_correct' is None for non-assistant turns and bool for assistant turns.
    Mask = 1 means the model learns to predict this segment.
    Mask = 0 means the segment is ignored during loss computation.
    """
    masked = []
    for turn in trajectory:
        role = turn["role"]
        if role in ("user", "tool", "system"):
            mask = 0
        elif role == "assistant":
            # Learn only from assistant turns marked correct.
            mask = 1 if turn.get("is_correct", True) else 0
        else:
            mask = 0
        masked.append({"text": f"{role}: {turn['content']}", "mask": mask})
    return masked


# Example: a trace where the agent mispredicts the tool, then recovers.
trace = [
    {"role": "user", "content": "Weather in Tokyo?"},
    {"role": "assistant", "content": "I'll check the weather.", "is_correct": True},
    {"role": "assistant", "content": 'search_web(q="Tokyo sushi")', "is_correct": False},
    {"role": "tool", "content": "...irrelevant results..."},
    {"role": "assistant", "content": "That was wrong. I'll use the weather tool.", "is_correct": True},
    {"role": "assistant", "content": 'get_weather(city="Tokyo")', "is_correct": True},
    {"role": "tool", "content": "{\"temp\": 22, \"cond\": \"cloudy\"}"},
    {"role": "assistant", "content": "It is 22°C and cloudy in Tokyo.", "is_correct": True},
]

for item in apply_partial_mask(trace):
    tag = "LEARN" if item["mask"] else "MASK"
    print(f"[{tag}] {item['text']}")
```

Running this produces `MASK` on the erroneous `search_web` call and `LEARN` on the reflection and corrected action. The model internalizes recovery patterns without internalizing the mistake.

### 1.3 Synthetic Trajectories: Simia and STeP

The biggest bottleneck in agent training is not compute — it is data. Human demonstrations of expert tool use are expensive, slow to collect, and limited in diversity. The frontier solution is to generate training trajectories synthetically.

**Simia** (Li et al., 2025) removes the dependency on real environment implementations entirely. Instead of calling live APIs or running real browsers, Simia uses a strong reasoning model to simulate realistic environment feedback from a small seed set of trajectories. It offers two modes: **Simia-SFT**, which amplifies seed data into large volumes of multi-turn trajectories through LLM-based pre-filtering and rule-based sanity checks; and **Simia-RL**, which trains agents inside an LLM-simulated environment where the simulator provides both observations and rewards. A Simia-Tau 32B model surpasses GPT-4o on the customer-service benchmark Tau2-Bench, despite never interacting with a real CRM system during training.

**STeP** takes a complementary angle. Rather than simulating the environment, it improves the *quality* of trajectories by embedding reflection. A strong teacher model watches a weaker student agent interact with the real environment. When the student errs, the teacher interjects a reflection — a corrected thought and action — producing a self-reflected trajectory. These refined traces are then used for SFT with partial masking. On ALFWorld and WebShop, a LLaMA-2-7B model trained with STeP outperforms the same model trained on raw expert trajectories, using less data.

Both approaches share a crucial dependency: **the verifier**. Simia needs a rule-based or model-based checker to ensure simulated observations are coherent. STeP needs the teacher model to be genuinely stronger than the student. Synthetic data scales infinitely, but its value scales only with the quality of the oracle that filters it.

## 2. RL for Tool Use and Long-Horizon Agents

### 2.1 From Imitation to Exploration

Supervised fine-tuning teaches an agent to imitate expert behavior, but imitation has a hard ceiling: the agent cannot exceed the quality of its training data. If every trajectory in the dataset solves a web-search task in seven steps, the agent never learns that five steps is possible. Worse, SFT cannot teach recovery from errors, because the dataset contains only successful paths.

Reinforcement learning breaks this ceiling by replacing fixed demonstrations with an environment and a reward function. The agent explores the action space, receives feedback, and learns to maximize long-term return. In the context of single agents, "long-term" can mean hundreds of steps — 40 turns in AppWorld, 20+ web pages in WebArena, or 600 consecutive tool calls in OpenAI's o3 system. The challenge is no longer whether the model can generate a tool call; it is whether the model can learn *strategy* over a horizon longer than any human demonstration dataset can cover.

### 2.2 GRPO, DAPO, and Rule-Based Rewards

**GRPO (Group Relative Policy Optimization)** eliminated the separate value network that classic PPO requires. Instead of training a critic to estimate expected returns, GRPO samples a group of responses to the same prompt and computes advantages relative to the group mean and standard deviation:

$$A_i = \frac{r_i - \text{mean}(\{r_1, \dots, r_G\})}{\text{std}(\{r_1, \dots, r_G\})}$$

This halves memory overhead and simplifies the training loop, making it feasible to run RL on consumer-grade GPU clusters. DeepSeek-R1 used GRPO with only two reward signals: an **accuracy reward** (deterministic correctness checks for math or code) and a **format reward** (enforcing reasoning inside specific XML tags). They deliberately avoided neural reward models, which are expensive to train and vulnerable to reward hacking at scale.

By early 2025, practitioners discovered that vanilla GRPO suffers from **entropy collapse** and **advantage collapse** — when every response in a group receives the same reward, the gradient vanishes. **DAPO** (Dynamic Advantage Policy Optimization, ByteDance + Tsinghua, March 2025) fixed this with four adjustments: **clip-higher** (asymmetric clipping that preserves exploration), **dynamic sampling** (rejecting batches with zero reward variance), **token-level policy gradient loss** (ensuring long reasoning chains receive proportional gradient weight), and **overlong reward shaping** (a soft penalty that prevents runaway response length). On Qwen2.5-32B, DAPO reached 50 AIME 2024 points — three above DeepSeek-R1-Zero-Qwen-32B — using 50% fewer training steps.

Here is a minimal, pure Python rule-based reward function suitable for a math or code agent. It checks answer correctness with a deterministic verifier and format compliance with a simple regex.

```python
import re

def rule_based_reward(predicted, ground_truth, reasoning_text):
    """Return a scalar reward for an agent's response.

    predicted: the model's final answer (string).
    ground_truth: the correct answer (string).
    reasoning_text: the full reasoning trace (string).
    """
    reward = 0.0

    # Accuracy reward: exact match after stripping whitespace.
    if predicted.strip() == ground_truth.strip():
        reward += 1.0

    # Format reward: reasoning must be wrapped in <think> tags.
    if re.search(r"<think>.*?</think>", reasoning_text, re.DOTALL):
        reward += 0.2

    # Error-avoidance shaping: small bonus for admitting uncertainty.
    if "unsure" in reasoning_text.lower() or "verify" in reasoning_text.lower():
        reward += 0.05

    return reward


# Example usage.
print(rule_based_reward("42", "42", "<think>Let me compute...</think>"))
# 1.25
print(rule_based_reward("42", "43", "<think>I guess...</think>"))
# 0.25
```

Rule-based rewards are not a limitation — they are a design choice. Where correctness is programmatically verifiable (math, code, structured extraction), deterministic rewards are cheaper, faster, and unhackable. Where correctness is open-ended, LLM-as-judge rewards fill the gap, but they introduce latency, cost, and the risk of sycophantic escalation (the policy learns to please the judge rather than solve the task).

### 2.3 WebAgent-R1: End-to-End RL for Web Agents

Most RL research for LLMs has focused on single-turn reasoning: one prompt, one answer, one reward. Agents are multi-turn by nature. **WebAgent-R1** (Wei et al., 2025) was the first system to apply end-to-end RL across full web-browsing episodes. The agent receives a binary reward only at episode termination — task success or failure — and must learn from sparse signal distributed over 20+ turns.

The key innovation is **asynchronous trajectory generation**. Rather than generating one trajectory at a time and waiting for the environment to respond, WebAgent-R1 runs multiple agent-environment interactions in parallel, dramatically improving exploration. The authors also studied initialization: a model behavior-cloned on long chain-of-thought data (**WebAgent-R1-CoT**) outperforms a raw base model (**WebAgent-R1-Zero**) by a wide margin, confirming that a reasoning-heavy warm-up matters for long-horizon RL.

The results are striking. A LLaMA-3.1-8B model trained with WebAgent-R1 achieves 44.8% on WebArena-Lite, up from 8.5% baseline — a 5.3× relative improvement that surpasses strong proprietary models including OpenAI o3. The takeaway is that for web agents, **test-time scaling works through interaction depth**: allowing the agent more turns at inference time directly improves success rate, and RL is the mechanism that teaches it to use those turns wisely.

### 2.4 Apple LOOP: Memory-Efficient PPO for Long-Horizon Agents

If GRPO eliminates the value network, **Apple LOOP** (Leave-One-Out Proximal Policy Optimization, 2025) eliminates memory bloat from the entire PPO pipeline. Training a 32B-parameter agent with standard PPO typically requires multiple copies of the model in GPU memory: one for the policy, one for the value function, one for the reference, and one for gradients. LOOP maintains exactly **one copy** of the underlying LLM, making it as memory-efficient as ordinary fine-tuning.

LOOP replaces the learned critic with a **leave-one-out baseline**. For a group of responses, it estimates the value of a response by averaging the rewards of all *other* responses in the group. This requires no separate value network, no extra parameters, and no auxiliary training loop. The best-performing variant uses **per-token clipping**, which is more stable than per-trajectory or per-turn weights for agents that generate thousands of tokens across dozens of turns.

The benchmark is **AppWorld**, a stateful multi-domain environment with 9 apps and 457 API endpoints. An agent must interact via direct Python API calls across up to 40 turns and 32,000 tokens. Trained on only 24 scenarios (72 tasks), a Qwen2.5-32B-Instruct agent with LOOP achieves 71.3% task goal completion on AppWorld Test-Normal and 45.7% on Test-Challenge — outperforming OpenAI o1 by 9 percentage points.

Beyond the headline numbers, LOOP produced emergent behaviors that no human demonstration dataset contained: a 60% increase in consulting API documentation, a 30× reduction in unwarranted assumptions, and a 3× improvement in recovering from API errors. The agent learned these behaviors because the reward signal was simple — task completion — and the environment was complex enough that the only reliable path to reward was thoroughness.

## 3. Process Reward Models

### 3.1 The Failure of Outcome Rewards

Imagine debugging a twenty-step Python program with only a single pass/fail signal at the end. If the program fails, you know something is wrong, but you have no idea which line caused the crash. This is the predicament of an agent trained with an **Outcome Reward Model (ORM)**. The ORM assigns +1 if the final answer is correct and 0 otherwise. For a multi-turn agent that chains ten tool calls across a web session, a binary outcome reward provides almost no gradient information about which intermediate step was wrong.

The consequence is **credit assignment failure**. A model that calls `search_web`, then `extract_text`, then `calculate` receives the same failure signal whether the error was a bad query, a missed HTML element, or a math mistake. Without step-level feedback, the model cannot localize errors and therefore cannot correct them efficiently.

### 3.2 PRM Architectures: Scoring Each Step

A **Process Reward Model (PRM)** is a secondary model — or an additional head on the base model — trained to emit a scalar reward for every intermediate step in a trajectory. The agent policy then receives dense feedback: a small reward after each tool call, each reasoning step, each web navigation action. If step 3 is wrong, the PRM flags it immediately, and the policy can adjust before steps 4 through 20 compound the error.

The practical obstacle has always been data. Human annotators can label whether a final answer is correct, but asking them to judge whether step 7 of a 15-step reasoning trace was sound is expensive and slow. Until recently, PRMs were a theoretical luxury. That changed in 2025.

### 3.3 PRIME: Implicit Process Rewards from Outcome Labels

**PRIME** (Process Reinforcement through Implicit Rewards, 2025) showed that step-level rewards can be derived from outcome labels alone — no human step annotations required. The core idea is to train an **implicit PRM** initialized from the same base model used for the agent policy. During RL, both the policy and the implicit PRM are updated online using policy rollouts and outcome verifiers.

The implicit PRM does not output an explicit score per step. Instead, it predicts the probability of eventual success after each prefix of the trajectory. The process reward for step $t$ is the improvement in that probability:

$$r_t = V(\text{prefix}_t) - V(\text{prefix}_{t-1})$$

where $V$ is the implicit value model and the subtraction is ordinary scalar difference. (Matrix operations inside the Transformer use juxtaposition, e.g. $W_Q h \in \mathbb{R}^{d_k}$, where $W_Q$ (shape $d_k \times d$) multiplies hidden state $h$ (shape $d$) via standard matrix multiplication.) If adding step $t$ substantially raises the chance of eventual success, the step is implicitly rewarded. If it lowers the chance, the step is penalized. Because the value model is trained only on final outcomes, the entire pipeline avoids costly human step labels while still providing dense, token-level rewards.

On Eurus-2-7B-PRIME, this approach improved average reasoning benchmark performance by 15.1% over SFT and surpassed Qwen2.5-Math-7B-Instruct using only 10% of the training data. The sample efficiency gain over outcome-reward-only RL was 2.5×.

### 3.4 AgentPRM: Actor-Critic for Multi-Turn Agents

PRMs were originally developed for math reasoning, where each step has a clear right or wrong. Agent tasks are messier. A web agent's decision to scroll before clicking is not objectively correct or incorrect; it depends on context. Two independent research groups addressed this in 2025, both under the name **AgentPRM**.

**Cornell's AgentPRM** (Choudhury, 2025) treats the PRM as a lightweight actor-critic integrated into standard RLHF pipelines. Stage 1 runs asynchronous Monte Carlo rollouts to compute Q-targets for process rewards. Stage 2 trains the PRM via supervised learning with a soft BCE loss. Stage 3 updates the agent policy with online DPO using the PRM scores. A surprising result: a Llama3.2-3B model trained with this pipeline outperforms GPT-4o and Claude-3.5-sonnet on ALFWorld.

**Fudan's AgentPRM** (Xi et al., 2025) argues that agent steps should be evaluated on two criteria: **Promise** (the probability of eventual success, i.e. the Q-value $Q^\pi(s_t, a_t)$) and **Progress** (the incremental contribution relative to the previous state, i.e. the advantage $A^\pi(s_t, a_t)$). They train with a combined loss on both quantities, using Temporal Difference estimation with Generalized Advantage Estimation to avoid expensive Monte Carlo rollouts. Their system achieves over 8× compute efficiency compared to baseline ORMs and supports robust test-time scaling via beam search and Best-of-N.

Both variants share a core insight: for agents, a step reward must be contextual. It is not enough to ask "Was this step correct?" You must ask "Did this step move the agent closer to the goal, given where it started?"

### 3.5 PQM: Q-Value Rankings for Theoretically Principled PRMs

**PQM** (Process Q-value Model, Li & Li, ICLR 2025) reframes process reward modeling as a ranking problem rather than a classification problem. Traditional PRMs treat each step as an independent binary label: correct or incorrect. PQM defines a Q-value function for each state-action pair as the probability of eventual success. The authors prove that under an optimal policy, Q-values should satisfy three properties:

1. **Ascend** as correct steps accumulate.
2. **Descend** as wrong steps proceed.
3. Maintain a **significant gap** between correct and incorrect trajectories.

Instead of cross-entropy loss on individual steps, PQM uses a **margin-based comparative loss** that optimizes the ranking of Q-values across full trajectories. When verifying solutions from Llama-3-70B-Instruct, PQM improved Best-of-N accuracy on MATH500 from 39.8% (with a standard BCE PRM) to 51.4% — an 11.6 point gain. Theoretically and empirically, ranking is a better objective for sequential reasoning than independent classification.

### 3.6 R-PRM: Reasoning-Driven PRMs with Trace Generation

**R-PRM** (Reasoning-Driven Process Reward Modeling, 2025) pushes the PRM from a scalar scorer to a generative critic. Instead of directly predicting a reward value for each step, R-PRM generates an explicit natural-language evaluation trace that covers five analyses:

- **Previous Steps Analysis**: reviews earlier steps for errors.
- **Now Step Analysis**: explains what the current step attempts.
- **Data Source Analysis**: verifies whether values come from the problem statement or prior steps.
- **Consistency Analysis**: checks logical coherence.
- **Calculation Analysis**: independently checks math via reverse operations.

After this analysis, the model emits a binary judgment ("Yes" or "No") on whether the step is correct. At inference time, R-PRM samples $K=10$ independent analytical trajectories per step and averages the probability of "Yes" across them, making the evaluation robust and compute-scalable.

R-PRM is remarkably data-efficient. With only 12,800 training samples, it surpasses most open-source PRMs. At 64,000 samples, it outperforms a baseline trained on 265,000 samples. When used to guide test-time scaling (Best-of-N and Greedy Guide Search), it improves accuracy by 8.4–8.6 points across six challenging math datasets. The broader lesson is that **transparent, generative verifiers outperform opaque scalar classifiers** because the reasoning trace provides both a reward and an explanation, which is invaluable for debugging agent behavior.

| PRM Variant | Training Signal | Key Innovation | Best For |
|-------------|-----------------|--------------|----------|
| PRIME | Outcome labels only | Implicit value differences → dense rewards | Scalable RL without step labels |
| AgentPRM (Cornell) | Async MC rollouts | Actor-critic with Online DPO | Multi-turn agents (ALFWorld) |
| AgentPRM (Fudan) | TD + GAE | Promise (Q) + Progress (Advantage) | Long-horizon credit assignment |
| PQM | Q-value rankings | Margin-based comparative loss | Theoretically principled verification |
| R-PRM | Self-generated traces | Generative critique + inference scaling | Transparent, interpretable evaluation |


*Table 15.1 — Five Process Reward Model paradigms. PRIME scales without step labels. AgentPRM variants adapt PRMs to messy agent tasks. PQM grounds PRMs in ranking theory. R-PRM makes the verifier generative and interpretable.*

## 4. Project — Training a PRM from Scratch

### 4.1 Problem Setup

In this project we build a minimal Process Reward Model in pure PyTorch. The model learns to predict the probability of eventual task success after every prefix of a reasoning trace. We then derive implicit process rewards by comparing consecutive prefix predictions — the PRIME approach simplified to its essence. The domain is synthetic math reasoning traces, where each step is a vector of features and the final outcome is known.

### 4.2 Data and Architecture

Each reasoning trace is a sequence of $T$ steps. Every step is represented by a fixed-size feature vector in $\mathbb{R}^{d_{\text{step}}}$ (in a real system these would be hidden states from a language model; here we use synthetic vectors with a hidden correctness pattern). The model is a causal Transformer encoder: at step $t$, the model attends only to steps $1 \dots t$ and outputs a scalar $v_t \in [0,1]$ representing the predicted probability of eventual success.

Causality is enforced with a causal mask so that step $t$ cannot "peek ahead" to step $t+1$. This mirrors the real-world constraint that a PRM must judge a step before seeing what happens next.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class PrefixValuePRM(nn.Module):
    """Causal Transformer that predicts success probability from every prefix.

    Input: (batch, seq_len, d_step) — float tensor of step features.
    Output: (batch, seq_len) — success probability after each prefix.
    """
    def __init__(self, d_step=64, d_model=128, nhead=4, num_layers=2):
        super().__init__()
        # Project raw step features into the model dimension.
        self.input_proj = nn.Linear(d_step, d_model)  # (d_model, d_step) matrix multiply
        encoder_layer = nn.TransformerEncoderLayer(
            d_model=d_model, nhead=nhead, batch_first=True
        )
        self.encoder = nn.TransformerEncoder(encoder_layer, num_layers=num_layers)
        self.value_head = nn.Linear(d_model, 1)  # (1, d_model) matrix multiply

    def forward(self, x):
        """x: (batch, seq_len, d_step). Returns (batch, seq_len)."""
        batch, seq_len, _ = x.shape
        x = self.input_proj(x)  # (batch, seq_len, d_model)

        # Causal mask: step i can only attend to steps 0..i.
        mask = nn.Transformer.generate_square_subsequent_mask(seq_len, device=x.device)

        h = self.encoder(x, mask=mask, is_causal=True)  # (batch, seq_len, d_model)
        v = self.value_head(h).squeeze(-1)  # (batch, seq_len)
        return torch.sigmoid(v)
```

The `input_proj` layer performs a standard matrix multiplication $x W_{\text{proj}}^{\top}$ (where $W_{\text{proj}} \in \mathbb{R}^{d_{\text{model}} \times d_{\text{step}}}$ and $x \in \mathbb{R}^{\text{batch} \times \text{seq} \times d_{\text{step}}}$). The `value_head` projects each step's hidden state to a scalar, and the sigmoid squashes it to a probability.

### 4.3 Training on Outcome Labels

We train the model so that every prefix prediction $v_t$ matches the final outcome $z \in \{0,1\}$. The loss is the sum of binary cross-entropies across all prefixes. Because the model is causal, it cannot cheat by looking at future steps; it must learn which patterns in the prefix indicate eventual success or failure.

```python
def generate_synthetic_traces(num_traces=256, max_steps=8, d_step=64):
    """Generate random traces where a hidden pattern signals correctness.

    We inject a 'correctness' bias into a specific feature dimension.
    Traces with positive total bias tend to succeed; negative ones fail.
    """
    traces = []
    outcomes = []
    for _ in range(num_traces):
        steps = torch.randn(max_steps, d_step)
        # Flip sign of the hidden signal for some traces to create failures.
        label = 1 if torch.rand(1).item() > 0.5 else 0
        if label == 0:
            steps[:, 0] *= -1.0  # corrupt the signal dimension
        traces.append(steps)
        outcomes.append(float(label))
    return torch.stack(traces), torch.tensor(outcomes)  # (N, T, d), (N,)


# Hyperparameters.
BATCH = 32
STEPS = 8
D_STEP = 64
EPOCHS = 200
LR = 1e-3

model = PrefixValuePRM(d_step=D_STEP)
optimizer = torch.optim.Adam(model.parameters(), lr=LR)

train_x, train_z = generate_synthetic_traces(256, STEPS, D_STEP)

def outcome_to_prefix_labels(z, seq_len):
    """Broadcast scalar outcome z to every prefix: shape (batch, seq_len)."""
    return z.unsqueeze(1).expand(-1, seq_len)

for epoch in range(EPOCHS):
    perm = torch.randperm(train_x.size(0))
    total_loss = 0.0
    for i in range(0, train_x.size(0), BATCH):
        idx = perm[i : i + BATCH]
        xb = train_x[idx]          # (batch, seq_len, d_step)
        zb = train_z[idx]          # (batch,)
        prefix_labels = outcome_to_prefix_labels(zb, STEPS)  # (batch, seq_len)

        v = model(xb)              # (batch, seq_len)
        loss = F.binary_cross_entropy(v, prefix_labels)

        optimizer.zero_grad()
        loss.backward()
        optimizer.step()
        total_loss += loss.item()

    if (epoch + 1) % 40 == 0:
        print(f"Epoch {epoch + 1}: loss = {total_loss:.4f}")
```

The training loop is intentionally simple. For each batch, the model predicts a success probability after every prefix. The target for all prefixes is the same scalar outcome — 1 for success, 0 for failure. By backpropagating the error at every prefix, we force the model to learn which early patterns predict the final result.

### 4.4 Deriving and Evaluating Process Rewards

Once trained, we derive the implicit process reward for step $t$ as the difference in predicted success probability before and after that step:

$$r_t = v_t - v_{t-1}$$

with $v_0 = 0$ by convention. A positive $r_t$ means the step improved the outlook; a negative $r_t$ means it hurt. Because the model was trained only on final outcomes, these step-level rewards emerge without any step-level labels — the PRIME principle in miniature.

```python
model.eval()
with torch.no_grad():
    # Generate a test trace that should fail (corrupted signal).
    test_x, test_z = generate_synthetic_traces(1, STEPS, D_STEP)
    test_x[0, :, 0] *= -1.0  # force failure

    v = model(test_x)        # (1, seq_len)
    v_prev = torch.cat([torch.zeros(1, 1), v[:, :-1]], dim=1)
    process_rewards = v - v_prev  # (1, seq_len)

    print("Prefix success probabilities:", v[0].tolist())
    print("Process rewards per step:     ", process_rewards[0].tolist())
```

Running this on a corrupted trace typically yields declining prefix probabilities and negative process rewards after the first few steps — the model learns that the corrupted signal pattern predicts failure, and the derived rewards correctly localize the problem to the early steps where the signal diverged.

To evaluate the PRM quantitatively, we can measure **error localization accuracy**: on a held-out set of traces where we know exactly which step was corrupted, does the most negative process reward occur at that step? In our synthetic setup, the first step is corrupted, and the trained model consistently assigns the largest negative reward to step 1, confirming that implicit process rewards can localize errors without explicit step labels.

> **💡 Key Insight**
>
> A Process Reward Model does not need human annotators to label every step. By training a causal value model to predict final outcomes from prefixes, the difference between consecutive predictions becomes a dense, automatic reward signal. This is the PRIME principle: process supervision from outcome data alone.

## Summary

- **Trajectory data is the foundation.** AgentBank scaled trajectory SFT to 50,000+ examples across 16 tasks, proving that diversity and difficulty calibration matter more than perfect cleanliness. Simia and STeP remove the human data bottleneck by synthesizing environments and embedding reflection into trajectories.
- **RL discovers what SFT cannot imitate.** WebAgent-R1 and Apple LOOP demonstrated that end-to-end RL over multi-turn episodes yields dramatic gains on web and app-automation benchmarks. GRPO and DAPO make RL feasible at scale by eliminating the value network and fixing entropy collapse.
- **Rule-based rewards are the production default** where correctness is verifiable, because they are deterministic and unhackable. LLM-as-judge rewards work for open-ended tasks but introduce cost, latency, and sycophancy risk.
- **Outcome rewards fail for long chains.** A binary pass/fail signal at the end of a 20-step episode provides no gradient for intermediate errors. Process Reward Models solve this by scoring each step.
- **PRMs are now scalable.** PRIME derives implicit process rewards from outcome labels alone. AgentPRM adapts PRMs to messy agent tasks via actor-critic training. PQM reframes step scoring as Q-value ranking. R-PRM makes the verifier generative and interpretable.
- **A PRM can be trained in pure PyTorch** as a causal prefix-value model. By predicting final outcomes from partial traces and differencing consecutive predictions, you obtain dense step-level rewards with no manual step annotations.
