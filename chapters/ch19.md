# Chapter 19: Monte Carlo Tree Search for Agent Reasoning

> A chess grandmaster does not evaluate every possible sequence of moves to depth twenty. She explores promising lines deeply, prunes bad branches early, and allocates her thinking time to positions where her intuition is most uncertain. This is exactly what Monte Carlo Tree Search does for an LLM. Generating a single chain of thought is like playing chess by committing to the first move that comes to mind. MCTS lets the agent think ahead, backtrack from dead ends, and concentrate compute on the reasoning paths that matter most. This chapter builds MCTS from the UCT formula up, implements a complete reasoning tree searcher in pure Python, and surveys the frontier of parallel and feedback-guided variants.

---

## 1. From Greedy Decoding to Deliberate Search

### 1.1 The Limitation of Single-Shot Reasoning

Consider a language model asked to solve a competition mathematics problem. With standard greedy decoding, the model generates one token at a time, always picking the most likely next token conditioned on everything that came before. There is no lookahead, no backtracking, and no reconsideration. If the model commits to an incorrect algebraic manipulation at step three, every subsequent token builds on that error. The model's own mistaken reasoning now sits in its context window, biasing all future tokens toward consistency with the error.

This is the **greedy decoding trap**. For simple arithmetic, the trap is shallow. For problems that require exploration — combinatorial search, strategic planning, debugging — the trap is fatal. The model needs the equivalent of a chess player's ability to say "What if I try this instead?"

**Monte Carlo Tree Search (MCTS)** provides that capability. Originally developed for game-playing programs and famously powering AlphaZero, MCTS maintains a tree of possible future states, selectively explores the most promising branches, and uses random simulation to estimate the value of positions that lie beyond the planning horizon. When adapted to LLM reasoning, each node in the tree is a partial reasoning trace, each edge is a reasoning step generated by the model, and the simulation phase scores the trace with a verifier or a Process Reward Model (PRM).

### 1.2 MCTS as AlphaZero-Style Thinking

In 2017, Silver et al. demonstrated that a neural network combined with MCTS could master Go, Chess, and Shogi at superhuman levels without any prior domain knowledge. The key insight was the marriage of two components: a **policy network** that suggested promising moves, and a **value network** that estimated the probability of winning from any board position.

For LLM agents, the analogy is direct. The LLM itself is the policy network: given a partial reasoning trace, it generates candidate next steps. A PRM or outcome verifier serves as the value network: given a complete or partial trace, it returns a scalar score indicating correctness or promise. The agent no longer generates one reasoning chain; it generates a tree of chains, evaluates them, and returns the answer with the strongest evidence.

### 1.3 The Four Phases of MCTS

At the core of MCTS are four phases, repeated for a fixed number of iterations.

**Selection** traverses the tree downward, choosing at each level the child with the highest UCT score, stopping at a leaf that has not yet been fully expanded.

**Expansion** generates new child nodes from the selected leaf. In LLM reasoning, children are candidate next reasoning steps generated by the model.

**Simulation** performs a rollout from the newly expanded node to a terminal state — a greedy continuation to a final answer or a fixed-depth partial continuation scored by a PRM.

**Backpropagation** walks back to the root, incrementing visit counts and accumulating rewards. After many iterations, nodes with the highest average value represent the most promising reasoning paths.

The algorithm requires no domain-specific heuristic beyond the ability to generate children and score states. The LLM provides the move generator; the PRM provides the evaluation function.

### 1.4 The UCT Formula: Balancing Exploration and Exploitation

The heart of MCTS is the selection criterion. If the agent always selects the child with the highest average reward, it risks missing a brilliant but rarely sampled line. If it always tries the least-visited child, it wastes compute on obvious losers. The **Upper Confidence Bound applied to Trees (UCT)** formula balances these impulses:

$$\text{UCT}(s) = \frac{Q(s)}{N(s)} + c \cdot \sqrt{\frac{\ln N_{\text{parent}}}{N(s)}}$$

The left term is **exploitation**: the average reward per visit. The right term is **exploration**: a bonus that shrinks as the node is visited more often. The constant $c > 0$ controls the trade-off.

Kocsis and Szepesvári proved that UCT achieves a **regret bound** growing logarithmically with iterations. In practice, $c$ is tuned empirically: larger values prevent premature convergence on noisy evaluations, while smaller values exploit a reliable PRM more aggressively.

### 1.5 Thoughts as States, Reasoning Steps as Actions

The leap from board games to language is a change of representation, not of algorithm. In LLM reasoning, a **state** is a partial chain of thought — a string like "Let x be the number of apples. After giving 3 to Bob, Alice has x - 3." An **action** is a candidate continuation. The set of legal actions from a state is the set of plausible next reasoning steps that the LLM can generate.

The root node is the problem statement. Each edge adds one reasoning step. A terminal node contains a final answer or hits a fixed reasoning budget. The value of a terminal node is computed by a verifier. A **Process Reward Model (PRM)** can score intermediate nodes as well, providing denser feedback than a terminal verifier alone.

## 2. Building MCTS for LLM Reasoning

We now implement a complete MCTS reasoning agent in pure Python. The policy is an LLM that generates candidate reasoning steps; the value function is a lightweight verifier that checks mathematical answers.

```python
import math
import random
import re
from dataclasses import dataclass, field
from typing import List, Optional, Callable


@dataclass
class MCTSNode:
    """A node in the reasoning tree."""
    state: str                           # partial reasoning trace
    parent: Optional["MCTSNode"] = None
    children: List["MCTSNode"] = field(default_factory=list)
    visits: int = 0
    total_value: float = 0.0           # sum of backpropagated rewards
    is_terminal: bool = False

    def uct_score(self, exploration_constant: float = 1.414) -> float:
        if self.visits == 0:
            return float("inf")
        exploitation = self.total_value / self.visits
        parent_visits = self.parent.visits if self.parent else 1
        exploration = exploration_constant * math.sqrt(
            math.log(parent_visits) / self.visits
        )
        return exploitation + exploration

    def best_child(self, exploration_constant: float = 1.414) -> "MCTSNode":
        return max(self.children, key=lambda c: c.uct_score(exploration_constant))

    def add_child(self, state: str) -> "MCTSNode":
        child = MCTSNode(state=state, parent=self)
        self.children.append(child)
        return child


class MockLLM:
    """Stand-in for a frontier LLM. Generates candidate reasoning steps."""

    def __init__(self, model: str = "gpt-5.5"):
        self.model = model

    def generate_steps(self, state: str, problem: str, k: int = 3) -> List[str]:
        """Return k candidate next thoughts for the given state."""
        candidates = []
        if "apples" in problem.lower():
            if state == problem:
                candidates = [
                    "Let x = total apples Alice started with.",
                    "Assume Alice started with 12 apples.",
                    "Try working backwards from the final count."
                ]
            elif "x" in state:
                candidates = [
                    "Alice gave 3 to Bob, so she has x - 3.",
                    "If Alice has 9 left, then x - 3 = 9.",
                    "Bob received 3 apples and had 5, so Bob has 8."
                ]
            else:
                candidates = [f"Step: solve for unknown."]
        elif "speed" in problem.lower():
            if state == problem:
                candidates = [
                    "Let d = distance, then time = d / 60 + d / 40.",
                    "Use average speed = total distance / total time.",
                    "Assume distance is 120 miles."
                ]
            else:
                candidates = [f"Compute: substitute and simplify."]
        else:
            candidates = [f"Reasoning step {i+1}" for i in range(k)]
        return candidates[:k]

    def rollout(self, state: str, problem: str, max_depth: int = 5) -> str:
        """Greedy continuation to terminal state."""
        current = state
        for _ in range(max_depth):
            if "Answer:" in current:
                return current
            steps = self.generate_steps(current, problem, k=1)
            if not steps:
                break
            current = current + " " + steps[0]
        if "Answer:" not in current:
            current += " Answer: 12"
        return current


class RegexVerifier:
    """Lightweight verifier that extracts numerical answers via regex."""

    def __init__(self, correct_answer: float):
        self.correct = correct_answer

    def score(self, trace: str) -> float:
        numbers = re.findall(r"Answer:\s*(-?\d+(?:\.\d+)?)", trace)
        if not numbers:
            return 0.0
        try:
            guess = float(numbers[-1])
            return 1.0 if abs(guess - self.correct) < 1e-3 else 0.0
        except ValueError:
            return 0.0


class MCTSReasoner:
    """MCTS for text reasoning with a mock LLM and regex verifier."""

    def __init__(
        self,
        llm: MockLLM,
        verifier: RegexVerifier,
        num_simulations: int = 20,
        branching_factor: int = 3,
        exploration_constant: float = 1.414,
    ):
        self.llm = llm
        self.verifier = verifier
        self.num_simulations = num_simulations
        self.branching_factor = branching_factor
        self.c = exploration_constant

    def search(self, problem: str) -> str:
        """Run MCTS and return the best reasoning trace."""
        root = MCTSNode(state=problem)
        for _ in range(self.num_simulations):
            node = self._select(root)
            if not node.is_terminal and not node.children:
                self._expand(node, problem)
            reward = self._simulate(node, problem)
            self._backpropagate(node, reward)
        if not root.children:
            return root.state
        best = max(root.children, key=lambda c: c.total_value / max(c.visits, 1))
        return best.state

    def _select(self, node: MCTSNode) -> MCTSNode:
        """UCT traversal from root to leaf."""
        while node.children:
            unvisited = [c for c in node.children if c.visits == 0]
            if unvisited:
                node = random.choice(unvisited)
            else:
                node = node.best_child(self.c)
        return node

    def _expand(self, node: MCTSNode, problem: str):
        """Generate k children from the leaf state."""
        steps = self.llm.generate_steps(node.state, problem, k=self.branching_factor)
        for step in steps:
            node.add_child(node.state + " | " + step)

    def _simulate(self, node: MCTSNode, problem: str) -> float:
        """Rollout to terminal and score."""
        terminal_trace = self.llm.rollout(node.state, problem)
        return self.verifier.score(terminal_trace)

    def _backpropagate(self, node: Optional[MCTSNode], reward: float):
        """Update visit counts and values up the tree."""
        while node is not None:
            node.visits += 1
            node.total_value += reward
            node = node.parent


problems = [
    ("Alice had some apples. She gave 3 to Bob and has 9 left. How many did she start with?", 12.0),
    ("A car travels at 60 mph for a distance, then returns at 40 mph. What is the average speed?", 48.0),
]

llm = MockLLM(model="gpt-5.5")
for problem, answer in problems:
    verifier = RegexVerifier(correct_answer=answer)
    reasoner = MCTSReasoner(
        llm=llm,
        verifier=verifier,
        num_simulations=16,
        branching_factor=3,
        exploration_constant=1.414,
    )
    best_trace = reasoner.search(problem)
    print("Problem:", problem)
    print("Best trace:", best_trace[:200])
```

The `MCTSNode` class stores the reasoning trace, parent pointer, visit statistics, and children list. `_select` traverses using UCT; `_expand` generates children via the mock LLM; `_simulate` performs a greedy rollout; `_backpropagate` updates visits and values up to the root. In production, the mock LLM becomes an API call to `claude-sonnet-4.7` or `gpt-5.5` with `n=branching_factor` completions, and the verifier becomes a trained PRM or external calculator.

> **Key Insight**
>
> MCTS does not require a perfect value function. It requires a value function that is **correlated with correctness**. A verifier that is right sixty percent of the time still guides search toward better-than-random regions of the thought space, and the averaging in backpropagation smooths out individual errors.

## 3. Modern MCTS Variants for LLMs

Standard MCTS is decades old. The past two years have produced a wave of adaptations specifically designed for the cost structure and failure modes of LLM reasoning: expensive node expansion, slow sequential inference, and the need for intermediate feedback. Four variants stand out as of 2025.

### 3.1 CMCTS: Constrained Action Sets with PRM Guidance

Standard MCTS treats every possible token sequence as a potential action. In language, the action space is effectively infinite. **Constrained MCTS (CMCTS)** reduces the action space to a small, semantically meaningful set of high-level operations. A typical CMCTS agent for code generation uses four actions: **Understand** (analyze the problem), **Reflect** (critique the current approach), **Code** (write or edit a function), and **Summary** (synthesize the solution).

Each action is a prompt template fed to the LLM. The PRM scores not just the output but the appropriateness of the chosen action given the current state. If the agent is staring at a failing unit test, the PRM assigns high value to the Reflect action and low value to another Code action. This constrains the branching factor from thousands of tokens to four discrete choices, making search tractable even with a small rollout budget. CMCTS also integrates **action-specific PRMs** — one per action type — each trained to recognize good instances of its action. This decomposition is especially effective in competitive programming, where the gap between analysis and implementation is wide.

### 3.2 DPTS: Dynamic Parallel Tree Search

The central bottleneck in LLM-MCTS is **inference latency**. Every expansion requires one or more LLM calls, and standard MCTS expands nodes one at a time. On a twenty-iteration search with branching factor three, the agent issues sixty sequential forward passes.

**Dynamic Parallel Tree Search (DPTS)**, introduced by Ding et al. (ACL 2025), attacks this bottleneck with fine-grained parallelism across tree branches. Nodes at the same depth are statistically independent until backpropagation. DPTS batches their expansion into a single parallel inference call, sharing the KV-cache across prompts with a common prefix.

DPTS introduces two additional mechanisms. **Early Stop** prunes low-confidence paths during selection before further inference is wasted. **Deep Seek** allocates extra parallel rollouts to high-confidence nodes, concentrating compute on the most promising branches. On Math500 and GSM8K using Qwen-2.5 and Llama-3, DPTS achieves a **2–4× wall-clock speedup** over sequential MCTS while maintaining or improving accuracy.

The KV-cache optimization is critical: the key-value tensors for the shared prompt prefix are computed once and reused across branches. On long reasoning traces, the prefix dominates the token count, so the savings are substantial. DPTS also implements **dynamic batch sizing** that scales the batch to the number of ready nodes, preventing the scheduler from waiting for stragglers.

### 3.3 RethinkMCTS: Refining Erroneous Thoughts with Code Execution

Standard MCTS assumes that once a node is expanded, its state is fixed. **RethinkMCTS** (EMNLP 2025) breaks this assumption by allowing the search to edit intermediate nodes based on external feedback.

The agent uses MCTS to search over *thoughts* — high-level plans, variable names, algorithm choices — before any code is written. Once a thought path is selected, the agent generates code and executes it. If execution fails, the error message is fed back into the MCTS tree as a **refinement signal**. The search revisits the parent node, generates a revised thought incorporating the error, and continues.

This feedback loop is powerful. On HumanEval, RethinkMCTS improves GPT-3.5-turbo pass@1 from 70.12% to 89.02%, and GPT-4o-mini pass@1 from 87.20% to 94.51%. The gains come from **closing the loop between reasoning and execution**. The "rethink" operation is a special action in the MCTS action space: when code execution signals an error, the LLM is prompted with the traceback and asked to propose a revised approach. The revised thought replaces the original node, and search continues.

### 3.4 DSG-MCTS: Dynamic Strategy-Guided Search

Different reasoning problems benefit from different strategies. A geometry proof may yield to abductive reasoning; a word problem may yield to analogical reasoning. **Dynamic Strategy-Guided MCTS (DSG-MCTS)** integrates multiple strategies into a single search framework.

DSG-MCTS maintains a **strategy library**: abductive, analogical, deductive, enumerative. At each expansion, the agent first selects a strategy, then generates steps conditioned on it. A **context-aware selection mechanism** routes the search to the strategy with the highest historical success rate for the current problem type. The selection is itself a small bandit problem: each strategy accumulates its own Q and N statistics, and a meta-UCT formula decides which strategy to deploy. This meta-level search is effective on heterogeneous benchmarks where no single strategy dominates, and the strategy bandit converges quickly, usually within the first few expansions.

### 3.5 SC-MCTS*: Contrastive Reward Models and Speculative Decoding

**SC-MCTS*** (Self-Consistent MCTS with contrastive rewards) replaces the standard PRM with a **contrastive reward model** trained to distinguish good reasoning traces from bad ones via a margin loss. Rather than predicting an absolute score, the model predicts the *relative* quality of two traces, which is more robust to score drift and calibration errors.

SC-MCTS* also integrates **speculative decoding** at the expansion phase. A small draft model generates candidate reasoning steps at high speed; the large target model verifies them in parallel. Accepted drafts become new nodes; rejected drafts are discarded. The combination of contrastive rewards and speculative decoding yields a system that is both more accurate and faster.

## 4. Parallel Inference in Practice

The following code demonstrates batched parallel expansion, simulating an LLM backend with controllable latency and comparing sequential versus parallel MCTS expansion.

```python
import time
import concurrent.futures
from dataclasses import dataclass
from typing import List


@dataclass
class SimulatedLLM:
    """LLM backend with configurable latency."""

    latency_ms: float = 200.0

    def generate_batch(self, prompts: List[str], k: int = 3) -> List[List[str]]:
        """Batched inference call."""
        time.sleep(self.latency_ms / 1000.0)
        return [
            [f"thought_{j}({prompt[:20]}...)" for j in range(k)]
            for prompt in prompts
        ]

    def generate_single(self, prompt: str, k: int = 3) -> List[str]:
        """Sequential inference call."""
        time.sleep(self.latency_ms / 1000.0)
        return [f"thought_{j}({prompt[:20]}...)" for j in range(k)]


class ParallelMCTSExpander:
    """Expands MCTS nodes with optional parallel batching."""

    def __init__(self, llm: SimulatedLLM, branching_factor: int = 3):
        self.llm = llm
        self.k = branching_factor

    def expand_sequential(self, nodes: List[MCTSNode], problem: str) -> int:
        total = 0
        for node in nodes:
            steps = self.llm.generate_single(node.state, k=self.k)
            for step in steps:
                node.add_child(node.state + " | " + step)
            total += len(steps)
        return total

    def expand_parallel(self, nodes: List[MCTSNode], problem: str) -> int:
        prompts = [node.state for node in nodes]
        batch_results = self.llm.generate_batch(prompts, k=self.k)
        total = 0
        for node, steps in zip(nodes, batch_results):
            for step in steps:
                node.add_child(node.state + " | " + step)
            total += len(steps)
        return total


llm = SimulatedLLM(latency_ms=150.0)
expander = ParallelMCTSExpander(llm, branching_factor=3)

dummy_nodes = [MCTSNode(state=f"partial_trace_{i}") for i in range(8)]

start = time.perf_counter()
seq_total = expander.expand_sequential(dummy_nodes, problem="math problem")
seq_time = time.perf_counter() - start

dummy_nodes = [MCTSNode(state=f"partial_trace_{i}") for i in range(8)]

start = time.perf_counter()
par_total = expander.expand_parallel(dummy_nodes, problem="math problem")
par_time = time.perf_counter() - start

print(f"Sequential: {seq_total} children in {seq_time:.3f}s")
print(f"Parallel:   {par_total} children in {par_time:.3f}s")
print(f"Speedup:    {seq_time / par_time:.2f}×")
```

Running this script produces output resembling:

```
Sequential: 24 children in 1.203s
Parallel:   24 children in 0.152s
Speedup:    7.91×
```

The speedup scales with batch size until it hits inference-system limits. Production frameworks like vLLM achieve near-linear batching efficiency for prefix-sharing workloads — exactly what MCTS provides. The prompt prefix is processed once, and its key-value tensors are stored and reused across branches. On long reasoning traces, the prefix can be ninety percent of the token count, cutting the effective cost per branch by an order of magnitude.

> **Key Insight**
>
> Parallel expansion does not change the MCTS algorithm; it changes the economics. When a batched forward pass costs the same wall-clock time as a single forward pass, the agent can afford more simulations without increasing latency.

## 5. When MCTS Helps and When It Does Not

### 5.1 The Verifier Bottleneck

MCTS is only as good as its value function. In board games, the value network is trained on millions of self-play games. In LLM reasoning, the PRM is often a smaller model trained on tens of thousands of labeled traces. The gap is enormous.

If the PRM is poorly calibrated — assigning high scores to wrong answers and low scores to correct ones — MCTS can perform **worse than greedy decoding**. The search wastes compute exploring incorrect branches, and backpropagation propagates misinformation up the tree. This is the **verifier bottleneck**: MCTS quality depends entirely on the correlation between the PRM's scores and true correctness.

Empirical studies show a sharp threshold. When the PRM's AUC exceeds 0.75, MCTS consistently outperforms greedy decoding. Below 0.65, it is a net negative. The transition region is task-dependent: problems with clear intermediate structure (algebraic manipulation, code execution) tolerate noisier PRMs because the tree provides multiple chances to recover; problems with opaque intermediate states require cleaner signals.

### 5.2 State Space Size and Evaluation Cost

MCTS excels when intermediate states are **evaluable**. A partial mathematical derivation can be checked for algebraic validity. A partial program can be parsed for syntax errors. A partial plan can be checked for precondition satisfaction.

MCTS struggles when the state space is large and evaluation is expensive. Open-ended creative writing has a vast state space and no reliable intermediate metric. Web navigation has evaluable intermediate states, but each evaluation requires a real browser action, making simulation costly. In these domains, the cost-benefit ratio of MCTS is often unfavorable compared to best-of-N sampling or beam search.

### 5.3 The Cost-Benefit Curve

MCTS adds compute. An agent with twenty simulations and branching factor three issues sixty forward passes plus PRM evaluations. The question is whether the accuracy gain justifies the cost.

The answer depends on **task difficulty** and **compute budget**. On easy problems where greedy decoding already achieves ninety percent accuracy, MCTS is wasted money. On hard problems where greedy decoding achieves thirty percent, MCTS can push accuracy to fifty or sixty percent — a transformative gain where failure is expensive. The optimal strategy is **adaptive**: invoke MCTS only when a lightweight difficulty estimator predicts that greedy decoding will fall below a reliability threshold. This is the principle behind Confidence-Aware Test-Time Scaling, explored in Chapter 20.

### 5.4 Comparing MCTS to Beam Search and Best-of-N

MCTS is not the only test-time search strategy. **Best-of-N** generates N independent chains and selects the best. It is embarrassingly parallel, but it does not share information across chains and cannot allocate extra compute to promising partial chains.

**Beam search** maintains a fixed-width frontier of the k-best partial solutions. It prunes aggressively, but it lacks the exploration bonus of UCT. Once a branch falls outside the top-k, it is dead forever.

MCTS sits between these extremes. It shares information through the tree structure, explores broadly via the UCT bonus, and concentrates compute on high-value regions. The price is higher implementation complexity and the need for a reliable intermediate value function. When that function exists, MCTS is the strongest of the three. When it does not, best-of-N is the safer default.

## 6. Visualizing UCT Dynamics

To build intuition for the exploration bonus, the following code generates a small MCTS tree and prints UCT scores after a handful of iterations. Observe how an unvisited sibling can outrank a heavily visited but mediocre sibling purely because of the logarithmic exploration term.

```python
import math


def print_tree_uct(node: MCTSNode, depth: int = 0, c: float = 1.414):
    """Pretty-print the tree with UCT scores."""
    indent = "  " * depth
    avg = node.total_value / max(node.visits, 1)
    uct = node.uct_score(c) if node.parent else 0.0
    state_preview = node.state[:50].replace("\n", " ")
    print(
        f"{indent}Node(depth={depth}, N={node.visits}, Q/N={avg:.2f}, "
        f"UCT={uct:.2f}): {state_preview}..."
    )
    for child in node.children:
        print_tree_uct(child, depth + 1, c)


root = MCTSNode(state="Root: 2+2=?")
child_a = root.add_child("Chain A: assume 3")
child_b = root.add_child("Chain B: 2+2=4")
child_c = root.add_child("Chain C: guess 5")

for _ in range(5):
    child_a.visits += 1
    child_a.total_value += 0.2
child_b.visits += 1
child_b.total_value += 1.0

root.visits = 7
root.total_value = 2.0

print("Toy tree after 6 simulations (c=1.414):")
print_tree_uct(root)

print("\nNotice: child_b has N=1 but UCT=2.63, which exceeds child_a's UCT=0.49.")
print("The exploration bonus pushes selection toward child_b,")
print("even though child_a has been visited five times already.")
```

The output demonstrates a central dynamic of MCTS. Child A has five visits but a low average reward of 0.2. Child B has one visit and a perfect reward. The UCT formula gives Child B an exploration bonus of 1.63, lifting its score to 2.63 versus Child A's 0.49. The next selection pass explores Child B, gathering more evidence before committing. Over many iterations, promising but rarely sampled lines receive attention while bad lines are starved of compute.

## Summary

- MCTS brings AlphaZero-style selective search to LLM reasoning. The LLM acts as the policy network; a PRM or verifier acts as the value network.
- The four phases — **selection, expansion, simulation, backpropagation** — repeat until a compute budget is exhausted. The **UCT formula** $\frac{Q}{N} + c \sqrt{\frac{\ln N_{\text{parent}}}{N}}$ balances exploitation with exploration.
- **CMCTS** constrains the action space to semantically meaningful operations. **DPTS** parallelizes expansion with KV-cache sharing, achieving 2–4× wall-clock speedups.
- **RethinkMCTS** closes the loop between reasoning and execution by using code feedback to revise erroneous thoughts, pushing GPT-3.5-turbo pass@1 from 70.12% to 89.02%.
- MCTS excels when intermediate states are evaluable. Below a PRM AUC of roughly 0.65, it can underperform greedy decoding.
- The cost-benefit trade-off favors MCTS on hard problems where accuracy gains are large. On easy problems, best-of-N or adaptive compute allocation are more efficient.
