# Chapter 10: Code Generation Agents

> In 2023, the best language models could write a function that sorted a list, but they could not open a repository, read the existing tests, patch a bug, run the test suite, and submit a pull request. The distance between a code snippet and a shipped fix is measured not in tokens but in actions: reading files, reasoning about dependencies, editing precisely, observing failure, and trying again. This chapter closes that gap. By the end, you will build a complete code agent in pure Python that plans, edits, tests, and debugs autonomously — and you will understand why the agents that dominate SWE-bench treat code generation as a search process, not a creative writing task.

---

## 1. The Code Agent Paradigm

### 1.1 From Program Synthesis to Agentic Coding

Program synthesis is an old dream: describe what you want in English, receive working code. Early neural approaches treated it as a translation problem — natural language in, code out, done. Codex (Chen et al., 2021) and StarCoder (Li et al., 2023) pushed this paradigm to impressive accuracy on synthetic benchmarks, yet the resulting code was a static artifact. It could not be verified against a real test suite, could not adapt when the compiler complained, and could not explore a foreign codebase to find where to inject a fix.

The **code interpreter pattern** changed the framing. Instead of generating code in a single forward pass, the model writes a snippet, executes it in a sandbox, observes the output, and iterates. The loop is deceptively simple: generate, execute, observe, refine. But it transforms code generation from prediction into interaction. The agent is no longer guessing the right answer; it is probing the environment and learning from ground-truth feedback.

```mermaid
flowchart TD
    A[Task Description] --> B[Plan]
    B --> C[Generate Code]
    C --> D[Execute / Test]
    D --> E{Pass?}
    E -->|Yes| F[Return Solution]
    E -->|No| G[Observe Error]
    G --> H[Debug / Refine]
    H --> C
```


*Figure 10.1 — The code interpreter loop. The agent treats generation as a hypothesis and test execution as the falsification mechanism.*


This loop is why modern code agents outperform their one-shot ancestors by an order of magnitude on real tasks. A model that generates ten candidate functions and picks the one that passes tests is not ten times smarter; it is simply willing to learn from failure.

### 1.2 Test-Driven Generation

The single technique that separates mediocre coding agents from excellent ones is **running tests after every edit and feeding the output back into the next LLM call**. Tests provide unambiguous ground truth. A traceback is not an opinion; it is a precise diagnosis of where the hypothesis diverged from reality.

Chen et al. (2023) formalized this as **self-debugging**: the model generates code, executes it, receives error messages, and generates a revised version conditioned on those errors. The intuition is straightforward. Humans debug by reading stack traces; agents should do the same. The twist is that the agent must parse unstructured error output — a `SyntaxError` on line 47, a failed assertion `assert x == 5` but `x was 3` — and map it back to a concrete edit.

Test-driven agents do not merely generate code; they generate **falsifiable hypotheses**. Each edit is followed by a test run. If the test passes, the agent moves on. If it fails, the failure becomes an observation in the ReAct trace (see Chapter 2), and the next reasoning step is devoted to repair. This is why agents with execution loops score 50-60% on SWE-bench while naive one-shot generation languishes in single digits.

> **💡 Key Insight**
>
> The test suite is the agent's external memory of correctness. Without it, the agent has no grounding; with it, the agent has a compass that points unambiguously toward the solution.

### 1.3 SWE-bench: The Gold Standard

If the code interpreter pattern gave agents a body, SWE-bench gave them a gym. SWE-bench (Jimenez et al., 2024) is a dataset of real GitHub issues drawn from popular Python repositories. Each task provides a bug report, a codebase, and a test suite. The agent must check out the repository, understand the issue, edit the code, and run the tests until the new test passes and old tests do not break.

SWE-bench has three tiers:

| Variant | Tasks | Description | Difficulty |
|---------|-------|-------------|------------|
| Verified | 500 | Human-verified, solvable without hidden traps | Moderate |
| Pro | 731 | Harder, requires deeper reasoning | Hard |
| Live | Monthly | Fresh issues, no train/test overlap | Expert |

**HumanEval** (Chen et al., 2021) remains useful as a lightweight synthetic benchmark — 164 hand-written Python problems with docstrings and tests — but it is not representative of real-world software engineering. HumanEval problems are self-contained, single-function, and lack the contextual noise of a real repository. SWE-bench is the gold standard because it forces the agent to navigate unfamiliar code, understand issue descriptions written by humans for humans, and respect existing conventions.

## 2. Agent Architectures for Coding

### 2.1 The Plan-Edit-Test-Debug Loop

The dominant architecture for code agents in 2025–2026 is a structured loop with four phases: **Plan**, **Edit**, **Test**, and **Debug**. This is not optional ornamentation; it is the minimal control structure that prevents an agent from thrashing randomly across a codebase.

In the **Plan** phase, the agent reads the task, explores the repository to identify relevant files, and proposes a concrete set of changes. A good plan names the files to modify, the nature of each modification, and the expected outcome. Without a plan, the agent risks "hopeful editing" — changing lines and praying the tests pass.

In the **Edit** phase, the agent generates the actual code modification. This is the most fragile step because the agent must respect syntax, indentation, imports, and existing API contracts.

In the **Test** phase, the agent runs the test suite (or a targeted subset) and collects structured feedback. The feedback is not a binary pass/fail; it includes traceback text, failed assertion details, and coverage gaps.

In the **Debug** phase, the agent ingests the failure, localizes the likely cause, and generates a revised edit. The loop repeats until success or a budget limit is reached.

```mermaid
flowchart TD
    subgraph Plan
        P1[Read task] --> P2[Explore repo]
        P2 --> P3[Propose changes]
    end
    subgraph Edit
        E1[Generate diff or rewrite]
    end
    subgraph Test
        T1[Run pytest / tests]
        T1 --> T2{Pass?}
    end
    subgraph Debug
        D1[Parse traceback]
        D1 --> D2[Localize bug]
        D2 --> D3[Propose fix]
    end
    Plan --> Edit
    Edit --> Test
    T2 -->|Yes| Done[Submit / Return]
    T2 -->|No| Debug
    Debug --> Edit
```


*Figure 10.3 — The Plan-Edit-Test-Debug loop. Each phase is a checkpoint; skipping Plan leads to undirected thrashing, and skipping Test removes the ground-truth signal.*


ARCS (Los Alamos National Laboratory, 2025) formalized this loop as a budgeted synthesize-execute-repair process with provable termination and monotonic improvement guarantees. The agent halts when tests pass or when a fixed iteration budget is exhausted.

### 2.2 File-Level Context Management

The hardest problem in code agents is not generating code; it is deciding **which files to read and which to edit**. A production repository may contain thousands of files, but only three or four are relevant to a given bug. The agent must navigate this haystack without exhausting its context window.

Context management strategies fall into three families:

1. **Retrieval-based.** The agent uses `grep`, `find`, or vector search to locate symbols mentioned in the issue (function names, error messages, file paths). This is fast but brittle: the relevant logic may be in a file that does not contain the keyword.

2. **Dependency-based.** The agent parses imports and call graphs to trace dependencies. If the bug report mentions `payment_processor.py`, the agent follows imports to `gateway.py`, `models.py`, and `tests/test_payment.py`.

3. **Iterative expansion.** The agent starts with a seed file, reads it, and asks the model which other files are likely needed. This is slower but more accurate because the model can infer implicit dependencies from code structure.

In practice, agents combine all three. They `grep` for the error string, read the hit, follow imports, and repeat until the model reports that it has enough context to propose an edit. Anthropic's Claude Code implements this as an explicit tool loop: the model calls `view` (read file), `search` (grep), and `edit` (modify) in sequence, accumulating context before any edit is attempted.

### 2.3 Diff Generation vs. Full-File Rewrite

Once the agent knows what to change, it must decide *how* to express the change. There are two broad strategies: emit a **diff** (a patch describing only the changed lines) or emit a **full-file rewrite** (the entire file content, including unchanged lines).

The trade-off is shaped by token economics. Output tokens cost 5-6 times more than input tokens, and cached read tokens are 50-60 times cheaper. For a 3,000-line file, a full rewrite can burn 15,000 output tokens for a one-line color change. Diffs are asymptotically cheaper because the agent emits only the changed region plus a small anchor to locate it.

However, not all diff formats are equal. Research in 2025 (Cao et al., *To Diff or Not to Diff?*) found that **unified diffs are problematic for LLMs**: fragile line offsets and `@@` headers make generation "highly unnatural." Aider's benchmark showed that switching from strict unified diff to **search-replace blocks** raised GPT-4's edit accuracy from 26% to 59% with no other changes.

The industry has converged on a practical hierarchy:

| Format | Token Cost | Reliability | Best For |
|--------|-----------|-------------|----------|
| Full rewrite | O(file length) | Highest | Files < 300 tokens |
| Search-replace block | O(search + replace) | High | Most production agents |
| Unified diff | O(changed lines) | Low-Medium | Humans, not LLMs |
| Hash-anchored edit | O(replace only) | High | Stateful agents (emerging) |

Claude Code uses `str_replace`: the model emits `old_string` and `new_string`, and the harness performs an exact match. Cursor decouples reasoning from formatting with an **apply model** — a smaller, fine-tuned model that merges a loose edit proposal into the file mechanically. Dirac (2026) introduced **single-token hash anchors** that achieve O(replacement-only) output, cutting costs by roughly 60% on benchmarked tasks.

The practical heuristic in 2026 is simple: for files under ~300 tokens, full rewrites are cheaper because anchor overhead exceeds file size. Above that threshold, search-replace or structure-aware diffs win.

### 2.4 Test Execution and Feedback Integration

The test phase is where the agent receives its ground-truth signal. But raw test output is noisy: stack traces are long, assertions may be buried in helper functions, and flaky tests can mislead the agent into chasing ghosts.

Effective agents **structure the feedback** before feeding it back to the LLM. Instead of dumping the entire `pytest` log into the context window, the harness parses the output into a compact format: failed test name, assertion expected vs. actual, and the relevant line in the agent's edit. This keeps the context window clean and focuses the model's attention on the actionable signal.

EngineersOfAI (2025) documented a production-ready pattern for test-driven agent loops. The agent runs only the tests it expects to be affected by its edit, not the full suite. If a failure occurs, the agent checks whether the same failure existed before the edit (to avoid blaming itself for pre-existing breakage). It also caps retries — Stripe caps agents at two CI retry rounds — to prevent burning tokens on diminishing returns.

Coverage-guided iteration is another emerging technique. After the agent's edit passes the obvious tests, it checks code coverage to find untested paths that the bug might still traverse. If coverage reveals a branch the agent missed, the loop continues.

## 3. Current State of Coding Agents (2025–2026)

### 3.1 Frontier Systems

The landscape of coding agents shifted dramatically between 2024 and 2026. Three systems define the frontier:

**Claude Code / Opus 4.7** (Anthropic, 2025–2026) is an agentic coding CLI built around the `str_replace` edit format and an explicit tool loop. Claude Opus 4.7 scores **~87.6% on SWE-bench Verified** and dominates the Pro category at ~64.3%. The system runs in the user's terminal, reads files via tool calls, and asks for human approval on large edits. Its strength is precision: exact-string replacement prevents the silent corruption that plagues diff-based agents.

**OpenAI Codex CLI / Symphony** (2025–2026) pairs a headless agent with an open-source orchestrator. Codex CLI runs in a sandboxed environment and can execute commands, read files, and write code. Symphony, released in April 2026, is an open-source Elixir-based orchestrator that turns an issue tracker into a continuous control plane for agents. OpenAI reported that some internal teams saw a **500% increase in landed pull requests** within three weeks of adopting Symphony, though this figure reflects ideal conditions with strong ticket hygiene and CI guardrails.

**Devin 2.0 / 2.2** (Cognition, 2025–2026) evolved from a fully autonomous agent into an **agent-native IDE experience**. Devin 2.0 (April 2025) introduced parallel multi-agent execution, interactive planning, and a $20/month entry price. Devin 2.2 (February 2026) added computer-use desktop testing, self-review with autofix, and 3× faster startup. Independent testing suggests Devin handles simple PR reviews at 70-80% success but drops to 15-30% on complex migrations. Cognition is valued at ~$4 billion, with Goldman Sachs piloting Devin alongside 12,000 human developers.

**OpenHands** (formerly OpenDevin) remains the leading open-source alternative, offering self-hostable sandboxed cloud VMs and a modular agent architecture. It is the reference implementation for researchers who need full control over the agent loop.

### 3.2 Production Deployments

Agentic coding is no longer a research curiosity; it is a production workflow at major technology companies.

**Uber** (March 2026) reported that **84% of developers** are active agentic coding users and **11% of pull requests** are opened by agents. Uber's platform, called **Minion**, accepts tasks via Slack, CLI, or web UI and returns ready GitHub PRs. Supporting agents include **Autocover** (~5,000 merged unit tests per month, saving ~21,000 developer hours), **uReview** (AI code review on ~90% of diffs), and **Shepherd** (end-to-end migration campaigns). Uber also built an **MCP Gateway** — a centralized proxy that handles auth, telemetry, and cost attribution for every AI agent. AI development costs have grown 6× since 2024, but adoption was driven by bottom-up peer sharing rather than top-down mandates.

**Stripe** (early 2026) built **Minions**, a fully unattended coding agent system producing **over 1,000 merged pull requests per week** that contain no human-written code (though they are human-reviewed). Minions are built on a fork of Block's open-source agent **Goose**, customized to interleave LLM loops with deterministic steps. Agents run in isolated, pre-warmed devboxes that spin up in about 10 seconds against Stripe's monorepo. An internal MCP server called **Toolshed** hosts 400+ tools. Local linting runs in under 5 seconds, and CI selectively runs tests from a pool of over 3 million tests. Agents are capped at two CI retry rounds.

**Shopify** (April 2026) replaced the closed frontier model powering Shopify Flow with a **fine-tuned Qwen3-32B** open-weights agent. The result is **68% cheaper** inference and **2.2× faster** latency. Shopify closes the production-quality gap with a weekly retraining flywheel: production conversations are tagged, scored by an LLM judge, and high-quality sessions are fed back into the training set every week on two H200 GPU nodes.

### 3.3 The Generalist vs. Specialist Debate

A central question in 2026 is whether a single agent can handle all coding tasks, or whether the future belongs to **specialized agents** — one for bug fixes, one for migrations, one for test generation, one for code review.

The evidence is mixed. Frontier generalist agents like Claude Opus 4.7 and GPT-5.5 score >85% on SWE-bench Verified, suggesting that a single model can span a wide range of tasks given enough reasoning depth. But production deployments tell a different story. Stripe's Minions are optimized for well-scoped, repetitive tasks; Devin is described by analysts as a "super-intern" rather than a peer teammate. Uber's Shepherd handles migrations that would strain a generalist agent's context window and planning horizon.

The emerging consensus is **hierarchical specialization**: a generalist planner decomposes the task, then dispatches specialist executors. This mirrors the multi-agent patterns we will explore in Chapter 27, but applied within a single coding workflow. For now, the lesson is that the generalist-specialist debate is not either-or; it is about where to draw the boundary between planning and execution.

## 4. Building a Code Agent from Scratch

### 4.1 Architecture and Tools

We now build a minimal but complete code agent in pure Python. The agent follows the Plan-Edit-Test-Debug loop, uses search-replace edits, runs tests with pytest, and halts when tests pass or a budget is exhausted.

The agent exposes six tools:
- `read_file(path)` — returns file contents.
- `write_file(path, content)` — overwrites a file.
- `search_replace(path, old_string, new_string)` — exact-match edit.
- `shell(command, timeout)` — runs a shell command with timeout.
- `python_exec(code, timeout)` — runs Python code in a subprocess and returns stdout/stderr.
- `pytest_run(test_path)` — runs pytest on a file or directory and returns structured results.

We implement the `CodeAgent` class with a loop that accumulates a trace of observations, reasoning steps, and actions.

We implement the core `CodeAgent` class with tool bindings and loop control.

```python
import subprocess, json, os, textwrap, shlex
from dataclasses import dataclass, field
from typing import List, Dict, Optional, Tuple

@dataclass
class Observation:
    tool: str
    input: Dict
    output: str
    error: bool = False

class CodeAgent:
    def __init__(self, llm_backend, workspace: str, max_steps: int = 10):
        self.llm = llm_backend          # callable: prompt -> response string
        self.workspace = workspace      # root directory for all file operations
        self.max_steps = max_steps
        self.trace: List[Observation] = []
        self.plan: Optional[str] = None

    def read_file(self, path: str) -> str:
        full = os.path.join(self.workspace, path)
        try:
            with open(full, "r") as f:
                return f.read()
        except Exception as e:
            return f"Error reading {path}: {e}"

    def write_file(self, path: str, content: str) -> str:
        full = os.path.join(self.workspace, path)
        os.makedirs(os.path.dirname(full), exist_ok=True)
        with open(full, "w") as f:
            f.write(content)
        return f"Wrote {len(content)} characters to {path}"

    def search_replace(self, path: str, old_string: str, new_string: str) -> str:
        full = os.path.join(self.workspace, path)
        try:
            with open(full, "r") as f:
                text = f.read()
        except Exception as e:
            return f"Error reading {path}: {e}"
        if old_string not in text:
            return f"Error: old_string not found in {path}"
        if text.count(old_string) > 1:
            return f"Error: old_string appears {text.count(old_string)} times in {path}; match must be unique"
        text = text.replace(old_string, new_string, 1)
        with open(full, "w") as f:
            f.write(text)
        return f"Edited {path}"

    def shell(self, command: str, timeout: int = 30) -> str:
        try:
            result = subprocess.run(
                shlex.split(command),
                capture_output=True, text=True, timeout=timeout,
                cwd=self.workspace
            )
            out = result.stdout + "\n" + result.stderr
            return out.strip() if out.strip() else "(no output)"
        except subprocess.TimeoutExpired:
            return f"Error: command timed out after {timeout}s"
        except Exception as e:
            return f"Error: {e}"

    def python_exec(self, code: str, timeout: int = 10) -> str:
        return self.shell(f"python3 -c {shlex.quote(code)}", timeout=timeout)

    def pytest_run(self, test_path: str) -> Dict:
        cmd = f"pytest {test_path} -q --tb=short"
        raw = self.shell(cmd, timeout=60)
        # Structured parse: count passes/failures/errors
        passed = raw.count(" passed")
        failed = raw.count(" failed")
        errors = raw.count(" error")
        return {
            "raw": raw,
            "passed": passed,
            "failed": failed,
            "errors": errors,
            "all_pass": failed == 0 and errors == 0 and "passed" in raw
        }
```

The `search_replace` tool enforces exact, unique matches. This design choice trades flexibility for safety: the agent cannot silently corrupt a file by matching the wrong occurrence of a string.

### 4.2 The Plan-Edit-Test-Debug Loop

The agent loop prompts the LLM with the task, available tools, and the full trace of past observations. The LLM responds with a JSON action specifying which tool to call and with what arguments. The harness executes the tool, appends the observation to the trace, and repeats.

We implement the loop with a system prompt that constrains the agent to the structured Plan-Edit-Test-Debug workflow.

```python
    SYSTEM_PROMPT = textwrap.dedent("""\
        You are a code generation agent. You solve programming tasks by following
        a strict loop: Plan -> Edit -> Test -> Debug.

        Available tools (respond with JSON only):
        - {"tool": "read_file", "path": "relative/path.py"}
        - {"tool": "write_file", "path": "relative/path.py", "content": "..."}
        - {"tool": "search_replace", "path": "relative/path.py", "old_string": "...", "new_string": "..."}
        - {"tool": "shell", "command": "..."}
        - {"tool": "python_exec", "code": "..."}
        - {"tool": "pytest_run", "test_path": "..."}
        - {"tool": "finish", "answer": "..."}

        Rules:
        1. First, read relevant files and write a short plan.
        2. Make edits with search_replace when possible; use write_file only for new files.
        3. Run tests after every meaningful edit.
        4. If tests fail, read the error, reason about the cause, and fix it.
        5. Prefer small, testable edits over large rewrites.
    """)

    def run(self, task: str) -> str:
        history = f"Task: {task}\n"
        for step in range(self.max_steps):
            prompt = self.SYSTEM_PROMPT + "\n---\nTrace so far:\n" + history + "\nNext action (JSON):"
            response = self.llm(prompt)
            # Parse JSON action (production code would use stricter JSON parsing)
            try:
                action = json.loads(response.strip().split("\n")[0])
            except Exception:
                # Fallback: treat raw text as a reasoning step
                history += f"\nStep {step}: (reasoning) {response[:500]}"
                continue

            tool = action.get("tool")
            if tool == "finish":
                return action.get("answer", "Done")

            # Execute tool
            if tool == "read_file":
                out = self.read_file(action["path"])
            elif tool == "write_file":
                out = self.write_file(action["path"], action["content"])
            elif tool == "search_replace":
                out = self.search_replace(action["path"], action["old_string"], action["new_string"])
            elif tool == "shell":
                out = self.shell(action["command"], timeout=action.get("timeout", 30))
            elif tool == "python_exec":
                out = self.python_exec(action["code"], timeout=action.get("timeout", 10))
            elif tool == "pytest_run":
                out = str(self.pytest_run(action["test_path"]))
            else:
                out = f"Unknown tool: {tool}"

            obs = Observation(tool=tool, input=action, output=out, error="Error" in out)
            self.trace.append(obs)
            history += f"\nStep {step}: {json.dumps(action)} -> {out[:800]}"

            # Early stop if pytest reports all_pass
            if tool == "pytest_run" and "'all_pass': True" in out:
                return f"Tests passed after {step + 1} steps."

        return f"Max steps ({self.max_steps}) reached. Last observation: {self.trace[-1].output if self.trace else 'None'}"
```

The loop is intentionally simple: no framework abstractions, no hidden state beyond the trace. This is the same ReAct pattern from Chapter 6, specialized for code. The system prompt enforces planning before editing and testing after editing, which prevents the undirected thrashing that naive agents exhibit.

### 4.3 Evaluation Harness

To measure the agent's capability, we build a lightweight evaluation harness that runs it on a set of synthetic coding tasks and reports pass rate, syntax-error recovery rate, and average steps per task.

We define a benchmark task as a JSON object with a description, a set of seed files, and a test file. The harness creates a temporary workspace, writes the seed files, runs the agent, and checks whether the tests pass.

```python
import tempfile, shutil, statistics
from typing import Callable

class CodeBenchmark:
    def __init__(self, tasks: List[Dict]):
        self.tasks = tasks

    def evaluate(self, llm_backend: Callable[[str], str]) -> Dict:
        results = []
        for task in self.tasks:
            with tempfile.TemporaryDirectory() as tmpdir:
                # Write seed files
                for rel_path, content in task.get("files", {}).items():
                    full = os.path.join(tmpdir, rel_path)
                    os.makedirs(os.path.dirname(full), exist_ok=True)
                    with open(full, "w") as f:
                        f.write(content)
                # Run agent
                agent = CodeAgent(llm_backend, workspace=tmpdir, max_steps=12)
                final = agent.run(task["description"])
                # Evaluate
                test_path = os.path.join(tmpdir, task["test_file"])
                pytest_out = agent.pytest_run(test_path)
                all_pass = pytest_out["all_pass"]
                has_syntax_error = any("SyntaxError" in o.output for o in agent.trace)
                recovered = has_syntax_error and all_pass
                results.append({
                    "task": task["name"],
                    "pass": all_pass,
                    "steps": len(agent.trace),
                    "syntax_error": has_syntax_error,
                    "recovered": recovered,
                    "final": final,
                })

        passes = sum(1 for r in results if r["pass"])
        recoveries = sum(1 for r in results if r["recovered"])
        avg_steps = statistics.mean(r["steps"] for r in results) if results else 0
        return {
            "total": len(results),
            "pass_rate": passes / len(results) if results else 0.0,
            "syntax_recovery_rate": recoveries / len(results) if results else 0.0,
            "avg_steps": avg_steps,
            "details": results,
        }
```

The harness reports three metrics:
- **Pass rate**: the fraction of tasks where the agent's final code passes all tests.
- **Syntax recovery rate**: the fraction of tasks where the agent emitted at least one syntactically invalid edit but eventually recovered and passed the tests. This measures the debug loop's effectiveness.
- **Average steps**: the mean number of tool calls per task. Lower is better, but only if the pass rate remains high.

A minimal benchmark task might ask the agent to implement a function and provide a pytest file that asserts its correctness. For example:

```python
{
    "name": "list_reverse",
    "description": "Implement reverse_list(items) that returns a reversed copy without mutating the input.",
    "files": {
        "solution.py": "def reverse_list(items):\n    pass\n",
    "test_solution.py": "from solution import reverse_list\ndef test_reverse():\n    assert reverse_list([1,2,3]) == [3,2,1]\n"
    },
    "test_file": "test_solution.py"
}
```

Running the benchmark on a small suite of ten tasks gives a quantitative profile of the agent's strengths and weaknesses. Tasks that require planning across multiple files will stress the context-management strategy; tasks with edge-case failures will stress the debug loop; tasks with strict typing will stress the search-replace precision.

> **⚠️ Warning**
>
> A common failure mode in code agents is "edit amnesia" — the agent reads a file, plans a change, generates an edit, and then forgets that the file has changed when planning the next edit. Always include the latest file contents (or a diff summary) in the context window after every edit. Stateless harnesses that re-read files before each planning step avoid this pitfall at the cost of extra token usage.

## Summary

- Code generation is best understood as an iterative search process, not a one-shot creative task. The Plan-Edit-Test-Debug loop provides the minimal control structure needed for reliable autonomous coding.
- Tests are the agent's ground-truth compass. Agents that execute tests and feed structured failure output back into the loop outperform one-shot generators by an order of magnitude on real benchmarks.
- SWE-bench Verified / Pro / Live is the gold standard for evaluating code agents because it forces navigation of real codebases, human-written issue descriptions, and existing test suites. HumanEval remains useful for quick sanity checks but is not representative of production software engineering.
- The diff-vs-rewrite trade-off is governed by token economics: search-replace blocks are the practical default for files over ~300 tokens, while full rewrites are cheaper for very small files. Unified diffs are an anti-pattern for LLMs due to fragile line offsets.
- Production deployments at Uber, Stripe, and Shopify demonstrate that agentic coding is already a mainstream workflow, but success requires strong harness engineering: deterministic tool execution, CI guardrails, fast linting, and token-budget discipline.
