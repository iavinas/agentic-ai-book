# Chapter 7: Pre-training and Fine-Tuning for Agency

> In 2023, GPT-4 could describe a Python function but could not reliably execute it. It could explain how to book a flight but could not actually visit a booking site. The gap was not intelligence — it was training. The model had learned to predict text, not to act. Then researchers began fine-tuning models on trajectories of real tool use: sequences of thought, action, and observation that teach an LLM not just what to say but what to do. The results were dramatic. Fine-tuned 7B models surpassed hundred-billion-parameter generalists at API calling. Reinforcement learning turned o3 into a system that chains 600 tool calls without human prompting. This chapter traces how we move from a text predictor to an agent: what pre-training provides, what fine-tuning adds, and what reinforcement learning unlocks.

---

## 1. Pre-trained LLMs as Agent Foundations

### Why pre-trained models matter

A language model pre-trained on internet-scale text has already absorbed an enormous amount of implicit knowledge about the world. It knows that Python has a `json` module, that HTTP requests return status codes, that files have paths, and that search engines exist. This knowledge is not structured as an API reference — it is compressed into billions of parameters as statistical patterns — but it is there. When we turn such a model into an agent, we are not starting from zero. We are starting from a system that already understands causality, syntax, and common sense.

The key insight is that **agency is built on top of general competence**. A model that cannot write coherent prose will not suddenly learn to use tools effectively. A model that confuses Python with JavaScript will generate broken tool calls. Pre-training gives us the substrate: world knowledge, reasoning patterns, code understanding, and linguistic fluency. Fine-tuning then specializes that substrate toward action.

### The scaling hypothesis

One of the most robust findings in modern AI is that larger models exhibit stronger emergent capabilities — abilities that appear abruptly at scale rather than improving gradually. Agency is no exception. A 1B parameter model might struggle to follow a two-step instruction. A 7B model can use a single tool if carefully prompted. A 70B model can chain multiple tools, recover from errors, and adapt to new tool descriptions on the fly. This is the **scaling hypothesis for agents**: more parameters yield not just better text but better action.

But scale is not the only variable. The *quality* of pre-training data matters enormously. Models trained on code (like CodeLlama, DeepSeek-Coder, or Qwen-Coder) show dramatically stronger tool-use capabilities even at smaller sizes, because API calls and function signatures are abundant in their training corpus. A 7B code-trained model often outperforms a 13B generalist at tool-calling tasks. This tells us that the *composition* of pre-training data shapes agentic potential as much as scale does.

### In-context learning as zero-shot agency

Before any fine-tuning, a pre-trained model already exhibits a remarkable form of agency: **in-context learning**. Show the model a few examples of how to use a calculator tool, describe the tool with a JSON schema, and ask it to solve a math problem. It will often generate the correct tool call format on the first try, without a single weight update. This is zero-shot agency — no training, just prompts and tools.

The mechanism is simple but powerful. The model has seen countless API documentation, Stack Overflow posts, and tutorial examples during pre-training. When we present a tool description in the same format, the model recognizes the pattern and generates a plausible completion. It is pattern matching at scale, and it works surprisingly well for simple tasks.

However, in-context learning has sharp limits. The model may hallucinate parameter names, confuse argument order, or fail to chain multiple tools correctly. It is also sensitive to prompt phrasing: a slight change in tool description can flip success to failure. Fine-tuning exists to push the model past these limits — to internalize tool use so deeply that it becomes as natural as grammar.

> **💡 Key Insight**
>
> Pre-training gives the model *knowledge of the world*. Fine-tuning gives it *knowledge of action*. In-context learning bridges the gap for simple tasks but cannot reliably handle complexity. The three together form a hierarchy: pre-training for competence, fine-tuning for specialization, prompting for adaptability.

## 2. Fine-Tuning for Tool Use

### The problem: from pattern matching to reliable action

In-context learning can get a model to call a tool once. But what about ten times in a row, with error recovery, parameter validation, and multi-tool chaining? For that, the model needs to have seen thousands of successful tool-use patterns during training. It needs to internalize the syntax of JSON schemas, the semantics of error messages, and the pragmatics of when to stop versus when to try again. Fine-tuning is how we encode these patterns into the model's weights.

The core idea is straightforward: collect trajectories of agents successfully using tools, format them as training examples, and update the model's parameters to maximize the likelihood of generating correct tool calls. The challenge is in the details — what to collect, how to format it, and what to learn from.

### ToolLLM and Gorilla: early pioneers

Two landmark projects established the blueprint for tool-use fine-tuning.

**Gorilla** (Patil et al., 2023) fine-tuned a 7B-parameter LLaMA model on 1,645 machine-learning APIs from HuggingFace, TorchHub, and TensorHub. The training data consisted of natural-language queries paired with correct API calls. Gorilla introduced **Retriever-Aware Training (RAT)**: the model learned to condition on retrieved API documentation, enabling it to adapt to new API versions and documentation changes at test time. On APIBench, Gorilla surpassed GPT-4 in AST accuracy while dramatically reducing hallucinated API names.

**ToolLLM** (Qin et al., 2023) scaled this approach to 16,464 real-world RESTful APIs from RapidAPI Hub, spanning 49 categories from finance to social media. ToolLLM introduced **ToolBench**, an instruction-tuning dataset containing both single-tool and multi-tool scenarios, and **DFSDT** (Depth-First Search-based Decision Tree), a reasoning algorithm that improved planning over standard ReAct. ToolLLaMA, the fine-tuned LLaMA-2 7B, showed strong zero-shot generalization: despite never being trained on ML APIs, it matched Gorilla's performance on APIBench.

| Aspect | Gorilla | ToolLLM |
|--------|---------|---------|
| Base model | LLaMA 7B | LLaMA-2 7B |
| APIs covered | ~1,645 ML APIs | 16,464 real-world REST APIs |
| Key technique | Retriever-Aware Training | DFSDT reasoning + neural retriever |
| Multi-tool chaining | Limited | Supported |
| OOD generalization | Good on ML APIs | Strong across domains |


*Table 7.1 — Gorilla vs. ToolLLM. Gorilla specialized in ML APIs with retriever-aware training. ToolLLM scaled to general REST APIs with explicit reasoning.*


### Training data construction: trajectories as training examples

The fundamental data unit for agent fine-tuning is the **trajectory**: a sequence of (query, thought, action, observation) tuples that record how an agent solved a task. A single trajectory might look like this:

```
User: What is the weather in Tokyo?
Thought: The user wants current weather. I should use the weather tool.
Action: get_weather(location="Tokyo")
Observation: {"temperature": 22, "condition": "cloudy"}
Thought: I have the data. I can answer directly.
Final Answer: It is 22°C and cloudy in Tokyo.
```

To turn this into training data, we format it as a multi-turn conversation where the model learns to predict the assistant's messages (thoughts and actions) given the user's query and tool observations. The critical detail is **masking**: the model should learn from its own actions, not from the observations (which it did not generate). If we train the model to predict observations, we teach it to hallucinate tool outputs. If we train it to predict actions, we teach it to choose tools wisely.

Here is how trajectory formatting and masking work in practice. We implement a simple converter that takes a ReAct trace and produces token-level training labels.

```python
def format_trajectory_for_training(trajectory):
    """Convert a ReAct trace into (text, labels) pairs for supervised fine-tuning.

    trajectory: list of dicts with keys 'role', 'content', 'type'
        role is one of: 'user', 'assistant', 'tool', 'system'
        type is one of: 'thought', 'action', 'observation', 'query'
    Returns a list of (text_segment, label) where label is 1 if the model
    should learn to predict this segment, and 0 if it should be masked.
    """
    examples = []
    for turn in trajectory:
        role = turn["role"]
        content = turn["content"]
        seg_type = turn.get("type", "")

        # Mark assistant-generated thoughts, actions, and final answers for learning.
        # Mask user queries, tool observations, and system prompts.
        if role == "assistant" and seg_type in ("thought", "action", "final_answer"):
            label = 1   # Learn from this
        else:
            label = 0   # Do not learn from this

        examples.append({
            "text": f"{role}: {content}",
            "label": label,
            "type": seg_type,
        })
    return examples


# Example ReAct trace from a weather-query task.
trace = [
    {"role": "system",  "content": "You have access to get_weather().", "type": "system"},
    {"role": "user",    "content": "What is the weather in Tokyo?",   "type": "query"},
    {"role": "assistant", "content": "I need current weather data.", "type": "thought"},
    {"role": "assistant", "content": 'get_weather(location="Tokyo")', "type": "action"},
    {"role": "tool",    "content": '{"temp": 22, "cond": "cloudy"}', "type": "observation"},
    {"role": "assistant", "content": "The user can be told directly.", "type": "thought"},
    {"role": "assistant", "content": "It is 22°C and cloudy in Tokyo.", "type": "final_answer"},
]

training_examples = format_trajectory_for_training(trace)
for ex in training_examples:
    marker = "LEARN" if ex["label"] else "MASK"
    print(f"[{marker}] {ex['text']}")

# Verify masking logic: thoughts, actions, and final answers are learnable;
# system prompts, user queries, and tool observations are masked.
labels_by_type = {ex["type"]: ex["label"] for ex in training_examples}
assert labels_by_type["system"] == 0, "System prompts must be masked"
assert labels_by_type["query"] == 0,   "User queries must be masked"
assert labels_by_type["thought"] == 1,  "Assistant thoughts must be learned"
assert labels_by_type["action"] == 1,   "Assistant actions must be learned"
assert labels_by_type["observation"] == 0, "Tool observations must be masked"
assert labels_by_type["final_answer"] == 1, "Final answers must be learned"

# Verify only assistant-generated content is learned.
learn_count = sum(1 for ex in training_examples if ex["label"] == 1)
mask_count  = sum(1 for ex in training_examples if ex["label"] == 0)
assert learn_count == 4, f"Expected 4 learnable turns, got {learn_count}"
assert mask_count == 3,  f"Expected 3 masked turns, got {mask_count}"
```

Running this produces:

```
[MASK] system: You have access to get_weather().
[MASK] user: What is the weather in Tokyo?
[LEARN] assistant: I need current weather data.
[LEARN] assistant: get_weather(location="Tokyo")
[MASK] tool: {"temp": 22, "cond": "cloudy"}
[LEARN] assistant: The user can be told directly.
[LEARN] assistant: It is 22°C and cloudy in Tokyo.
```

Notice that observations are masked — the model must learn to generate tool calls that produce observations, but it should not learn to predict the observations themselves. This prevents a subtle but dangerous failure mode: the model learning to "guess" tool outputs instead of actually invoking tools.

### Avoiding catastrophic forgetting

Fine-tuning is a double-edged sword. When we specialize a model for tool use, we risk degrading its general knowledge — a phenomenon called **catastrophic forgetting**. A model fine-tuned exclusively on weather API calls might forget how to write poetry, translate languages, or answer history questions. The weights that encoded general knowledge have been overwritten by weights that encode API syntax.

Several techniques mitigate this:

**LoRA (Low-Rank Adaptation)** instead of full fine-tuning. Rather than updating all billions of parameters, LoRA introduces small trainable rank-decomposition matrices alongside the frozen original weights. Only these small matrices are updated during training. The model retains its general knowledge while learning tool use through the adapter layers. In practice, LoRA reduces trainable parameters by 99% while preserving 90–95% of the performance gain from full fine-tuning.

**Replay buffers** mix old and new training data. During tool-use fine-tuning, a fraction of each batch comes from the original pre-training distribution — general text, code, and conversations. This reminds the model of its prior knowledge while it learns the new skill.

**Progressive training** adds skills incrementally. Instead of jumping from general model to full agent, we first fine-tune on simple tool calls, then on multi-tool chains, then on error recovery. Each stage builds on the previous without erasing it.

> **⚠️ Warning**
>
> Catastrophic forgetting is insidious. You will not notice it in your tool-use benchmark scores, because those tests only measure the new skill. The degradation in general knowledge reveals itself weeks later, when a user asks a non-tool question and gets gibberish. Always evaluate on a general-knowledge benchmark before and after fine-tuning.

## 3. Instruction Tuning for Agents

### System prompts as behavioral priors

A system prompt is the first text the model sees, and it functions as a behavioral prior — a set of constraints and instructions that shape every subsequent turn. For agents, the system prompt is not decorative; it is the foundation of reliable behavior. A well-crafted system prompt tells the model: "You are an agent. You have these tools. You must think before acting. You must stop when the task is complete."

The difference between a system prompt that works and one that fails is often a single sentence. "Use tools when helpful" produces erratic behavior — the model sometimes uses tools, sometimes ignores them. "You MUST use a tool for any question involving external data" produces consistent tool use but may lead to unnecessary tool calls for questions the model already knows the answer to. The art is in calibrating the constraint strength.

Fine-tuning with system prompts hardens these instructions into the model's weights. During training, every trajectory begins with the same system prompt. The model learns not just to follow the prompt but to internalize its intent. A fine-tuned agent does not need to be reminded to think step by step; it does so automatically because that pattern is now part of its parameter space.

### Multi-turn dialogue formatting

Agent training requires a specific conversation format that can accommodate tool calls and observations. Unlike a simple chat where user and assistant alternate, agent conversations include a third participant: the tool.

The standard format looks like this:

```
System: You are a helpful assistant with access to tools.
User: What is 15% of 847?
Assistant (thought): I need to calculate this precisely. I will use the calculator tool.
Assistant (action): calculate(expression="0.15 * 847")
Tool: 127.05
Assistant: 15% of 847 is 127.05.
```

During fine-tuning, this is flattened into a token sequence where the model learns to predict the assistant's turns (thoughts, actions, final answers) while the user query and tool observations are provided as context. The key challenge is that tool observations can be long — a web search might return thousands of tokens, a code execution might produce pages of output. Truncation strategies, summary injection, and selective retention all become necessary to fit within the context window.

### Data formats: ShareGPT, AgentInstruct, ToolBench

The agent community has converged on several standard data formats for fine-tuning:

**ShareGPT** is a simple conversation format: a list of messages, each with a `role` (system, user, assistant, tool) and `content`. It is human-readable, widely supported by training frameworks, and easy to inspect. The downside is that it lacks explicit typing for thoughts versus actions — the model must infer which assistant messages are tool calls based on content patterns.

**AgentInstruct** extends ShareGPT with explicit action annotations. Each assistant message includes metadata: `action_type` (tool_call, final_answer, thought), `tool_name`, and `arguments`. This explicit structure makes masking easier and reduces ambiguity during training.

**ToolBench** (from the ToolLLM project) uses a custom format optimized for multi-tool scenarios. Each training example includes a task description, a list of available tools with JSON schemas, a reasoning trace, and the final answer. The format is richer but more complex — it encodes tool schemas inline rather than referencing them externally.

### Evaluating fine-tuned tool callers

Before deploying a fine-tuned agent, we need to know whether it actually improved. The evaluation of a tool-calling model has three dimensions: **syntax**, **semantics**, and **orchestration**.

**Syntax**: Does the model output valid JSON or function calls? Does it respect the parameter schema? A syntax error means the tool invocation fails before it even runs.

**Semantics**: Given the user's intent, did the model choose the right tool with the right arguments? A model that calls `get_weather(location="Paris, France")` when asked about London has correct syntax but wrong semantics.

**Orchestration**: For multi-step tasks, does the model chain tools correctly? Does it use the output of one tool as input to the next? Does it know when to stop?

We implement a lightweight evaluator that checks syntax and semantics without requiring live tool execution.

```python
import json
from typing import List, Dict, Any


class ToolCallEvaluator:
    """Evaluate a fine-tuned model's tool-calling capability.

    Checks three levels:
      1. Syntax: is the tool call valid JSON matching the schema?
      2. Semantics: did it pick the right tool with correct arguments?
      3. Orchestration: for multi-step tasks, is the sequence correct?
    """

    def __init__(self, tool_schemas: Dict[str, Dict[str, Any]]):
        self.tool_schemas = tool_schemas

    def evaluate_syntax(self, tool_name: str, arguments: str) -> bool:
        """Check if arguments are valid JSON matching the tool schema."""
        schema = self.tool_schemas.get(tool_name, {})
        required = schema.get("required", [])
        try:
            parsed = json.loads(arguments)
        except json.JSONDecodeError:
            return False
        return all(k in parsed for k in required)

    def evaluate_semantics(
        self,
        user_query: str,
        predicted_tool: str,
        predicted_args: Dict[str, Any],
        gold_tool: str,
        gold_args: Dict[str, Any],
    ) -> Dict[str, Any]:
        """Compare predicted tool call against ground truth."""
        tool_correct = predicted_tool == gold_tool
        args_correct = (
            tool_correct and
            all(predicted_args.get(k) == v for k, v in gold_args.items())
        )
        return {
            "tool_correct": tool_correct,
            "args_correct": args_correct,
            "fully_correct": tool_correct and args_correct,
        }

    def evaluate_batch(self, predictions: List[Dict[str, Any]]) -> Dict[str, float]:
        """Run evaluation over a batch of (query, prediction, gold) examples."""
        n = len(predictions)
        if n == 0:
            return {"syntax_accuracy": 0.0, "tool_accuracy": 0.0, "full_accuracy": 0.0}

        syntax_pass = 0
        tool_pass = 0
        full_pass = 0

        for ex in predictions:
            pred_tool = ex["predicted_tool"]
            pred_args = ex["predicted_args"]
            gold_tool = ex["gold_tool"]
            gold_args = ex["gold_args"]

            # Syntax check
            syntax_ok = self.evaluate_syntax(
                pred_tool, json.dumps(pred_args)
            )
            if syntax_ok:
                syntax_pass += 1

            # Semantic check
            sem = self.evaluate_semantics(
                ex["query"], pred_tool, pred_args, gold_tool, gold_args
            )
            if sem["tool_correct"]:
                tool_pass += 1
            if sem["fully_correct"]:
                full_pass += 1

        n = len(predictions)
        return {
            "syntax_accuracy": syntax_pass / n,
            "tool_accuracy": tool_pass / n,
            "full_accuracy": full_pass / n,
        }


# Example usage with two test cases.
schemas = {
    "get_weather": {
        "required": ["location"],
        "properties": {"location": {"type": "string"}},
    },
    "calculate": {
        "required": ["expression"],
        "properties": {"expression": {"type": "string"}},
    },
}

evaluator = ToolCallEvaluator(schemas)
predictions = [
    {
        "query": "Weather in Tokyo?",
        "predicted_tool": "get_weather",
        "predicted_args": {"location": "Tokyo"},
        "gold_tool": "get_weather",
        "gold_args": {"location": "Tokyo"},
    },
    {
        "query": "What is 15% of 847?",
        "predicted_tool": "get_weather",  # Wrong tool!
        "predicted_args": {"location": "847"},
        "gold_tool": "calculate",
        "gold_args": {"expression": "0.15 * 847"},
    },
]

results = evaluator.evaluate_batch(predictions)
assert results["syntax_accuracy"] == 1.0, f"Expected 1.0, got {results['syntax_accuracy']}"
assert results["tool_accuracy"] == 0.5, f"Expected 0.5, got {results['tool_accuracy']}"
assert results["full_accuracy"] == 0.5, f"Expected 0.5, got {results['full_accuracy']}"

# Edge case: empty predictions must not divide by zero.
assert evaluator.evaluate_batch([]) == {
    "syntax_accuracy": 0.0, "tool_accuracy": 0.0, "full_accuracy": 0.0
}
```

This evaluator gives us three numbers that tell a nuanced story. A syntax accuracy of 1.0 but tool accuracy of 0.5 means the model knows *how* to call tools but not *which* tool to call. This is common after early-stage fine-tuning: the model has learned the format but not the semantics. A full accuracy of 0.5 tells us that half the time, the model gets both the tool and the arguments exactly right. These metrics guide what to fix next — more training data, better tool descriptions, or harder negative examples.

## 4. Emerging Training Paradigms

### The shift from supervised fine-tuning to reinforcement learning

Supervised fine-tuning teaches a model to imitate expert behavior. But imitation has a ceiling: the model cannot exceed the quality of its training data. If the expert trajectories contain suboptimal tool choices, the model learns those too. Worse, SFT cannot teach the model to recover from errors, because the training data only contains successful paths.

Reinforcement learning (RL) breaks this ceiling. Instead of imitating fixed trajectories, the model explores an action space, receives rewards for success, and learns to maximize long-term return. In the context of agents, RL trains the model not just to replicate tool calls but to *discover* better strategies — to chain tools more efficiently, to retry failed calls with corrected parameters, and to know when a simpler approach suffices.

The results have been transformative. OpenAI's o3 and o4-mini, trained with RL, can chain 600+ tool calls in complex reasoning tasks. DeepSeek-R1 emerged pure reflection and verification behaviors from RL alone — no human demonstrated those behaviors; the model invented them because they led to higher reward. This is the frontier: from imitation to invention.

### RL for tool use: the reward loop

Training an agent with RL requires three components: a **policy** (the model that decides actions), an **environment** (the set of tools and their outputs), and a **reward function** (the signal that tells the model whether it succeeded).

The reward function is the design challenge. For simple tasks like calculator use, the reward is binary: did the final answer match the ground truth? For complex tasks like web navigation, the reward might be sparse — only received at the end of a 20-step episode — making learning slow. The solution is **dense rewards**: intermediate signals that guide the model toward the goal. A web agent might receive small rewards for reaching the right page, filling a form field correctly, or extracting relevant text.

Modern RL for agents has converged on several key techniques:

**GRPO (Group Relative Policy Optimization)**, popularized by DeepSeek-R1, simplifies the RL pipeline by eliminating the need for a separate value function. Instead of estimating expected returns, GRPO compares multiple sampled responses from the same prompt and rewards the better ones relative to the group average. This is simpler to implement and more stable than classic PPO, especially for reasoning tasks.

**ToolOmni** (April 2026) introduced **Decoupled Multi-Objective GRPO**, which separates the reward for tool retrieval from the reward for tool execution. The insight is that coupling these rewards creates gradient conflict: a model might learn to retrieve the wrong tool but execute it perfectly, receiving a muddled training signal. By decoupling the objectives, ToolOmni achieved up to 10.8% improvement in end-to-end execution success.

**ATLAS** (March 2026, Microsoft Research) uses **rubric-based reinforcement fine-tuning (RFT)** for agents operating in MCP environments. Instead of a single success signal, ATLAS decomposes tasks into structured criteria: task fulfillment, tool appropriateness, grounding accuracy, and parameter correctness. Each criterion yields a separate reward, enabling fine-grained credit assignment. Surprisingly, small language model judges outperform frontier models like GPT-4o when evaluating with structured rubrics — a key insight for scalable training.

**Tool-R0** (February 2026) takes the radical step of **zero-data self-play**. It co-evolves a Generator and a Solver from the same base LLM. The Generator creates challenging tool-calling tasks; the Solver learns to execute them. A difficulty-aware reward targets the Solver's "competence frontier," allowing the system to self-evolve without any human-curated data. Tool-R0 achieved 92.5% relative improvement over its base model purely through self-play — a compelling demonstration that agents can generate their own training data.

### Process Reward Models: rewarding intermediate steps

The classic RL reward is an **outcome reward**: you get +1 if the final answer is correct, 0 otherwise. But for multi-step agent tasks, this is frustratingly sparse. If an agent makes a mistake on step 3 of 20, it receives the same signal (failure) as if it made a mistake on step 19. It cannot tell which step went wrong.

**Process Reward Models (PRMs)** solve this by scoring each intermediate step. A PRM is a secondary model — often a smaller LLM or a classifier — trained to evaluate whether each reasoning step or tool call is correct. The agent policy receives a reward not just at the end but at every step where the PRM approves.

The practical challenge is acquiring step-level labels. Human annotators can label whether step 3 was correct, but this is expensive. **PRIME** (2025) showed that process rewards can be derived implicitly from outcome labels alone — no human step labels required. The model learns to predict which steps led to successful outcomes and which led to failures, bootstrapping process supervision from sparse outcome data.

**AgentPRM** (Cornell + Fudan, 2025) extends PRMs to multi-turn agents by treating each turn as a step in an actor-critic framework. The critic (PRM) evaluates each tool call and thought, while the actor (policy) learns to maximize the critic's scores. This is particularly effective for long-horizon tasks where outcome rewards are too delayed to guide learning.

### Synthetic trajectory generation

The biggest bottleneck in agent training is not compute or algorithms — it is data. Human demonstrations of expert tool use are expensive to collect and limited in diversity. The solution is to generate training data synthetically.

**STeP (Self-training via Planning, 2024)** generates trajectories by having a strong model solve tasks and then using those solutions as training data for a weaker model. The strong model acts as a teacher, generating correct tool calls and reasoning traces that the student model learns to imitate. This is distillation at the trajectory level rather than the output level.

**Simia** takes a different approach: it simulates environments where agents can practice. Instead of using real APIs (which cost money and rate-limit), Simia creates simulated tool environments with controllable complexity. An agent can practice booking flights on a fake airline API, learning from mistakes without real-world consequences. The simulated trajectories are then used to fine-tune the real model.

Both approaches share a crucial insight: **the quality of synthetic data depends on the verifier**. If the verifier that judges success is flawed, the synthetic trajectories contain hidden errors that the model will learn to replicate. A trajectory generator is only as good as its evaluation function.

### The three paradigms of agentic tool use

Research in 2025–2026 has crystallized three distinct paradigms for giving LLMs tool-use capabilities. Understanding where each sits on the cost-performance curve is essential for building production agents.

**Prompting as plug-and-play** requires no parameter updates. You describe the tool in a system prompt, provide a few examples, and hope the model generalizes. Cost: negligible. Performance: sufficient for 1–2 step tasks with common tools. Reliability: brittle — small prompt changes can break behavior.

**Supervised tool learning** fine-tunes the model on curated or synthetic trajectories. Cost: moderate — requires GPU hours and data curation. Performance: reliable for known tools and task types. Reliability: good within distribution, poor out-of-distribution. Does not generalize to unseen tools without re-training.

**Reward-driven tool policy learning** uses RL to optimize tool use strategies dynamically. Cost: high — requires environment simulation, reward engineering, and extensive sampling. Performance: superior for complex multi-step tasks. Reliability: highest for in-distribution tasks; can discover novel strategies. The frontier of agent capability.

| Paradigm | Training cost | Performance ceiling | Generalization | Best for |
|----------|---------------|--------------------|----------------|----------|
| Prompting | Negligible | Low (1–2 steps) | Good to new tools | Prototyping, simple tasks |
| Supervised learning | Moderate | Medium (5–10 steps) | Poor OOD | Production agents with fixed tools |
| Reinforcement learning | High | High (100+ steps) | Moderate | Complex, dynamic tasks requiring strategy |


*Table 7.2 — The three paradigms of agentic tool use. Cost and capability scale together. Choose based on task complexity and budget constraints.*

## Summary

- Pre-trained LLMs provide the cognitive substrate — world knowledge, code understanding, reasoning patterns — on which agency is built. In-context learning enables zero-shot tool use but hits reliability limits quickly.
- Fine-tuning specializes models for tool use by training on trajectories of (thought, action, observation). Critical technique: mask observations during loss computation so the model learns to generate actions, not predict tool outputs.
- Catastrophic forgetting is the hidden risk of fine-tuning. Mitigate with LoRA (train small adapter matrices instead of all parameters) and replay buffers that mix old and new data.
- Instruction tuning formats matter. ShareGPT is simple and universal; AgentInstruct adds explicit action metadata; ToolBench encodes full tool schemas for multi-tool training.
- Reinforcement learning is the frontier. GRPO simplifies RL for reasoning; decoupled multi-objective rewards prevent gradient conflict; rubric-based RFT enables fine-grained credit assignment; zero-data self-play removes the human data bottleneck.
- Process Reward Models provide step-level feedback where outcome rewards are too sparse. PRIME derives process rewards from outcome labels alone, removing the need for expensive human step annotations.
- Synthetic trajectory generation (STeP, Simia) scales training data without human demonstrations, but quality depends entirely on the verifier.
