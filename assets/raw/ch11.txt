# Chapter 11: Web Agents and Browser Automation

> The web is the largest API ever built — and it was never designed for machines. It has no schema, no rate limits you can trust, and no guaranteed stability: a button moves five pixels left, a login form gains a CAPTCHA, and an entire checkout flow vanishes behind an A/B test. Yet this chaotic environment is where humans spend most of their digital lives, which means it is where agents must operate if they are to be useful. By the end of this chapter you will build a complete browser agent in pure Python that navigates real pages, perceives them through both screenshots and DOM structure, executes actions with retry logic, and evaluates itself against a curated task suite — all using Playwright and a ReAct loop, with no high-level framework between you and the browser.

---

## 1. The Web as an Environment

### 1.1 DOM Parsing and Element Extraction

A web page is a tree. The Document Object Model (DOM) represents every paragraph, button, and input field as a node in a hierarchy that the browser renders into pixels. For an agent, the DOM is the closest thing to a structured state representation the web offers.

Extracting useful structure from raw HTML is harder than it looks. Modern pages are bloated: a single commercial landing page can contain 50,000 DOM nodes, most of them invisible wrappers injected by CSS frameworks and analytics scripts. An agent that tries to ingest the entire HTML tree into its context window will quickly drown in noise. The art of DOM-aware agency is **selective extraction**.

The most useful DOM signals are:
- **Interactive elements**: buttons, links, inputs, selects, and anything with an `onclick` handler.
- **Semantic landmarks**: ARIA roles (`role="navigation"`, `role="main"`, `role="search"`) that tell the agent where it is.
- **Text density**: paragraphs and headings that carry the informational payload.
- **Form structure**: labels tied to inputs via `for` attributes, which ground typed actions.

A minimal but effective extraction strategy is to filter the DOM to only visible, interactive, and semantic nodes, then serialize them as a simplified HTML string or an accessibility tree. This reduces 50,000 nodes to a few hundred tokens the LLM can actually reason about.

### 1.2 The Action Space

An agent can only do what its action space allows. Design the space too broadly — arbitrary JavaScript execution, unconstrained URL navigation — and the agent wanders into error states it cannot recover from. Design it too narrowly — only clicks, no scrolling — and the agent stalls on any page longer than one viewport.

The convergent action vocabulary across the 2025–2026 literature is surprisingly small and human-like. Every major system, from Browser-Use to WEBAGENT-R1 to OpAgent, uses a core set of atomic operations:

| Action | Parameters | Purpose |
|--------|------------|---------|
| `click` | element id or (x, y) | Activate a button or link |
| `type` | element id or (x, y), text, press_enter | Fill an input field |
| `scroll` | direction, amount | Reveal content below the fold |
| `navigate` | URL | Load a new page |
| `wait` | duration or condition | Allow the page to settle after an action |
| `extract` | query or selector | Pull structured data from the page |
| `done` | answer | Terminate the task and return a result |

The critical design choice is **granularity**. Research consistently shows that atomic actions — one operation per step — outperform compound actions in long-horizon tasks. A "buy now" macro that clicks, fills, and submits in a single turn looks efficient, but if the checkout page loads a new address form, the macro fails in a way the agent cannot debug. Atomic actions let the agent observe intermediate states and recover.

### 1.3 Visual vs. DOM-Based Agents

There are two ways an agent can "see" a web page. It can read the DOM tree, or it can look at the rendered screenshot. Both have strengths, both have failure modes, and the frontier in 2026 is neither pure strategy but a hybrid of the two.

**DOM-aware agents** parse the HTML or accessibility tree and ground their actions on element IDs, CSS selectors, or ARIA labels. This is the approach taken by Gemini-based systems and by open-source frameworks like Browser-Use. DOM agents are fast — perception takes 1–3 seconds — and cheap, because no vision model is invoked. They are precise: clicking by element ID is deterministic. But they are brittle. When a developer changes a `div` class from `btn-primary` to `button--v2`, the agent's selectors break. Canvas elements, WebGL, and poorly structured pages are invisible to them.

**Visual agents** feed screenshots into a vision-language model (VLM) such as Claude 3.5 Sonnet or GPT-4o, which predicts click coordinates and typed text directly from pixels. This is the approach behind Anthropic's Computer Use API and the WEBSIGHT agent. Visual agents are robust to DOM changes — a button that moves five pixels is still a button — and they handle canvas, PDFs, and native desktop applications. But they are 2–4× slower per step, roughly 2× more expensive, and probabilistic: the model might misread a coordinate by twenty pixels and click the wrong link.

The practical choice is not either-or. In 2026, the consensus architecture is **hybrid**: use the DOM for 80% of interactions — clicks, form fills, navigation — because it is fast and deterministic, and fall back to vision for edge cases: CAPTCHAs, canvas elements, visual verification ("does this chart look correct?"), and sites with unstable selectors.

### 1.4 Headless Browsers: Playwright and Puppeteer

Every browser agent needs a programmatic browser. In 2026, the default choice for new projects is **Playwright**. It is faster than Selenium, has built-in auto-waiting (actions retry until elements are ready), and supports all major browsers with a single API. Puppeteer remains viable for Chrome-centric stacks. Selenium, while historically important, is now viewed as legacy for new agent projects: its API is verbose, its waiting semantics are manual, and its performance lags behind Playwright by a meaningful margin on long-horizon tasks.

Playwright's key features for agents are:
- **Auto-waiting**: `page.click()` automatically waits for the element to appear and be enabled, eliminating an entire class of race-condition bugs.
- **Context isolation**: Each browser context has its own cookies, local storage, and session state, so agents can run multiple sessions in parallel without cross-contamination.
- **Accessibility tree snapshot**: `page.accessibility.snapshot()` returns a structured tree of roles, names, and values that is cleaner than raw HTML.
- **Screenshot with bounding boxes**: `page.screenshot()` can overlay element bounding boxes, giving visual agents structured grounding without losing pixel context.

## 2. Web Agent Architectures

### 2.1 DOM-Aware Agents

A DOM-aware agent treats the accessibility tree as its primary observation. The pipeline is: snapshot the tree, simplify it into a text representation the LLM can ingest, prompt the LLM to choose an action expressed as an element ID or CSS selector, and execute that action via Playwright.

The Gemini family of models popularized this approach. Because Gemini has a very long context window (up to 2 million tokens in some configurations), it can ingest large accessibility trees without aggressive filtering. The model is fine-tuned to output structured action tokens: `click [id_12]`, `type [id_45] "search query"`, `scroll down`. The fine-tuning is critical. A generalist LLM asked to navigate a web page from raw HTML will hallucinate selectors or invent element IDs that do not exist.

The weakness of pure DOM agents is **dynamic content**. Single-page applications (SPAs) rewrite the DOM asynchronously. An element ID valid at step 3 may be stale at step 4 because a React re-render regenerated the component. DOM agents must re-snapshot after every action and validate that their target still exists before clicking.

### 2.2 Visual Agents

Visual agents reverse the pipeline: screenshot first, reasoning second. The agent captures a full-page or viewport screenshot, encodes it through a VLM, and receives back a predicted action in pixel coordinates.

Claude's Computer Use API is the canonical example. The agent sends a screenshot to the model along with a system prompt that describes the available actions (`computer.terminate`, `computer.click`, `computer.scroll`, etc.). The model returns a structured XML action: `<action><click x="123" y="456"></click></action>`. The agent executes it, takes a new screenshot, and loops.

The accuracy of visual agents depends on the VLM's **visual grounding** capability — its ability to map a text instruction like "click the Submit button" to the correct (x, y) coordinates. In 2025, grounding error was the single largest failure mode for vision-only systems on WebArena. A button mis-click does not just fail the task; it can navigate to an entirely different page, leaving the agent lost.

### 2.3 Hybrid Agents

The hybrid architecture combines both observation streams. The agent maintains a Playwright page object for DOM manipulation and a screenshot pipeline for visual verification. At each step, the agent decides which modality to trust: if the DOM snapshot contains the target element with high confidence, it clicks by ID; if the DOM is ambiguous or the element is a canvas, it falls back to the screenshot and the VLM.

A clean implementation pattern is the **perception router**:

1. Attempt a DOM-based action using the accessibility tree.
2. If the element is not found or the action fails (e.g., `TimeoutError`), capture a screenshot.
3. Prompt the VLM with the screenshot and the original goal.
4. Execute the VLM's coordinate-based action.
5. On the next step, re-snapshot the DOM to re-synchronize.

This pattern is used by the `chromium-screenshots` MCP server and by the WebNavigator architecture, which achieved strong results on WebArena Verified by reframing navigation as retrieval over an "Interaction Graph" built from both DOM structure and rendered screenshots.

### 2.4 Session Management

Web agency is stateful. A task that requires logging into an account, adding an item to a cart, and checking out spans multiple pages, cookies, and form submissions. The agent must manage **session state** as carefully as a human tester would.

Playwright contexts make this tractable. A `BrowserContext` encapsulates cookies, `localStorage`, and HTTP authentication. The agent can create a context at the start of a task, persist it to disk between steps if the process restarts, and destroy it afterward to ensure no cross-task leakage.

```python
# Isolated session per task
context = await browser.new_context()
page = await context.new_page()
await page.goto("https://app.example.com/login")
# ... agent executes task ...
await context.close()   # wipes all cookies and storage
```

Authentication is the sharpest edge. Realistic web tasks require logging into services, and agents must handle credentials without exposing them. The safe pattern is **proxy injection**: the agent never holds passwords in its prompt or memory. A credential proxy (a small HTTP service or environment variable injection layer) inserts cookies or basic-auth headers after the agent has requested a login action. This prevents prompt-injection attacks from exfiltrating secrets and aligns with the least-privilege principle discussed in Chapter 48.

Multi-tab workflows are another under-discussed challenge. Agents trained on single-page tasks fail when a link opens a new tab or a popup demands attention. The action space must include `switch_tab` and `close_tab` operations, and the agent's state must track which tab is active.

## 3. Current State of Web Agents

### 3.1 WebArena Verified

WebArena (Shi et al., 2024) established the first rigorous benchmark for autonomous web navigation, but its original scoring pipeline had a critical flaw: brittle substring matching and subjective LLM-as-a-judge evaluators produced false negatives — agents that completed tasks correctly were marked as failures. **WebArena Verified**, introduced at NeurIPS 2025, fixed this by manually auditing all 812 tasks and replacing the scoring with deterministic, type-aware exact matching and backend state verification.

The result was a 11.3 percentage-point reduction in false negatives. Baselines rose accordingly: OpenAI Operator improved from 41.0% on the original benchmark to 49.0% ± 4.8% on WebArena Verified. The authors also introduced **WebArena Verified Hard**, a 137–258 task subset that preserves discriminative power while cutting evaluation cost by 68–83%.

As of early 2026, the leaderboard tells a clear story. Ensemble and modular architectures dominate raw model performance:

| Rank | Agent | Success Rate | Notes |
|------|-------|--------------|-------|
| 1 | OpAgent | 71.6% | Modular: Planner + Grounder + Reflector + Summarizer |
| 2 | Surfer 2 | 69.6% | Agentic orchestration, planning separate from execution |
| 3 | GPT-5.4 | 67.3% | Frontier model with WebArena-Verified protocols |
| 5 | WebNavigator + Claude Sonnet 4 | 57.1% | Hybrid: Interaction Graph + retrieval |
| 10 | WebNavigator + Qwen3-VL-32B | 47.8% | Strong open-weight performance with good scaffold |
| 15 | Claude 3.5 Sonnet | 36.0% | Base model without specialized scaffold |
| 18 | GPT-4o | 31.5% | Base model without specialized scaffold |

The gap between a frontier model alone (31–36%) and a well-engineered agent scaffold (57–72%) is 25–40 percentage points. This is one of the largest scaffold effects in any agent benchmark, and it proves that web navigation is primarily an architecture problem, not a raw-capability problem.

### 3.2 GAIA

GAIA (Mialon et al., 2024) tests multi-step reasoning with real-world tools, including web search, calculators, and file manipulation. Its three levels progress from simple lookups (Level 1) to long-horizon coordination (Level 3). In 2023, GPT-4 with plugins scored roughly 15% overall. By April 2026, the top systems have reached 92.36%, essentially matching the human baseline of 92%.

The GAIA leaderboard reveals two trends. First, **multi-model ensembles dominate**: the top five entries all orchestrate multiple frontier models (Claude 4.6, GPT-5, DeepSeek 3.2, Gemini 3 Pro, Qwen 3) rather than relying on a single LLM. Second, **scaffolding matters enormously**: the same Claude Opus 4 model scores 64.9% in one framework and 57.6% in another, a 7-point swing driven entirely by prompt design, tool access, and retry logic.

For web agents specifically, GAIA's most relevant contribution is its emphasis on **attribution and verification**. A successful GAIA agent does not merely find an answer; it must cite the sources it used and, in many cases, verify them against live web content. This makes GAIA a better predictor of production utility than pure navigation benchmarks.

### 3.3 Mind2Web 2

The original Mind2Web benchmark tested generalist web agents across 100+ sites with diverse tasks. In 2025, the authors released **Mind2Web 2**, focused on long-horizon **agentic search**: tasks like "find the best-reviewed wireless earbuds under $50 with active noise cancellation and return the top three options with prices and review summaries." These tasks are time-varying, require live browsing, and demand synthesis across multiple pages.

Mind2Web 2 introduced **Agent-as-a-Judge**, an evaluation protocol where automated judge agents score answers using tree-structured rubrics. Human validation found these judges to be ~99% accurate compared to human evaluators. The results are sobering: even the best system, OpenAI Deep Research, achieves only 28% full success rate (54% partial), compared to human 54% full success. Hallucination in attribution remains the dominant failure mode, affecting ~23% of tasks for the top system and over 50% for weaker ones.

The lesson is that **deep research is harder than deep browsing**. Navigating to a page and clicking a button is a solved problem for frontier agents; synthesizing a correct, attributed answer from ten tabs is not.

### 3.4 OpAgent and Frontier Results

OpAgent (Ant Group, February 2026) represents the current state of the art for modular web agent design. Rather than prompting a single model to both plan and execute, OpAgent splits the problem into four specialized modules:

1. **Planner**: Decomposes the user instruction into subgoals and selects strategies.
2. **Grounder**: A post-trained VLM (based on Qwen2.5-VL-72B) that translates instructions into precise UI coordinates and actions.
3. **Reflector**: Verifies whether the previous action succeeded, detects blockers like CAPTCHAs, and triggers self-correction.
4. **Summarizer**: Synthesizes the final answer from the trajectory history.

The full framework achieves 71.6% on WebArena. Its single-model variant, a Qwen3-VL-32B-Thinking model trained with online RL in the wild, reaches 38.1% — a 10.7-point improvement over the vanilla model. The RL pipeline uses a hybrid reward: a VLM-based `WebJudge` for outcome evaluation and a rule-based decision tree for step-level progress rewards. This is a practical application of process reward ideas from Chapter 15 to the web domain.

## 4. Project — Building a Browser Agent

We now build a complete browser agent in pure Python. The design follows the ReAct pattern from Chapter 2, extended with browser-specific perception and action primitives. The agent uses Playwright for browser control, captures both DOM snapshots and screenshots, and implements a hybrid perception router with retry logic.

### 4.1 Browser Setup and Teardown

We begin with a thin wrapper around Playwright that manages the browser context and exposes a clean observation interface.

```python
import asyncio
from playwright.async_api import async_playwright
from dataclasses import dataclass
from typing import Optional, Dict, List

@dataclass
class BrowserObservation:
    url: str
    title: str
    dom_text: str           # filtered accessibility tree as text
    screenshot_path: Optional[str] = None
    error: Optional[str] = None

class BrowserEnv:
    def __init__(self, headless: bool = True):
        self.headless = headless
        self.playwright = None
        self.browser = None
        self.context = None
        self.page = None

    async def start(self):
        self.playwright = await async_playwright().start()
        self.browser = await self.playwright.chromium.launch(headless=self.headless)
        self.context = await self.browser.new_context(viewport={"width": 1280, "height": 800})
        self.page = await self.context.new_page()

    async def observe(self, screenshot: bool = True) -> BrowserObservation:
        try:
            url = self.page.url
            title = await self.page.title()
            # Simplified DOM: visible text and interactive elements only
            dom_text = await self.page.evaluate("""
                () => {
                    const nodes = [];
                    const walker = document.createTreeWalker(
                        document.body, NodeFilter.SHOW_ELEMENT, null, false
                    );
                    let el;
                    while (el = walker.nextNode()) {
                        const rect = el.getBoundingClientRect();
                        if (rect.width > 0 && rect.height > 0) {
                            const tag = el.tagName.toLowerCase();
                            if (['a','button','input','select','textarea'].includes(tag)
                                || el.getAttribute('role')) {
                                const text = el.innerText?.slice(0, 120) || '';
                                nodes.push(`[${tag} id=${el.id} text="${text}"]`);
                            }
                        }
                    }
                    return nodes.slice(0, 100).join('\\n');
                }
            """)
            shot = None
            if screenshot:
                shot = "/tmp/agent_screenshot.png"
                await self.page.screenshot(path=shot, full_page=False)
            return BrowserObservation(url=url, title=title, dom_text=dom_text, screenshot_path=shot)
        except Exception as e:
            return BrowserObservation(url="", title="", dom_text="", error=str(e))

    async def act(self, action: str, **kwargs) -> str:
        """Execute a single atomic action. Returns observation or error."""
        try:
            if action == "navigate":
                await self.page.goto(kwargs["url"], wait_until="networkidle")
            elif action == "click":
                # Prefer selector; fallback to text-based lookup
                selector = kwargs.get("selector")
                if selector:
                    await self.page.click(selector)
                else:
                    await self.page.get_by_text(kwargs["text"]).click()
            elif action == "type":
                selector = kwargs.get("selector")
                text = kwargs["text"]
                if selector:
                    await self.page.fill(selector, text)
                else:
                    await self.page.get_by_text(kwargs["label"]).fill(text)
                if kwargs.get("press_enter"):
                    await self.page.press(selector or kwargs["label"], "Enter")
            elif action == "scroll":
                direction = kwargs.get("direction", "down")
                amount = kwargs.get("amount", 500)
                await self.page.evaluate(f"window.scrollBy(0, {'+' if direction=='down' else '-'}{amount})")
            elif action == "wait":
                await self.page.wait_for_timeout(kwargs.get("ms", 1000))
            elif action == "extract":
                return await self.page.evaluate(kwargs.get("js", "document.body.innerText"))
            return "ok"
        except Exception as e:
            return f"error: {type(e).__name__}: {e}"

    async def close(self):
        if self.context:
            await self.context.close()
        if self.browser:
            await self.browser.close()
        if self.playwright:
            await self.playwright.stop()
```

The `BrowserEnv` class isolates all Playwright interactions. The `observe` method filters the DOM to at most 100 visible interactive nodes, keeping the observation small enough for an LLM context window. The `act` method exposes the atomic action vocabulary: navigate, click, type, scroll, wait, extract.

### 4.2 The ReAct Loop with Browser Perception

The agent itself is a ReAct loop where observations are browser states and actions are Playwright operations. We simulate the LLM with a simple planner function; in production, this would be a call to an OpenAI, Anthropic, or local API.

```python
import json
from typing import Callable

class BrowserAgent:
    def __init__(
        self,
        llm_call: Callable[[str], str],
        env: BrowserEnv,
        max_steps: int = 15,
        use_screenshot_fallback: bool = True,
    ):
        self.llm = llm_call
        self.env = env
        self.max_steps = max_steps
        self.use_fallback = use_screenshot_fallback
        self.trace: List[Dict] = []

    def _build_prompt(self, task: str, observation: BrowserObservation) -> str:
        history = "\n".join(
            f"Step {i+1}: Thought: {step['thought']}\nAction: {step['action']}"
            for i, step in enumerate(self.trace)
        )
        return f"""You are a web-browsing agent. Your task: {task}
Current page: {observation.title} ({observation.url})
Visible elements:
{observation.dom_text}

Previous actions:
{history}

Respond with exactly one JSON object:
{{"thought": "...", "action": "navigate|click|type|scroll|wait|extract|done", "params": {{...}}}}
Rules:
- Use navigate only for new URLs.
- Use click with either selector or text.
- Use type with selector or label, plus text and optional press_enter.
- Use scroll with direction (up/down) and amount (pixels).
- Use done when the task is complete.
- If the target element is not in the DOM, use scroll to reveal it first.
"""

    async def run(self, task: str) -> str:
        await self.env.start()
        try:
            for step in range(self.max_steps):
                obs = await self.env.observe(screenshot=self.use_fallback)
                if obs.error:
                    return f"Observation failed: {obs.error}"

                prompt = self._build_prompt(task, obs)
                raw = self.llm(prompt)
                # Simple JSON parsing with retry for malformed outputs
                try:
                    parsed = json.loads(raw.strip().strip("`").removeprefix("json").strip())
                except Exception:
                    # If the LLM fails JSON, treat the whole output as a thought and wait
                    parsed = {"thought": raw, "action": "wait", "params": {"ms": 1000}}

                thought = parsed.get("thought", "")
                action = parsed.get("action", "wait")
                params = parsed.get("params", {})
                self.trace.append({"thought": thought, "action": action, "params": params})

                if action == "done":
                    return params.get("answer", "Task completed.")

                # Execute with retry logic
                result = await self._execute_with_retry(action, params, obs)
                self.trace[-1]["result"] = result

                if "error" in result.lower() and self.use_fallback and obs.screenshot_path:
                    # Fallback: if DOM action fails, instruct LLM to reason from screenshot
                    obs = await self.env.observe(screenshot=True)
                    fallback_prompt = prompt + f"\n\nPrevious action failed: {result}\nA screenshot is now available. Consider scrolling or using a different element."
                    raw2 = self.llm(fallback_prompt)
                    try:
                        parsed2 = json.loads(raw2.strip().strip("`").removeprefix("json").strip())
                    except Exception:
                        parsed2 = {"thought": raw2, "action": "wait", "params": {"ms": 1000}}
                    action2 = parsed2.get("action", "wait")
                    params2 = parsed2.get("params", {})
                    self.trace.append({"thought": parsed2.get("thought", ""), "action": action2, "params": params2})
                    result2 = await self._execute_with_retry(action2, params2, obs)
                    self.trace[-1]["result"] = result2
            return f"Max steps ({self.max_steps}) reached. Last thought: {self.trace[-1]['thought']}"
        finally:
            await self.env.close()

    async def _execute_with_retry(self, action: str, params: Dict, obs: BrowserObservation, retries: int = 2) -> str:
        for attempt in range(retries + 1):
            result = await self.env.act(action, **params)
            if "error" not in result.lower():
                return result
            if action == "click" and attempt < retries:
                # Common fix: scroll down and try again
                await self.env.act("scroll", direction="down", amount=800)
                await self.env.act("wait", ms=500)
        return result
```

The `BrowserAgent` class is the heart of the project. It maintains a `trace` of thoughts, actions, and results. The `_execute_with_retry` method embodies a key recovery strategy: if a click fails, scroll down and retry, because the element may be below the current viewport. This simple heuristic eliminates a large fraction of navigation failures on long pages.

The fallback mechanism is the hybrid router in practice. When a DOM action fails, the agent re-prompts the LLM with knowledge of the failure and the availability of a screenshot. The LLM can then decide to scroll, wait for dynamic content, or try a different selector.

### 4.3 A Mock LLM and Evaluation Harness

To make the project runnable without API keys, we provide a mock LLM that implements a hardcoded strategy for a simple task. Then we define a small curated evaluation set and a harness that runs tasks and scores success.

```python
class MockLLM:
    """Hardcoded planner for a simple task: find the price of 'Widget Pro' on a demo page."""
    def __init__(self):
        self.step = 0
        self.plan = [
            {"thought": "Start by navigating to the product listing.", "action": "navigate", "params": {"url": "https://example-agent-test.local/products"}},
            {"thought": "Scroll to find the Widget Pro card.", "action": "scroll", "params": {"direction": "down", "amount": 600}},
            {"thought": "Click the Widget Pro details link.", "action": "click", "params": {"text": "Widget Pro"}},
            {"thought": "Extract the price text from the page.", "action": "extract", "params": {"js": "document.querySelector('.price').innerText"}},
            {"thought": "Return the answer.", "action": "done", "params": {"answer": "Widget Pro costs $49.99"}},
        ]

    def __call__(self, prompt: str) -> str:
        if self.step < len(self.plan):
            out = json.dumps(self.plan[self.step])
            self.step += 1
            return out
        return json.dumps({"thought": "No more planned steps.", "action": "done", "params": {"answer": "unknown"}})

# Evaluation harness
class BrowserEvaluator:
    def __init__(self, agent_factory):
        self.agent_factory = agent_factory

    async def evaluate_task(self, task: str, checker: Callable[[str], bool]) -> Dict:
        agent = self.agent_factory()
        answer = await agent.run(task)
        success = checker(answer)
        return {
            "task": task,
            "answer": answer,
            "success": success,
            "steps": len(agent.trace),
        }

    async def run_suite(self, suite: List[Dict]) -> List[Dict]:
        results = []
        for item in suite:
            result = await self.evaluate_task(item["task"], item["checker"])
            results.append(result)
        return results
```

The evaluation pattern here is intentionally minimal. A `checker` function inspects the agent's final answer and returns `True` if the task is satisfied. For a real evaluation on WebArena or Mind2Web, the checker would query the benchmark's backend state or compare against a gold answer. For learning purposes, a curated 10–20 task suite with simple string checks is sufficient.

A complete runnable script ties the pieces together:

```python
async def main():
    def make_agent():
        env = BrowserEnv(headless=True)
        return BrowserAgent(llm_call=MockLLM(), env=env, max_steps=10)

    evaluator = BrowserEvaluator(make_agent)

    suite = [
        {
            "task": "Find the price of Widget Pro on the demo store.",
            "checker": lambda ans: "49.99" in ans or "$49" in ans,
        },
        {
            "task": "Navigate to the about page and extract the company mission statement.",
            "checker": lambda ans: "mission" in ans.lower() and len(ans) > 20,
        },
    ]

    results = await evaluator.run_suite(suite)
    passed = sum(1 for r in results if r["success"])
    print(f"Score: {passed}/{len(results)}")
    for r in results:
        print(f"  {'PASS' if r['success'] else 'FAIL'} | {r['steps']} steps | {r['task']}")

if __name__ == "__main__":
    asyncio.run(main())
```

This script demonstrates the **eval-first pattern** that runs through the book: define the task, define the checker, run the agent, and report pass/fail. On a real benchmark, you would replace `MockLLM` with an API call to a frontier model and `suite` with benchmark tasks loaded from WebArena Verified or Mind2Web.

> **🛠️ Implementation Note**
>
> When scaling to real LLM APIs, wrap the `llm_call` with timeout, exponential backoff, and token-usage logging. Browser agents are especially sensitive to latency because each step already incurs Playwright overhead. A slow LLM turns a 10-step task from 30 seconds into 3 minutes, which matters when evaluating hundreds of tasks.

## Summary

- The web is a hostile environment for agents: unstructured, dynamic, and visually grounded. Success requires treating the browser as a first-class environment, not just a source of text to scrape.
- The action space should be atomic and human-like: click, type, scroll, navigate, wait, extract, done. Compound macros fail silently on dynamic pages.
- There are two perception paradigms: DOM-aware (fast, precise, brittle to structure changes) and visual (robust, slow, probabilistic). The 2026 consensus is a hybrid router that uses DOM by default and vision as fallback.
- Playwright is the modern standard for browser automation in agent projects. Its auto-waiting, context isolation, and accessibility tree support eliminate entire classes of race-condition bugs.
- Benchmark progress has been rapid. WebArena Verified and WebArena Verified Hard provide rigorous navigation evaluation; GAIA tests multi-step reasoning with live tools; Mind2Web 2 focuses on long-horizon agentic search. The scaffold effect is enormous: the same model can swing 25–40 points depending on architecture.
- OpAgent shows the power of modular design: separate planner, grounder, reflector, and summarizer modules, each specialized and each potentially backed by a different model.
- A complete browser agent can be built in ~200 lines of pure Python: a ReAct loop, a Playwright environment wrapper, a hybrid perception router, and a retry-aware action executor.
