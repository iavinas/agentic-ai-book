# Chapter 18: Hierarchical Planning with LLMs

> Imagine asking a robotic assistant to prepare breakfast. A flat planner treats every motion as a raw action in a gigantic list: move arm forward three centimeters, close gripper, lift spoon, rotate wrist twelve degrees. The plan is thousands of steps long, brittle to the slightest perturbation, and impossible to debug. A hierarchical planner sees the task differently: prepare breakfast decomposes into gather ingredients, cook eggs, toast bread, and assemble plate. Each of those decomposes further, but only when needed. This chapter shows how to build that hierarchy — first with the classical formalism of Hierarchical Task Networks, then with modern LLM agents that implement the same structure implicitly through reasoner-planner-executor architectures. By the end you will have a working HTN planner in pure Python and a clear map of when hierarchy helps, when it hurts, and how frontier systems like RP-ReAct and ReAcTree put the pieces together.

---

## 1. Hierarchical Task Networks

### 1.1 The Problem with Flat Planning

A flat agent plans at the level of individual tool calls. Search flights, click result three, read policy, enter name, enter date. Every step is an atomic action with no structure above it. When step forty-seven fails — the website times out, the form rejects the input, the agent hallucinates a nonexistent button — there is no scaffolding to catch the fall. The agent must replan from scratch or wander in a local error loop.

Flat planning fails because it ignores abstraction. Human planners do not reason about every muscle twitch; they reason at multiple levels of detail, expanding a high-level goal into subgoals only when the current level is insufficient. **Hierarchical Task Networks (HTN)** formalize this intuition. Originally developed in classical AI planning, HTN represents a task as a network of nodes that decomposes downward until every leaf is an executable primitive. Modern LLM agents rarely call themselves HTN planners, but the best ones — SayCan, RP-ReAct, ReAcTree — are implementing HTN under the hood.

### 1.2 Tasks, Methods, and the Decomposition Boundary

An HTN domain consists of three core constructs: **tasks**, **methods**, and a **state**.

A **task** is something the agent must accomplish. Tasks come in two flavors:

- **Primitive tasks** are directly executable actions with known preconditions and effects. In a robotics domain, `pick_up(block_a)` is primitive. In a software agent, `search_web("Tokyo hotels")` is primitive.
- **Compound tasks** are abstract goals that must be decomposed before execution. `prepare_breakfast` and `book_a_flight` are compound tasks.

The boundary between primitive and compound is not universal — it is a design choice. A primitive in one agent may be a compound in another. The key rule is: a primitive task can be executed by a single tool call or API invocation without further planning. A compound task requires decomposition into subtasks.

A **method** defines one valid way to decompose a compound task. Each method has:

1. A **head**: the compound task it decomposes.
2. A **precondition**: the state conditions that must hold for this decomposition to be valid.
3. A **subtask network**: an ordered (or partially ordered) set of subtasks that accomplish the head.

For example, the compound task `make_coffee` might have two methods: `method_drip` (precondition: drip machine available) and `method_french_press` (precondition: french press available). Each method decomposes into different subtask networks.

The state is a set of ground predicates that evolve as primitives execute. In Blocksworld, the state contains predicates like `on(a, b)` and `clear(c)`. In a travel agent, the state contains `booked(flight_123)` and `confirmed(hotel_x)`. Methods check preconditions against this state; primitives update it through their effects.

### 1.3 Preconditions and Effects

Every primitive task declares what it requires and what it changes. A primitive with a precondition that is not satisfied in the current state cannot be executed. A primitive with an effect updates the state when execution succeeds.

Consider a Blocksworld primitive `stack(x, y)` with:
- Precondition: `holding(x) ∧ clear(y)`
- Effect: `on(x, y) ∧ clear(x) ∧ ¬holding(x) ∧ ¬clear(y)`

This is the language of STRIPS planning, and HTN inherits it. The precondition is a logical formula over state predicates. The effect is a set of additions and deletions. When the primitive executes, the state is updated: new predicates are added, negated predicates are removed.

For LLM agents, preconditions and effects need not be symbolic formulas. They can be natural language descriptions validated by an LLM or a lightweight classifier. The principle remains the same: before executing a subgoal, check whether the world is ready; after executing, update the world model.

### 1.4 Forward Decomposition and Backtracking

HTN planning is the process of repeatedly decomposing compound tasks until the network contains only primitives, then executing those primitives in an order consistent with the dependencies introduced by decomposition. The standard algorithm is **forward decomposition**:

1. Initialize the task network with the top-level goal.
2. Select a compound task from the network.
3. Choose an applicable method whose precondition is satisfied by the current state.
4. Replace the compound task with the method's subtask network.
5. Repeat until no compound tasks remain.
6. Execute primitives in a valid topological order.

If step 3 fails — no method is applicable — the planner **backtracks** to the most recent choice point and tries an alternative method. If all methods fail, the planner reports that the task is unsolvable from the current state.

This is a depth-first search through the space of possible decompositions. The branching factor is the number of methods available for each compound task. In well-structured domains, the branching factor is small and search is efficient. In open-ended domains where an LLM generates methods on the fly, the branching factor is unbounded and search must be guided by heuristics or constrained by a fixed library.

### 1.5 Building an HTN Planner in Pure Python

We now implement a minimal HTN planner for a Blocksworld domain. The domain supports three primitives — `pick_up`, `put_down`, and `stack` — and one compound task — `move_block(x, y)`, which means "move block x onto block y." The planner uses forward decomposition with backtracking.

```python
from dataclasses import dataclass, field
from typing import List, Set, Dict, Tuple, Optional


@dataclass
class State:
    """A set of ground predicates."""
    predicates: Set[str] = field(default_factory=set)

    def holds(self, pred: str) -> bool:
        return pred in self.predicates

    def add(self, pred: str):
        self.predicates.add(pred)

    def remove(self, pred: str):
        self.predicates.discard(pred)

    def clone(self) -> "State":
        return State(set(self.predicates))


@dataclass
class Primitive:
    """An executable action with preconditions and effects."""
    name: str
    preconditions: List[str]
    add_effects: List[str]
    del_effects: List[str]

    def applicable(self, state: State) -> bool:
        return all(state.holds(p) for p in self.preconditions)

    def apply(self, state: State):
        for p in self.del_effects:
            state.remove(p)
        for p in self.add_effects:
            state.add(p)


@dataclass
class Method:
    """A decomposition rule for a compound task."""
    name: str
    preconditions: List[str]
    subtasks: List  # list of Primitive or compound task names with args

    def applicable(self, state: State) -> bool:
        return all(state.holds(p) for p in self.preconditions)


@dataclass
class Task:
    """A node in the task network."""
    name: str
    args: Tuple[str, ...] = ()
    is_primitive: bool = False


class HTNPlanner:
    """Forward-decomposition HTN planner with backtracking."""

    def __init__(self, primitives: Dict[str, Primitive], methods: Dict[str, List[Method]]):
        self.primitives = primitives
        self.methods = methods

    def solve(self, goal: Task, state: State) -> Optional[List[Primitive]]:
        """Return a primitive plan or None if unsolvable."""
        plan, final_state = self._decompose([goal], state.clone(), [])
        return plan

    def _decompose(self, network: List[Task], state: State, plan: List[Primitive]) -> Tuple[Optional[List[Primitive]], State]:
        if not network:
            return plan, state

        task = network[0]
        rest = network[1:]

        if task.is_primitive:
            key = task.name
            if key not in self.primitives:
                return None, state
            prim = self.primitives[key]
            if not prim.applicable(state):
                return None, state
            prim.apply(state)
            plan.append(prim)
            return self._decompose(rest, state, plan)

        # Compound task: try each applicable method
        compound_key = task.name
        if compound_key not in self.methods:
            return None, state

        for method in self.methods[compound_key]:
            if not method.applicable(state):
                continue
            # Expand method subtasks into Task objects
            expanded = []
            for st in method.subtasks:
                expanded.append(Task(name=st.name, args=st.args, is_primitive=st.is_primitive))
            new_network = expanded + rest
            result_plan, result_state = self._decompose(new_network, state.clone(), plan.copy())
            if result_plan is not None:
                return result_plan, result_state

        return None, state
```

The `_decompose` method is the engine. It processes the task network left to right. Primitives are checked against state and executed immediately; compound tasks are expanded via methods. The recursive call clones the state so that backtracking is clean — if a method leads to a dead end, the original state is preserved for the next method attempt.

We instantiate this planner on a tiny Blocksworld problem: stack block A on block B, starting from A on the table and B on the table.

```python
# Blocksworld primitives
primitives = {
    "pick_up": Primitive(
        name="pick_up",
        preconditions=["clear(X)", "on_table(X)", "hand_empty"],
        add_effects=["holding(X)"],
        del_effects=["clear(X)", "on_table(X)", "hand_empty"]
    ),
    "put_down": Primitive(
        name="put_down",
        preconditions=["holding(X)"],
        add_effects=["clear(X)", "on_table(X)", "hand_empty"],
        del_effects=["holding(X)"]
    ),
    "stack": Primitive(
        name="stack",
        preconditions=["holding(X)", "clear(Y)"],
        add_effects=["on(X,Y)", "clear(X)", "hand_empty"],
        del_effects=["holding(X)", "clear(Y)"]
    ),
}

# Note: the generic primitive names above use placeholder (X,Y).
# In a production system we instantiate parameterized copies.
# For this minimal demo we register concrete instantiations:
primitives["pick_up_a"] = Primitive(
    name="pick_up_a", preconditions=["clear_a", "on_table_a", "hand_empty"],
    add_effects=["holding_a"], del_effects=["clear_a", "on_table_a", "hand_empty"]
)
primitives["pick_up_b"] = Primitive(
    name="pick_up_b", preconditions=["clear_b", "on_table_b", "hand_empty"],
    add_effects=["holding_b"], del_effects=["clear_b", "on_table_b", "hand_empty"]
)
primitives["stack_a_on_b"] = Primitive(
    name="stack_a_on_b", preconditions=["holding_a", "clear_b"],
    add_effects=["on_a_b", "clear_a", "hand_empty"],
    del_effects=["holding_a", "clear_b"]
)

# Compound task: move_block(X, Y) -> pick_up(X), stack(X, Y)
move_a_on_b = Method(
    name="move_a_on_b",
    preconditions=["clear_a", "on_table_a", "clear_b"],
    subtasks=[
        Task(name="pick_up_a", is_primitive=True),
        Task(name="stack_a_on_b", is_primitive=True),
    ]
)

methods = {"move_block": [move_a_on_b]}

# Initial state
state = State({"clear_a", "clear_b", "on_table_a", "on_table_b", "hand_empty"})

planner = HTNPlanner(primitives, methods)
goal = Task(name="move_block")
plan = planner.solve(goal, state)

if plan:
    print("Plan found:", [p.name for p in plan])
else:
    print("No plan found")
```

Running this script prints `['pick_up_a', 'stack_a_on_b']`. The planner selected the only applicable method for `move_block`, expanded it into two primitives, verified that each primitive's preconditions held in the evolving state, and produced a valid plan. In a larger domain, the `_decompose` method would backtrack through multiple methods until it found one that led to a complete primitive plan.

> **Key Insight**
>
> HTN's power comes from the method library. A well-curated library encodes domain expertise: chefs know how to make coffee; travel agents know how to book flights. When the LLM generates methods on the fly, it is doing approximate HTN planning with a learned rather than hand-coded library.

## 2. Modern Hierarchical Agents

Classical HTN assumes a fixed domain model: the predicates, primitives, and methods are engineered by a human expert. Modern LLM agents relax this assumption. The LLM itself proposes decompositions, estimates preconditions, and sometimes even invents new subgoals. The result is a hybrid: hand-coded primitives at the bottom (API calls, robotic motions, file operations) and LLM-generated compounds at the top.

### 2.1 SayCan: Grounding Language in Robotic Affordances

SayCan, introduced by Ahn et al. (2022), was among the first systems to pair LLM planning with physical execution constraints. The setup is a household robot receiving natural language commands like "bring me an apple." The LLM proposes a sequence of high-level actions: navigate to the kitchen, open the fridge, pick up the apple, navigate to the user, place the apple.

The twist is **affordance grounding**. Not every action the LLM proposes is physically possible. The robot may not be able to reach the fridge, or the apple may not be inside. SayCan scores each candidate action with a learned value function that estimates the probability that the robot can successfully execute it from the current state. The final plan is the highest-scoring sequence of grounded actions.

Formally, SayCan selects the action $a$ that maximizes:

$$\text{score}(a) = P_{\text{LLM}}(a \mid \text{task}, \text{history}) \cdot P_{\text{affordance}}(a \mid \text{state})$$

The first term is the LLM's probability that action $a$ is a useful next step. The second term is the robot's estimate that $a$ is executable. The product grounds the linguistic plan in physical reality. If the LLM proposes "pick up the apple" but the affordance model reports near-zero probability because the apple is out of reach, the action is rejected and the LLM is asked to propose an alternative — perhaps "open the drawer to get the tongs."

SayCan can be viewed as an HTN system in which the method library is the LLM's parametric knowledge and the precondition check is the affordance value function. The compound tasks are natural language instructions; the primitives are robot skills (pick, place, push, navigate). The decomposition is generated online rather than retrieved from a fixed library.

### 2.2 Inner Monologue: Implicit Hierarchy Through Reasoning

Inner Monologue, introduced by Huang et al. (2022), demonstrates that hierarchy need not be explicit. The system prompts an embodied agent with a stream-of-consciousness template: "I need to put the book on the shelf. First I should pick it up. Wait, the shelf is too high. I need the stool." The LLM generates this internal narrative, and the narrative itself encodes a hierarchical plan. High-level observations ("the shelf is too high") trigger subgoal generation ("I need the stool") which in turn triggers primitive actions ("move to stool, pick up stool, place under shelf").

Inner Monologue is HTN without the formal notation. The LLM is simultaneously the planner, the method library, and the state tracker. The advantage is flexibility: no hand-coded methods are needed. The disadvantage is brittleness: the plan exists only in the LLM's context window, and errors in the monologue propagate silently because there is no explicit dependency graph to validate.

The lesson for agent builders is that implicit hierarchy is a powerful prototyping tool but a risky production strategy. When the task horizon exceeds a dozen steps, the monologue drifts. Explicit task networks — even lightweight ones — provide the guardrails that implicit streams lack.

### 2.3 RP-ReAct: Decoupling Strategy from Execution

RP-ReAct (Reasoner-Planner supervising a ReAct Executor), introduced in late 2025, makes the hierarchy explicit in the architecture rather than the prompt. The system has two distinct agents:

1. The **Reasoner-Planner Agent (RPA)** receives the top-level task and produces a static or slowly evolving subgoal graph. It operates at a high level of abstraction: "search flights," "compare hotels," "check visa requirements."
2. The **Proxy-Execution Agent (PEA)** receives one subgoal at a time and executes it via a ReAct loop. If the subgoal is "search flights," the PEA issues web searches, parses results, and returns a structured summary to the RPA.

The decoupling is the critical design choice. The RPA maintains a global view of the task but never sees the noisy details of HTML parsing, API timeouts, or button clicks. The PEA handles the noise but never loses sight of its single subgoal. When the PEA fails, it reports a condensed failure summary to the RPA, which triggers **local replanning**: only the affected subgoal and its dependents are regenerated.

Tested on ToolQA — a benchmark of complex multi-hop questions requiring sequential tool use — RP-ReAct outperforms flat ReAct by a wide margin on hard tasks. The gains come from two sources: the RPA's global view prevents local loops (e.g., repeating the same search with slightly different queries), and the PEA's scoped context prevents self-conditioning on irrelevant execution noise. In production deployments, the RPA often runs a larger model such as `claude-sonnet-4.7` for strategic reasoning while the PEA dispatches to a faster `kimi-k2.6` endpoint for tool execution, reducing cost without sacrificing plan quality.

### 2.4 ReAcTree, ChatHTN, and ProcLLM: The Frontier of Hierarchical LLM Agents

Three 2025 systems push hierarchical LLM planning beyond the basic RP-ReAct pattern.

**ReAcTree** (AAMAS 2026) represents the plan as a dynamic tree rather than a static graph. Each node is an agent with its own context window and tool set. Control flow nodes — Sequence, Fallback, Parallel — orchestrate execution. A Sequence node executes children in order until one fails. A Fallback node tries children until one succeeds. A Parallel node runs children concurrently and aggregates results. ReAcTree agents maintain **episodic memory** (past task executions) and **working memory** (current subgoal state). On long-horizon household tasks, ReAcTree achieves sixty-one percent goal success versus thirty-one percent for flat ReAct. Remarkably, a seven-billion-parameter model running ReAcTree outperforms a seventy-two-billion-parameter flat ReAct baseline, demonstrating that structure can substitute for scale.

**ChatHTN** (NeuS 2025) bridges classical HTN and LLMs explicitly. Built on PyHop — a lightweight Python HTN planner — ChatHTN uses the LLM to fill decomposition gaps when the hand-coded method library is incomplete. If the domain contains methods for booking flights and hotels but not for visa applications, the LLM generates a new method on demand. A **verifier module** checks that LLM-generated methods are sound: preconditions are satisfiable, effects do not contradict existing state, and the subtask network is acyclic. ChatHTN demonstrates that neuro-symbolic hybrid planning outperforms either pure symbolic or pure neural approaches on structured domains with occasional novel requirements.

**ProcLLM** (2025) takes a different angle: instead of generating hierarchies at test time, it encodes procedural knowledge as hand-coded HTNs that are provided to the LLM as in-context structure. In experiments, a twenty-billion-parameter model with HTN guidance outperforms a one-hundred-twenty-billion-parameter baseline without HTN. The gains come from two mechanisms: **context complexity reduction** (the LLM sees a compact subgoal rather than a full history) and **action loop prevention** (the HTN structure prevents the agent from repeating the same action because the planner advances to the next subgoal when the current one verifies).

These three systems converge on a shared principle: hierarchy is not just a convenience for long-horizon tasks — it is a compression mechanism that reduces the effective context length and prevents repetitive failure modes.

### 2.5 An LLM Subgoal Generator with Dependency Validation

We now implement a lightweight subgoal generator that uses an LLM API to decompose a natural language task into a dependency graph, then validates the graph for cycles before execution.

```python
import json
from dataclasses import dataclass, field
from typing import List, Set, Dict
from collections import deque


@dataclass
class Subgoal:
    id: int
    description: str
    dependencies: List[int] = field(default_factory=list)


class LLMSubgoalGenerator:
    """Generates subgoals from a task description using an LLM backend."""

    def __init__(self, llm_backend):
        self.llm = llm_backend

    def generate(self, task: str, n_subgoals: int = 5) -> List[Subgoal]:
        prompt = (
            f"Decompose the following task into {n_subgoals} subgoals. "
            "Return a JSON list where each item has keys: "
            "id (int), description (str), dependencies (list of ids that must finish first).\n\n"
            f"Task: {task}\n\nJSON:"
        )
        raw = self.llm.complete(prompt)
        data = json.loads(raw)
        return [
            Subgoal(
                id=sg["id"],
                description=sg["description"],
                dependencies=sg.get("dependencies", []),
            )
            for sg in data
        ]


class DependencyValidator:
    """Validates a subgoal graph: no cycles, all dependencies exist."""

    def validate(self, subgoals: List[Subgoal]) -> bool:
        ids = {sg.id for sg in subgoals}
        adj: Dict[int, List[int]] = {sg.id: [] for sg in subgoals}

        for sg in subgoals:
            for dep in sg.dependencies:
                if dep not in ids:
                    return False
                adj[dep].append(sg.id)

        # Kahn's algorithm for cycle detection
        indeg = {sg.id: 0 for sg in subgoals}
        for sg in subgoals:
            for dep in sg.dependencies:
                indeg[sg.id] += 1

        queue = deque([i for i, d in indeg.items() if d == 0])
        visited = 0
        while queue:
            node = queue.popleft()
            visited += 1
            for nb in adj[node]:
                indeg[nb] -= 1
                if indeg[nb] == 0:
                    queue.append(nb)

        return visited == len(subgoals)

    def topological_order(self, subgoals: List[Subgoal]) -> List[int]:
        """Return a valid execution order. Assumes validate() passed."""
        adj = {sg.id: [] for sg in subgoals}
        indeg = {sg.id: 0 for sg in subgoals}
        for sg in subgoals:
            for dep in sg.dependencies:
                adj[dep].append(sg.id)
                indeg[sg.id] += 1

        queue = deque([i for i, d in indeg.items() if d == 0])
        order = []
        while queue:
            node = queue.popleft()
            order.append(node)
            for nb in adj[node]:
                indeg[nb] -= 1
                if indeg[nb] == 0:
                    queue.append(nb)
        return order
```

The generator produces a list of `Subgoal` objects. The validator checks two critical properties: every dependency must reference an existing subgoal ID, and the graph must be acyclic. If validation fails, the agent can retry the LLM call with a corrective prompt ("your previous decomposition contained a cycle; please revise") or fall back to a template-based decomposition.

Running this on the task "plan a seven-day trip to Tokyo" might yield a graph like:

| ID | Description | Dependencies |
|----|-------------|--------------|
| 0 | Determine travel dates and budget | [] |
| 1 | Search and book flights | [0] |
| 2 | Reserve accommodation in Shibuya | [0] |
| 3 | Research visa requirements | [0] |
| 4 | Create day-by-day itinerary | [1, 2] |

The topological order is `[0, 1, 2, 3, 4]` or `[0, 3, 2, 1, 4]` — note that flights, hotels, and visa are independent once dates are fixed, so they can be parallelized.

### 2.6 RP-ReAct Style Decoupled Agent

To make the RP-ReAct pattern concrete, here is a minimal implementation of the two-agent decoupling.

```python
from typing import List, Optional, Callable


class MockLLM:
    """Stand-in for an OpenAI or Anthropic client."""

    def __init__(self, model: str = "gpt-5.5"):
        self.model = model

    def complete(self, prompt: str) -> str:
        # In production: call openai.chat.completions.create
        return '{"subgoals": [{"id":0,"description":"search flights","dependencies":[]}, {"id":1,"description":"book hotel","dependencies":[]}, {"id":2,"description":"confirm itinerary","dependencies":[0,1]}]}'


class ReasonerPlanner:
    """High-level strategist: generates subgoal graph and dispatches one at a time."""

    def __init__(self, llm_backend: MockLLM):
        self.llm = llm_backend
        self.subgoals: List[Subgoal] = []
        self.completed: Set[int] = set()

    def plan(self, task: str):
        prompt = (
            f"Task: {task}\n"
            "Generate subgoals as JSON with fields id, description, dependencies."
        )
        raw = json.loads(self.llm.complete(prompt))
        self.subgoals = [
            Subgoal(id=sg["id"], description=sg["description"], dependencies=sg.get("dependencies", []))
            for sg in raw["subgoals"]
        ]

    def next_subgoal(self) -> Optional[Subgoal]:
        for sg in self.subgoals:
            if sg.id not in self.completed and all(d in self.completed for d in sg.dependencies):
                return sg
        return None

    def mark_done(self, sg_id: int):
        self.completed.add(sg_id)


class ReActExecutor:
    """Low-level executor: runs a ReAct loop on a single subgoal."""

    def __init__(self, llm_backend: MockLLM, tools: Dict[str, Callable]):
        self.llm = llm_backend
        self.tools = tools

    def execute(self, subgoal: Subgoal, max_steps: int = 5) -> dict:
        trace = []
        for step in range(max_steps):
            thought = f"Thinking about: {subgoal.description} (step {step})"
            action = f"tool_call_placeholder_for_{subgoal.description}"
            observation = f"Result of {action}"
            trace.append({"thought": thought, "action": action, "observation": observation})
            if "done" in observation.lower():
                return {"status": "success", "result": observation, "trace": trace}
        return {"status": "failure", "reason": "max_steps reached", "trace": trace}


# Wiring the two agents together
llm = MockLLM(model="gpt-5.5")
rpa = ReasonerPlanner(llm)
pea = ReActExecutor(llm, tools={})

task = "Plan a trip to Tokyo"
rpa.plan(task)

while True:
    sg = rpa.next_subgoal()
    if sg is None:
        break
    result = pea.execute(sg)
    if result["status"] == "success":
        rpa.mark_done(sg.id)
    else:
        # Local replanning: in production, ask RPA to generate an alternative
        print(f"Subgoal {sg.id} failed: {result['reason']}")
        break

print(f"Completed subgoals: {rpa.completed}")
```

The `ReasonerPlanner` maintains the global subgoal graph and tracks completion. The `ReActExecutor` handles the messy details of tool interaction for one subgoal at a time. The wiring loop is the orchestration layer: it pulls ready subgoals, dispatches them, and handles local failures without restarting the entire task.

## 3. When to Plan, When to React

### 3.1 Stable vs. Dynamic Environments

Hierarchy is not free. Maintaining a task network, checking preconditions, and propagating state updates all consume compute and tokens. The agent builder faces a fundamental question: when is the overhead of hierarchical planning justified?

The answer depends on **environment stability**. In a stable environment — a fixed codebase, a static theorem to prove, a batch data pipeline — the world state changes only when the agent acts. A full plan computed upfront is unlikely to become obsolete. Plan-then-execute with hierarchical decomposition is the right strategy because the plan can be globally optimized and executed with minimal replanning.

In a dynamic environment — a live website, a chat conversation, a physical robot navigating a cluttered room — observations arrive that invalidate the plan before it completes. A flight search may reveal that no direct routes exist, forcing a rewrite of the entire itinerary. A chat user may change their request mid-conversation. In these settings, interleaved planning and reactivity dominate.

The boundary is not binary. Most real-world tasks have stable substructures and dynamic surfaces. Booking a flight is stable at the level of "search, filter, select, pay" but dynamic at the level of which specific flights are available. The right architecture is a hybrid: hierarchical decomposition for the stable backbone, reactive execution for the dynamic surface.

### 3.2 The Planning-Replanning Trade-Off

Every planning decision has a cost. Generating a subgoal graph with a frontier model might consume thousands of tokens. If the plan is executed perfectly, the cost is amortized across the task. If the plan fails at step two, the tokens are wasted and a new plan must be generated.

Define the **planning cost** $C_p$ as the compute and latency required to generate a plan, and the **execution failure cost** $C_f$ as the cost of recovering from an incorrect or obsolete plan. The optimal strategy minimizes the total expected cost:

$$\mathbb{E}[C_{\text{total}}] = C_p + p_{\text{fail}} \cdot C_f$$

where $p_{\text{fail}}$ is the probability that the plan becomes invalid before completion. When $p_{\text{fail}}$ is low, invest in thorough upfront planning (low $C_p$ is a false economy). When $p_{\text{fail}}$ is high, minimize $C_p$ by generating partial plans and replanning frequently.

This trade-off explains why RP-ReAct uses a cheap, fast ReAct executor for low-level steps and reserves the expensive planner for high-level subgoal decisions. The RPA generates a coarse plan once; the PEA reacts cheaply within each subgoal. Failures are localized and cheap to repair.

### 3.3 Reactive Planning: Partial Plans and Continuous Revision

**Reactive planning** is the strategy of generating only the immediate next subgoal (or even the immediate next action), executing it, observing the result, and then deciding what to do next. It is the extreme end of the interleaved spectrum: no global plan, just local responses.

Pure reactivity fails on long-horizon tasks because the agent has no memory of the overall goal structure. It is prone to local optima: the agent books a cheap flight that arrives at midnight, not because it failed to search, but because it never maintained a "comfort" subgoal in its working memory.

The modern synthesis is **hierarchical reactive planning**: maintain a high-level plan that changes slowly, but execute it with reactive subroutines that adapt to local conditions. The high-level plan provides the guardrails ("we need flights, hotels, and transport"); the reactive layer provides the agility ("this hotel is full, try the next block").

ReAcTree implements this synthesis explicitly. The tree structure is the high-level plan; the individual nodes run reactive ReAct loops. When a node fails, the Fallback control flow node tries an alternative method — a different hotel search strategy, a different flight aggregator — without discarding the overall trip plan.

### 3.4 Designing the Planning Horizon

A practical question remains: how deep should the hierarchy go? A tree with three levels (goal, subgoals, primitives) is sufficient for most agent tasks. Deeper trees introduce coordination overhead: the planner spends more time managing the tree than executing useful work.

ProcLLM's experiments suggest that the sweet spot for LLM-guided HTN is two to three levels of decomposition. The top level captures the task structure (the "what"), the middle level captures domain-specific strategies (the "how"), and the bottom level is raw tool execution. Going deeper rarely helps unless the domain has deeply nested procedural knowledge, such as multi-stage scientific protocols or complex manufacturing workflows.

For agent builders, the design heuristic is simple: decompose until each leaf is a subgoal that a ReAct loop can complete in five to ten steps with high reliability. If a leaf still requires fifty steps, decompose further. If a leaf is a single API call, stop.

## Summary

- Hierarchical Task Networks (HTN) provide a formal framework for decomposing high-level goals into executable primitives through task, method, and state abstractions. The method library encodes domain expertise; forward decomposition with backtracking finds valid plans.
- **Primitive vs. compound tasks** is a design boundary, not a universal law. Primitives are single tool calls; compounds are anything that requires further decomposition. Modern LLM agents blur this boundary by generating compounds on the fly.
- **SayCan** grounds LLM plans in physical affordances; **Inner Monologue** encodes hierarchy implicitly in stream-of-consciousness reasoning; **RP-ReAct** decouples high-level planning from low-level execution, localizing failures and reducing context corruption.
- **ReAcTree, ChatHTN, and ProcLLM** demonstrate that explicit hierarchical structure can substitute for model scale. A seven-billion-parameter model with hierarchy outperforms a seventy-two-billion-parameter flat agent on long-horizon tasks.
- **Stable environments favor upfront hierarchical planning**; **dynamic environments favor reactive execution**. The optimal architecture is a hybrid: a slowly changing high-level plan with reactive subroutines that adapt to local conditions.
- The planning-replanning trade-off is governed by the probability of plan invalidation. Invest in planning when the environment is stable; minimize planning cost and replan frequently when it is volatile.
