# Chapter 14: Agents with Visual Perception

> In 2023, an agent asked to "disable dark mode in the settings menu" had no eyes. It parsed HTML text, guessed element IDs, and prayed its click coordinates hit the right pixel. The result was brittle, slow, and blind to the actual interface it was manipulating. Vision changes everything. When an agent processes screenshots, PDFs, and video frames as first-class observations, it grounds its decisions in the real visual world rather than in abstractions inferred from text. By the end of this chapter you will build a `VisualAgent` in pure Python that ingests images, reasons about their contents, and emits grounded actions — a complete multimodal loop with evaluation on visual reasoning tasks.

---

## 1. The Multimodal Agent Paradigm

### 1.1 Vision-Language Models as Agent Eyes

Before an agent can act on visual information, it must perceive it. The breakthrough that made visual agents practical was the unification of vision encoders and language decoders into a single end-to-end model. Early systems chained a separate object detector (YOLO, Faster R-CNN) to an LLM via text captions: "Image contains: dog, sofa, lamp." This pipeline was slow, error-prone, and lost spatial nuance. Modern **vision-language models** (VLMs) fuse the two modalities inside the transformer itself.

The architecture is conceptually simple. An image is passed through a vision encoder — typically a ViT — producing a sequence of patch embeddings $v_1, \dots, v_n \in \mathbb{R}^{d}$. These visual tokens are concatenated with text token embeddings and fed into a causal language model. The model learns to attend across both modalities in a single forward pass. Where earlier systems saw an image as metadata, the VLM sees it as part of the context window.

This fusion unlocks **visual question answering** (VQA) as the simplest agentic capability: the user uploads an image and asks a question. The VLM reasons jointly over pixels and text. But VQA is only the starting point. The frontier is agents that use vision to *act*: navigating UIs from screenshots, extracting structured data from PDFs, tracking objects across video frames, and pointing to specific coordinates with pixel-level precision.

### 1.2 From Passive Description to Active Grounding

Seeing is not enough; an agent must be able to point. **Visual grounding** is the capacity to connect linguistic references to specific regions in an image. When an agent is told "click the Submit button," it needs to output not just the word "Submit" but the bounding box $(x_1, y_1, x_2, y_2)$ where the button lives. This transforms the agent from a passive describer into an active participant in visual environments.

Three distinct approaches to coordinate prediction have emerged. **Kosmos-2** (Microsoft Research, 2023) discretized the image into a $32 \times 32$ grid and added special location tokens to the LLM vocabulary, such as `<patch_index_0044>`. The model emits grounding as a sequence of discrete tokens no different from words. **Shikra** (SenseTime, 2023) took the opposite approach: it represents coordinates as plain text numbers `[0.268, 0.372, 0.789, 0.999]`, normalized to $[0, 1]$, with no special vocabulary at all. **Qwen2-VL** (Alibaba, 2024) and its successors introduced structured token pairs: `<box_start>(348, 291),(412, 340)<box_end>`, explicitly demarcating spatial regions while keeping the representation human-readable.

Look at Figure 14.1. The crucial observation is that all three methods solve the same interface problem: the LLM speaks tokens, but the world operates in continuous $(x, y)$ space. The representation choice determines what the agent can express. Kosmos-2's grid is coarse but requires no custom parser. Shikra's raw numbers are maximally flexible. Qwen2-VL's structured tags enable reliable post-processing into tool calls. For production agents, the structured-token approach dominates because it lets the harness convert model output into `pyautogui.click(348, 291)` deterministically.

### 1.3 Document Understanding: Beyond OCR

Documents — PDFs, scanned forms, charts, and tables — present a distinct visual challenge. Text-only agents once relied on OCR pipelines (Tesseract) followed by heuristic layout extraction. This approach collapses on complex documents: multi-column newspapers, tables nested inside figures, equations spanning multiple lines, and forms with overlapping fields.

The 2025 breakthrough is **end-to-end document VLMs** that parse visual layout and content jointly. **olmOCR** (Allen Institute, 2025) fine-tuned a 7B Qwen2-VL to convert PDF pages into clean Markdown, using a technique called *document anchoring*: the model receives both the rasterized image and raw text metadata extracted from the PDF binary, dramatically reducing hallucination. **SmolDocling** (IBM/HuggingFace, 2025) pushed parameter counts down to 256M while introducing **DocTags**, a universal markup format that preserves spatial relationships and hierarchy in a single pass. **PaddleOCR-VL** (Baidu, 2025) and **MonkeyOCR v1.5** (KingSoft, 2025) achieved state-of-the-art on OmniDocBench v1.5 by decoupling layout analysis from content recognition in a two-stage pipeline.

For agents, this means a single tool call can ingest a 50-page financial report and return structured JSON with tables, charts, and cited sections. The agent no longer needs to know whether the document is a scanned receipt or a born-digital PDF; the VLM handles both.

## 2. Visual Agents in Action

### 2.1 UI Understanding from Screenshots

The most demanding visual agent task is operating a graphical user interface. The agent receives a screenshot and must decide what to click, where to type, and when to scroll. This is how Claude Computer Use, Gemini, and Qwen3-VL operate PCs and phones. Unlike web agents that parse HTML (Chapter 11), these systems are purely visual: no DOM, no accessibility tree, just pixels.

The challenge is **GUI grounding** — locating UI elements with pixel precision. Early attempts used OCR to extract text coordinates and heuristic matching to find buttons. Modern agents use VLMs trained on millions of UI screenshots. **SeeClick** (2024) introduced the ScreenSpot benchmark and showed that a 9.6B parameter model could outperform the larger 18B CogAgent on GUI grounding by focusing specifically on click prediction. **CogAgent** (CVPR 2024) pioneered a dual-resolution cross-attention module: a low-resolution 224×224 encoder for global context plus a high-resolution 1120×1120 encoder for tiny text and icons.

The action loop for a UI agent follows the same ReAct pattern from Chapter 2, but the observation is an image:

```mermaid
flowchart TD
    A[Task: Enable dark mode] --> B[Capture Screenshot]
    B --> C[VLM: Analyze UI]
    C --> D{Action Type?}
    D -->|Click| E[Predict (x, y)]
    D -->|Type| F[Predict (x, y) + text]
    D -->|Scroll| G[Predict direction]
    E --> H[Execute Action]
    F --> H
    G --> H
    H --> I[New Screenshot]
    I --> C
```

Look at the diagram. Notice the loop is identical to a text-based ReAct agent; only the observation modality changes. This is the power of multimodal agency: the *architecture* is general, while the *perception module* is swappable.

### 2.2 Spatial Reasoning and Coordinate Prediction

Spatial reasoning demands more than object detection. An agent must understand relative positions: "the third button in the second row," "the checkbox to the left of the price," "the icon below the search bar." This requires the VLM to maintain a coherent spatial map across the image.

Frontier models still struggle with precise spatial tasks. On the OlympicArena benchmark, even GPT-4o and Claude 3.5 Sonnet scored below 41% on multimodal science problems requiring diagrammatic reasoning. On CAPTCHA-style tasks requiring pixel-level interaction, specialized fine-tuned models such as Qwen2.5-VL-7B outperform generalist frontiers, proving that **scale alone does not guarantee visual precision**. The 2025 consensus, embodied by systems like **Orion** and **VisionAgent**, is to augment monolithic VLMs with specialized computer-vision tools: object detectors, OCR engines, and geometric analyzers orchestrated by an agentic planner.

### 2.3 Video Understanding and Temporal Reasoning

Static images capture a single moment. Video captures process, motion, and causality. A video-enabled agent can watch a tutorial, monitor a security feed, or navigate a dynamic game world. But video is expensive: a one-minute video at one frame per second contains 3,600 visual tokens per frame, quickly overwhelming context windows.

The solution is agentic temporal scoping. Rather than feeding every frame to the VLM, the agent treats time as a search space. **VideoARM** (2025) introduced a hierarchical memory architecture: a planner agent scopes relevant time intervals, a clip explorer samples key frames, and a local analyzer performs fine-grained QA on selected segments. This reduces token consumption by a factor of 34 on 30-minute videos while improving accuracy. **WorldMM** (2025) formalized three memory tiers for video agents:

- **Episodic memory:** fact-indexed events at multiple temporal scales
- **Semantic memory:** continuously updated conceptual knowledge
- **Visual memory:** detailed scene information preserved as image embeddings, not just text summaries

For agents, video understanding is not a batch inference problem; it is an interactive retrieval problem. The agent decides where to look, just as a human skims a video before rewinding to the key moment.

## 3. Visual Memory

### 3.1 Images as State

Text-based agents store memory as strings. Visual agents must store memory as images, or at least as embeddings derived from images. When an agent navigates a 20-page form, it needs to remember what Page 3 looked like after it has processed Page 17. When an agent explores a building in a robotics task, it needs a mental map of rooms it has already visited.

**Visual memory** stores image observations as part of the agent's persistent state. The simplest form is a rolling buffer of recent screenshots. More sophisticated systems use a **visual knowledge graph**: nodes represent scenes or objects, edges represent spatial or temporal relationships, and each node is associated with a CLIP or VLM embedding for retrieval.

The retrieval problem is multimodal. The agent might query its memory with text ("what did the settings page look like?") or with an image ("have I seen this dialog box before?"). Cross-modal embeddings make this possible: the same vector space represents both text and image queries.

### 3.2 Hierarchical Multimodal Memory

Production visual agents need memory hierarchies that mirror human cognition. **VideoARM's HM3** (Hierarchical Multimodal Memory) offers a practical blueprint:

1. **Sensory memory:** raw perceptual data — screenshots, frames, audio snippets — stored with timestamps and spatial metadata.
2. **Result memory:** structured outputs of perception tools — detected objects, OCR text, caption summaries — linked back to sensory sources.
3. **Working memory:** the agent's current reasoning trace, goals, and subtask stack.

When the agent encounters a new screenshot, it does not blindly append it to context. It queries result memory to see if similar scenes have been analyzed, retrieves relevant prior observations, and decides whether to zoom in on a specific region or skip redundant frames. This is the visual equivalent of the RAG loop from Chapter 13, but the retrieved documents are images rather than text passages.

## 4. Project — Building a Multimodal Agent

We now build a `VisualAgent` in pure Python. The agent accepts image paths as observations, uses a vision-enabled LLM backend for reasoning, and emits grounded actions such as answering visual questions or predicting click coordinates. We provide a mock VLM backend so the code runs without API keys, while preserving the exact interface you would use with GPT-4V, Claude 3, or Qwen3-VL.

### 4.1 Architecture

The agent has three components: a **perception module** that loads and resizes images, a **reasoning module** (the VLM backend) that processes image + text prompts, and an **action module** that parses structured output into executable tool calls.

```python
import json
from dataclasses import dataclass
from typing import List, Tuple, Optional, Dict, Any
from PIL import Image

@dataclass
class Action:
    action_type: str          # "answer", "click", "type", "scroll", "terminate"
    content: str              # text answer or input string
    coordinates: Optional[Tuple[int, int]] = None   # (x, y) for click / type
    reasoning: str = ""       # chain-of-thought from the VLM

class MockVisionBackend:
    """Pluggable VLM backend. In production, replace with OpenAI, Anthropic, or Qwen API."""
    
    def __init__(self, responses: List[Dict[str, Any]] = None):
        # Pre-programmed responses for deterministic evaluation
        self.responses = responses or []
        self.call_idx = 0
    
    def complete(self, image: Image.Image, prompt: str) -> str:
        """Return a structured JSON response given an image and text prompt."""
        if self.call_idx < len(self.responses):
            resp = self.responses[self.call_idx]
            self.call_idx += 1
            return json.dumps(resp)
        # Default fallback: answer with coordinates derived from prompt heuristics
        return json.dumps({
            "action": "answer",
            "content": "Mock response",
            "reasoning": "No programmed response available."
        })
```

The mock backend preserves the interface contract: `complete(image, prompt)` returns a JSON string describing the desired action. Swapping in a real backend requires only changing this one method.

```python
class OpenAIVisionBackend:
    """Production backend using OpenAI's GPT-4o vision capabilities."""
    
    def __init__(self, client, model: str = "gpt-4o"):
        self.client = client
        self.model = model
    
    def complete(self, image: Image.Image, prompt: str) -> str:
        """Encode image as base64 and call the vision-capable chat API."""
        import io, base64
        buffer = io.BytesIO()
        image.save(buffer, format="PNG")
        b64 = base64.b64encode(buffer.getvalue()).decode()
        
        response = self.client.chat.completions.create(
            model=self.model,
            messages=[
                {
                    "role": "user",
                    "content": [
                        {"type": "text", "text": prompt},
                        {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{b64}"}}
                    ]
                }
            ],
            response_format={"type": "json_object"}  # enforce JSON
        )
        return response.choices[0].message.content
```

The OpenAI backend streams a base64-encoded PNG into the chat completions API and forces JSON output via `response_format`. Anthropic's Messages API uses an analogous `image` block with `source={"type": "base64", "media_type": "image/png", "data": b64}`. Qwen3-VL via vLLM accepts the same OpenAI-compatible endpoint. The agent itself remains untouched: only the backend class changes.

### 4.2 The VisualAgent Class

The agent maintains a trace of observations and actions, exactly like the ReAct agent from Chapter 6. The difference is that observations may include images.

```python
class VisualAgent:
    def __init__(self, backend: MockVisionBackend, tools: Dict[str, Any]):
        self.backend = backend
        self.tools = tools
        self.trace: List[Dict[str, Any]] = []   # (observation, thought, action)
    
    def observe(self, image_path: Optional[str] = None, text: str = "") -> Dict[str, Any]:
        """Construct an observation dict. Images are loaded lazily."""
        obs = {"text": text}
        if image_path:
            obs["image"] = Image.open(image_path).convert("RGB")
        return obs
    
    def think_and_act(self, task: str, observation: Dict[str, Any]) -> Action:
        """Prompt the VLM with current task + trace + observation; parse action."""
        prompt = self._build_prompt(task, observation)
        image = observation.get("image")
        raw = self.backend.complete(image, prompt)
        parsed = json.loads(raw)
        return Action(
            action_type=parsed["action"],
            content=parsed.get("content", ""),
            coordinates=tuple(parsed["coordinates"]) if "coordinates" in parsed else None,
            reasoning=parsed.get("reasoning", "")
        )
    
    def _build_prompt(self, task: str, observation: Dict[str, Any]) -> str:
        """Assemble a structured prompt with the full reasoning trace."""
        lines = [
            "You are a visual agent. Respond ONLY with valid JSON containing keys:",
            "action: one of [answer, click, type, scroll, terminate]",
            "content: string (answer text or input text)",
            "coordinates: [x, y] integer pixel coordinates (optional)",
            "reasoning: brief chain-of-thought",
            "",
            f"Task: {task}",
            "Trace so far:",
        ]
        for step in self.trace:
            lines.append(f"- {step['action'].action_type}: {step['action'].reasoning}")
        if observation.get("text"):
            lines.append(f"Current text observation: {observation['text']}")
        lines.append("Analyze the provided image and choose the next action.")
        return "\n".join(lines)
    
    def run(self, task: str, observation: Dict[str, Any], max_steps: int = 10) -> str:
        """Execute the multimodal ReAct loop."""
        for step in range(max_steps):
            action = self.think_and_act(task, observation)
            self.trace.append({"observation": observation, "action": action})
            
            if action.action_type == "terminate":
                return action.content
            if action.action_type == "answer":
                return action.content
            
            # Execute tool and receive new observation
            if action.action_type in self.tools:
                result = self.tools[action.action_type](action, observation.get("image"))
                observation = self.observe(text=result)
            else:
                observation = self.observe(text=f"Unknown action: {action.action_type}")
        
        return "Max steps reached without termination."
```

The loop is identical to a text ReAct agent: observe, think, act, update trace. The only extension is that the observation dictionary may contain a PIL `Image` object, and the VLM backend must accept image inputs.

### 4.3 Visual Tools

Tools bridge the agent's decisions to the environment. We implement three visual tools: a **QA tool** that answers questions about an image, a **grounding tool** that predicts click coordinates for a described element, and a **scroll tool** that simulates viewport navigation.

```python
class VisualTools:
    """Environment interface for visual agents. Replace with Playwright, PyAutoGUI, or ROS."""
    
    @staticmethod
    def click_tool(action: Action, image: Optional[Image.Image]) -> str:
        """Simulate a click at predicted coordinates and return feedback."""
        if action.coordinates is None:
            return "Error: click action missing coordinates."
        x, y = action.coordinates
        w, h = image.size if image else (0, 0)
        if 0 <= x < w and 0 <= y < h:
            return f"Clicked at ({x}, {y}). Screen updated."
        return f"Invalid coordinates ({x}, {y}) for image size ({w}, {h})."
    
    @staticmethod
    def type_tool(action: Action, image: Optional[Image.Image]) -> str:
        """Simulate typing text at predicted coordinates."""
        if action.coordinates is None or not action.content:
            return "Error: type action requires coordinates and content."
        return f"Typed '{action.content}' at {action.coordinates}."
    
    @staticmethod
    def scroll_tool(action: Action, image: Optional[Image.Image]) -> str:
        """Simulate scrolling and return a description of new content."""
        direction = action.content.lower()
        return f"Scrolled {direction}. New viewport content loaded."
    
    @staticmethod
    def answer_tool(action: Action, image: Optional[Image.Image]) -> str:
        """Terminal tool: return the agent's answer."""
        return action.content
```

### 4.4 Evaluation on Visual Reasoning Tasks

To evaluate the agent without relying on live API calls, we construct a small benchmark of synthetic VQA tasks with programmed backend responses. This pattern — mock backend + deterministic test cases — is exactly how you unit-test a multimodal agent before deploying it to production APIs.

```python
def build_test_backend() -> MockVisionBackend:

        },

        {
```

> **🛠️ Implementation Note**
>
> For production UI agents, always validate predicted coordinates against the current screenshot dimensions before executing clicks. A VLM trained on mixed resolutions may emit normalized coordinates; convert them with `x = int(x_norm * width / 1000)` for Qwen-style outputs or `x = int(x_norm * width)` for Shikra-style outputs.

```python
def normalize_coordinates(
    raw: Tuple[float, float],
    image_size: Tuple[int, int],
    style: str = "qwen"           # "qwen" (0-1000), "shikra" (0-1), or "absolute"
) -> Tuple[int, int]:
    """Convert model-specific normalized coordinates to absolute pixel values.
    
    Args:
        raw: (x, y) from the VLM output.
        image_size: (width, height) of the current screenshot.
        style: which normalization scheme the model uses.
    
    Returns:
        (x_abs, y_abs) integer pixel coordinates clipped to image bounds.
    """
    w, h = image_size
    x_norm, y_norm = raw
    
    if style == "qwen":
        x_abs = int(x_norm * w / 1000)
        y_abs = int(y_norm * h / 1000)
    elif style == "shikra":
        x_abs = int(x_norm * w)
        y_abs = int(y_norm * h)
    else:
        x_abs, y_abs = int(x_norm), int(y_norm)
    
    # Clip to image bounds to prevent out-of-range clicks
    x_abs = max(0, min(x_abs, w - 1))
    y_abs = max(0, min(y_abs, h - 1))
    return x_abs, y_abs
```

The `normalize_coordinates` helper is defensive by design. VLMs occasionally hallucinate coordinates outside the image frame, especially when fine-tuned on datasets with varying aspect ratios. Clipping prevents crashes and reduces false-positive tool executions.

## Summary

- **Vision transforms the agent loop.** When observations include images, screenshots, and video frames, the agent grounds its reasoning in the real visual world rather than in inferred text abstractions.
- **Visual grounding is the critical bridge.** An agent must emit pixel coordinates, bounding boxes, or region references to act on visual input. Discrete location tokens, raw numeric coordinates, and structured spatial tags are the three dominant representation paradigms.
- **Document understanding is now end-to-end.** Modern document VLMs such as olmOCR, SmolDocling, and PaddleOCR-VL parse complex PDFs, charts, and tables in a single pass, replacing fragile OCR-heuristic pipelines.
- **Video demands agentic temporal scoping.** Processing every frame is infeasible; hierarchical memory architectures like VideoARM's HM3 let the agent search time selectively, reducing token costs by an order of magnitude.
- **Visual memory is multimodal RAG.** Storing and retrieving image embeddings alongside text traces enables agents to answer "have I seen this before?" and to maintain spatial maps across long episodes.
